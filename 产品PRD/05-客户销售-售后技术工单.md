# 05 · 客户/销售 — 售后技术工单（模块 05c）

> 本文是 PHOTONTECK（富泰）CRM+WMS **商业交付级业务 PRD** 的模块 05c，覆盖**售后 / 技术支持工单**这一条「签单之后、货已售出」才发生的轻量服务流程：客户报来技术问题 / 质量故障 → 建工单（关联客户 / 型号 / 可关联 RMA 或 SN）→ FAE（产品工程师）处理（答疑 / 判定 / 维修建议）→ 关闭。**产品部 FAE / PM 为主，SALES / SA 提报**。
>
> 边界与归属：
> - 本模块**只做轻量工单**：登记客诉、记 FAE 答疑 / 判定 / 维修建议、按状态推进、留痕、关闭。**不做复杂派工 / 工时计费 / 维修工序排程 / 备件管理**（访谈 10/11 明确售后「比较琐碎」、靠每个工程师的技术经验一对一处理，无固定复杂工作流，访谈 10:340/352、访谈 11:334/346）。
> - 工单**判定要退货 / 要换 / 要返修**时，**不在本单内处理实物退换**，而是**派生 / 关联一张 RMA 统一单**（`RMA`，决策⑨，在 04b 采购供应链模块），实物退运 / 货回入库 / 退客户全部走 RMA。本单只负责「技术侧的答疑与判定」，是 RMA 的**前置技术工单 / 触发源**之一。
> - 与 05a CRM 前段（签单前售前技术支持）区分：CRM 前段 FAE 做的是**售前**规格确认 / 需求转化（05a）；本模块是**售后**——「产品真的销售出去了，才会负责售后这一块」（访谈 10:238、访谈 11:247）。
>
> 公约一律继承 `00-导航与参与者总览.md`（下称「00 §x」）：角色中文名（SALES/SA/PM(PD)/FAE/PA/LOGISTICS/FINANCE/FINANCE_DIRECTOR/BOSS/ADMIN）、编号规则总表（00 §7）、字段隐藏清单（00 §8：成本 / 利润点 / 采购价对 SALES 隐藏）、金蝶推送总则（00 §6）、引擎映射通用约定（00 §4）、UX 公约（00 §5：表单网格 + 行内 cell 选择器、扫码兜底、预览→确认卡→提交）。本文不重述，只在落点处引用。
>

## 模块定位（业务价值一句话）

把现在散在飞书群 / 微信 / 电话 / 个人 Excel 里、「每个工程师靠技术经验一对一处理、解决了就解决了、不一定每条都留痕」（访谈 10:367-376）的售后技术答疑 / 维修判定，收口为**一张有状态机、有归属、有留痕、可统计的轻量工单**：

- **谁报、报什么、给谁处理、怎么答 / 怎么判、结没结**——客户报来的技术问题 / 故障一进系统就有工单号、有负责 FAE、有客户 / 型号 / 可选 SN 上下文，FAE 答疑 / 判定 / 给维修建议全程记录，关闭时有结论。
- **答疑 vs 维修两条轻量路径**（访谈 10:346/352）：①**答疑 / 远程指导**就能解决 → 工单内闭环关闭；②需**退回维修 / 报原厂**→ 工单做技术判定后**派生 / 关联 RMA**（实物流转交 RMA，04b），技术侧结论沉淀在本工单。
- **沉淀 + 统计**：替代「解决了就解决了、没记录」的现状——按产线 / 型号 / FAE 统计售后工单量、答疑 vs 维修占比、高频故障型号，为 PM 看本产线售后负担提供数据（呼应访谈 10:394 甲方认可「统计分析 OK」、反对售后做 AI 自动答疑）。

> 轻量边界（再次强调，避免过度设计）：本模块**不做**派工调度引擎、工时 / 维修费核算、备件库存、SLA 倒计时硬卡、维修工序流转。售后「琐碎、靠人」（访谈 10:340）——系统只提供**登记 + 归属 + 留痕 + 状态 + 统计**，处理动作本身仍是 FAE 人工完成。

## 05c-1 售后技术工单列表 / 台账 ⭐

### 定位与使用者
PM / FAE / SALES / SA 用来看「本产线 / 本人 / 本客户当前有哪些售后工单、各在什么状态、是答疑还是维修、有没有拖太久」。本质是 `SERVICE_TICKET` 的**台账视图 + 待办面**（决策⑦：明细为真相、汇总=同库视图），替代「散在群里 / 解决了就没记录」的现状（访谈 10:367-376）。

### 典型场景
1. FAE 打开台账，看分派 / 流转到本人或本产线的工单：工单号、客户、型号、问题摘要、处理方式（答疑 / 维修 / 报原厂）、状态、提报时间、已挂天数。
2. PM 看本产线全部售后工单 + 统计：答疑 vs 维修占比、哪些型号反复出问题（高频故障）、哪些工单挂太久未闭环。
3. SALES 看自己客户报的工单进度，把 FAE 结论回传客户。
4. 点任一工单**下钻**进 05c-2 详情 DocEditor 做处理 / 推进。

### 字段表（`SERVICE_TICKET` 台账投影，**只读网格 + 下钻**）

| 字段 | 类型 | 说明 | 引擎映射 |
|---|---|---|---|
| 工单号 `ticket_number` | string | 下钻入单 | ✅ DataExplorer 自动出列；`*_number` 模糊搜索 |
| 客户 `customer_id` | fk 标签 | — | ✅ FK 标签解析（引擎 01 §1.4.2）|
| 型号 `material_id` | fk 标签 | 故障型号 | ✅ FK 标签解析 |
| 问题摘要 `issue_summary` | text | 失效现象 | ✅ |
| 问题类型 `issue_type` | enum 药丸 | 答疑咨询 / 质量故障 / 使用指导 / 其他 | ✅ |
| 处理方式 `resolution_type` | enum 药丸 | 远程答疑 / 国内维修 / 寄原厂维修 / 报原厂换退（转 RMA）/ 内部消化 | ✅ |
| 负责 FAE `assignee_id` | fk 标签 | 本产线工程师 | ✅ FK 标签解析 |
| 产线 / 事业部 `product_line` | enum/fk | 数据范围与统计维度 | ✅（与商机 / RMA 一致）|
| 关联 RMA `rma_id` | fk 标签 | 转实物退换时 | ✅ FK 标签解析（指向 04b `RMA`）|
| 关联 SO `sales_order_id` | fk 标签 | 售出订单上下文 | ✅ FK 标签解析（指向 05b `SALES_ORDER`）|
| 提报时间 `created_at` | datetime | — | ✅ 系统列、ISO 格式化 |
| 已挂天数 `aging_days` | number | =今天 - 提报日 | ➕ 计算字段（同备货 / 样品超期算法，04b）|
| 状态 `status` | pill | OPEN/IN_PROGRESS/RESOLVED/CLOSED | ✅ status 药丸自动过滤 |

### 引擎映射 & 兼容性
- ✅ **DataExplorer 只读台账 + 下钻 DocEditor**：决策⑩「大台账=只读网格 + 下钻」（引擎 08 §8.3）；status 自动过滤、金额无（售后无金额主线）、FK 标签解析、CSV 导出。
- ➕ **已挂天数计算字段**：引擎序列化按列出列、无原生计算列 → 派生只读列 / 前端按 `today - created_at` 即时算（与 04b 备货 / 样品超期天数同机制）。**必入 engineFlags**。
- ➕ **售后统计看板**（答疑 vs 维修占比 / 高频故障型号 / 产线趋势）：属「报表非元数据化」（引擎 00 D7）→ 手写路由 + 租户作用域 + 产线作用域。**必入 engineFlags**。
- ✅ **本产线 / 本人 / 本客户数据范围**：FAE/PM 看本产线（行级 company_id + 产线归属过滤）；SALES 看本人客户工单；BOSS / 财务总监跨公司只读（read-privileged，00 §9）。

### 验收标准 + gap
- ✅ 台账按 `product_line` / `status` / `resolution_type` 过滤；下钻进详情。
- ✅ FAE 只看到分派 / 流转到本产线 / 本人的工单；SALES 只看本人客户工单。
- ✅ 已挂天数随今日自动变化；挂太久未闭环工单可排序置顶。
- gap-1：售后统计**维度全集**（除答疑 / 维修占比、高频故障型号外，是否要按客户 / 季度 / 原厂统计）——默认产线 × 处理方式 × 型号三维，待甲方（呼应访谈 10:394「统计分析 OK」）。

## 05c-2 售后技术工单详情（建单 / 处理 / 关闭）⭐

### 定位与使用者
建工单、FAE 处理、关闭的主界面。建单主用 SALES/SA（提报）、处理主用 FAE（答疑 / 判定 / 维修建议）、升级决策 PM。是本模块唯一的单据编辑页。

### 典型流程 / 场景（端到端，引用访谈 10/11 售后口径）

1. **提报建单**（访谈 10:181「签单发货后出质量问题=售后」）：SALES / SA 接到客户售后反馈 → 建工单：关联**客户**、**型号**（必填上下文）、可选关联**SN/LOT**（哪一只 / 哪一批，便于 FAE 倒查发货 / 批次）、可选关联**SO / 发货记录**（哪一单售出的）、填**失效现象 / 使用场景 / 紧急度**；指派 / 流转给本产线**负责 FAE**（产线=供应商，访谈 10:193/199）。生成**工单号**（编号规则见下）。
2. **FAE 接单处理**（访谈 10:346/352/364）：工单进 `IN_PROGRESS`，FAE 据型号 / 客户 / SN 上下文判断——
   - **能远程答疑 / 指导**：电话 / 邮件给客户指导，记**处理过程 + 结论** → 处理方式=`远程答疑` → 可直接推 `RESOLVED`（访谈 10:346「远程电话指导就可以解决」）。
   - **需技术判定**：判**是什么问题 / 什么原因**（访谈 10:364），做**质量判定（好 / 坏）**（蓝图 §2），给**维修建议**（国内修 / 寄原厂修）。
   - **自己解决不了**：**上报 PM**（升级），或**问原厂**（访谈 10:364「他不行得问原厂」）——问原厂的商务流程走采购侧 / RMA。
3. **判定要走实物退换 / 报原厂**（关键分叉，与 04b RMA「是否报原厂」决策对齐）：
   - 需把货**退回国内维修 / 寄原厂维修 / 报原厂换退**（访谈 10:346/364）→ 工单**派生 / 关联一张 RMA**（`RMA`，04b 决策⑨）：实物退运、货回入库带来源标记、退 / 换客户全部在 RMA 单进行；本工单进 `ESCALATED_RMA` 态，`resolution_type` 记对应值、`rma_id` 关联 RMA 号，**技术结论沉淀在本工单**。
   - 实物处理完、客户问题解决后，工单回 `RESOLVED`（与 RMA 状态可在台账 / 链路对照，非强绑定）。
4. **关闭**（访谈 10:367-376「处理过程要有记录」的落地）：工单 `RESOLVED` 后由 FAE / PM 确认结论无误 → `CLOSED`；留下完整处理记录（答疑内容 / 判定结论 / 维修建议 / 关联 RMA），沉淀为可统计资产。

> 与 RMA 的边界（再次明确）：**本工单不碰实物**。凡涉退 / 换 / 修的实物流转（核 SN/LOT、是否我方卖、是否过保、退关 180 天、货回入库、退客户）一律在 04b `RMA` 单完成（决策⑨ SA/PA 双视图）。本工单 = 售后的**技术答疑与判定层** + RMA 的**触发源之一**。RMA 也可由 SA 在 04b 直接发起（客户直接报退货）——两条入口并存，靠 `rma_id` / `service_ticket_id` 互相关联即可，不强制工单先行。

### 状态机 `SERVICE_TICKET`（OPEN→IN_PROGRESS→RESOLVED→CLOSED，轻量四态 + RMA 升级旁路）

| 状态 | 可操作角色 | 可编辑字段 | 硬规则 / 关卡 | 下一状态 |
|---|---|---|---|---|
| `OPEN`（提报，初始）| SALES/SA/FAE/PM | 客户 / 型号 / SN / 问题描述 / 问题类型 / 紧急度 / 负责 FAE / 产线 / 子表 | 客户 / 型号 / 问题描述 / 负责 FAE 必填才能提交 | →`IN_PROGRESS`（FAE 接单）/ →`CLOSED`（无效 / 误报，直接关）|
| `IN_PROGRESS`（FAE 处理中）| FAE/PM | resolution_type / resolution_notes / quality_verdict / repair_advice | 处理方式 + 处理过程录入；解决不了可升级 | →`RESOLVED`（答疑 / 处理完成）/ →`ESCALATED_RMA`（需实物退换报原厂）|
| `ESCALATED_RMA`（升级 / 转 RMA，旁路）| FAE/PM/PA | rma_id（关联 RMA 号）| **派生 / 关联 04b `RMA` 单**（决策⑨）；实物流转在 RMA | →`RESOLVED`（RMA 处理完、客户问题解决）|
| `RESOLVED`（已解决）| FAE/PM | closure_note | 处理结论必填；待确认无误 | →`CLOSED`（确认关单）/ →`IN_PROGRESS`（客户反馈未解决，重开处理）|
| `CLOSED`（关闭，终态）| — | — | — | — |

> 轻量说明：四态主干 `OPEN→IN_PROGRESS→RESOLVED→CLOSED`（任务规定），`ESCALATED_RMA` 是「需实物退换」的**旁路态**（处理完回 `RESOLVED`），不另设复杂派工 / 维修工序子流程。`OPEN→CLOSED` 直关用于无效 / 误报。`RESOLVED→IN_PROGRESS` 重开用于客户反馈未解决（呼应售后「琐碎、可能反复」）。**无财务硬关卡**（纯技术流程）。

### 引擎映射 & 兼容性（逐条）

- ➕ **新增 doc_type `SERVICE_TICKET`（+ 可选子表 `service_ticket_line`）**：一个 `Base` 子类 `__doc_types__=("SERVICE_TICKET",)` + 一次 alembic 迁移 + 一行 `WorkflowDefinition`（引擎 09 §9.1）；子表名含 `_line` + 指向父表 FK → 引擎自动识别为子表、DocEditor 自动渲 SubTableEditor 网格（引擎 01 §1.4.2、00 §4.3）。引擎无现成售后模型（引擎 02 §2.9 排除「退货 / 售后」业务）→ 全新增，但属标准加单据动作，不破坏底座。
- ✅ **轻量四态状态机 = 节点级 `allowed_roles`**：`OPEN`(SALES/SA/FAE/PM)→`IN_PROGRESS`(FAE/PM)→`RESOLVED`(FAE/PM)→`CLOSED`；用**节点级 `allowed_roles`** 控权（不依赖边级 `roles`），**规避引擎 02 D-02e 边级角色推进未校验坑**（FR-2.7）。状态推进经 `execute_transition` 三模式（创建 / 编辑 / 推进，引擎 02 §2.3）。
- ✅ **审批中心 / 待办收件箱**：`IN_PROGRESS`（待 FAE 处理）、`RESOLVED`（待 PM/FAE 确认关单）等节点单据由 `list_user_todos` 推到对应角色「我的待办」（00 §4.7、引擎 02 §2.5），FAE/PM 在待办里推进（预览→ChangeCard→commit，00 §5.6）。售后**无财务 ★关卡**，不进财务审批中心。
- ➕ **工单 → RMA 派生 / 关联（跨单据，可联动）**：`IN_PROGRESS→ESCALATED_RMA` 推进可选 `@register_transition_effect`（EXPLICIT 显式列名触发）**派生一张 04b `RMA` 草稿**（带 `service_ticket_id` 回链、客户 / 型号 / SN 预填），或**仅手工关联**已有 RMA（填 `rma_id`）。跨单据派生继承命令幂等 / 单事务 / 审计四件套（引擎 03/04）。**两条入口并存**（工单转 RMA / RMA 独立发起），靠 FK 互链。**必入 engineFlags**（跨单据派生 + 与 04b 联动）。
- ➕ **质量判定字段 = 与 04b RMA 货回好 / 坏标记同源**：`quality_verdict`（良 / 不良）落点与 04b RMA 货回 `source_marker` 品质标记、03 库存质量判定一致（蓝图 §2 FAE 质量判定）；工单内判定可作为 RMA / 库存判定的技术依据。**与 04b / 03 共用判定语义**，非新机制。
- ➕ **已挂天数 / 售后统计**：已挂天数 = 派生只读列 / 前端计算（同 04b 超期天数）；售后统计看板属「报表非元数据化」（引擎 00 D7）→ 手写路由 + 租户 / 产线作用域。**必入 engineFlags**。
- ✅ **行级隔离 + 本产线范围**：`SERVICE_TICKET` 带 `company_id`——一笔生意不跨公司（00 §1.2），香港卖的售后在香港公司、内地在内地公司，天然由公司隔离保证；FAE/PM 按 `product_line`（产线归属）+ company_id 过滤；SALES 看本人客户工单；read-privileged（BOSS / 财务总监）跨公司只读（00 §9）。
- ✅ **扫码兜底**：`serial_lot_number` 支持扫码枪录入 + 手填兜底（可 NA），符合 00 §5.4「所有 SN/LOT 字段可手工录入」。
- ✅ **留痕**：每次状态推进 + 字段变更落 WorkflowLog（字段 diff + effect 轨迹，引擎 07，`__queryable__` 业务可查）——直接解决访谈 10:367-376「处理过程要有记录、但现在不一定每条都记」的痛点。

### 业务单据推送（金蝶）
售后技术工单是**内部技术服务单**，**本身不推金蝶**（无财务凭证含义，00 §6 推送清单不含售后工单）。仅当工单升级派生 **RMA**（实物退换）时，RMA 的货回入库（其他 / 退货入库单）按 00 §6.2「其他入库单（样品 / 退货入库）」在**入库审核通过**推金蝶（入账，04b）；涉收费维修 / 退款 / Credit Note 时由财务域触发推送（05 财务），本工单只触发标记、做账在金蝶。

### 验收标准（可测） + 待甲方确认 gap
- ✅ 建单必填客户 / 型号 / 问题描述 / 负责 FAE；缺任一不能提交（`OPEN` 硬规则）。
- ✅ 工单流转 `OPEN→IN_PROGRESS→RESOLVED→CLOSED` 四态正确；`OPEN→CLOSED` 可直关（无效 / 误报）；`RESOLVED→IN_PROGRESS` 可重开。
- ✅ `IN_PROGRESS`/`RESOLVED` 工单出现在负责 FAE / PM 的待办收件箱；非本产线 FAE 看不到。
- ✅ 关闭前 `resolution_type` + `resolution_notes` + `closure_note` 必填（处理过程 / 结论留痕）。
- ✅ FAE 判「需退换报原厂」→ 进 `ESCALATED_RMA`、可派生 / 关联 RMA 号（`rma_id` 回链 `service_ticket_id`）；RMA 处理完工单可回 `RESOLVED`。
- ✅ SALES 只看本人客户工单且看不到成本 / 价格列（无价格主线 + 字段防火墙双保险）。
- ✅ 多型号工单可用 `service_ticket_line` 子表网格录入（多 SN 多行）。
- ✅ 工单按产线 / 处理方式 / 型号统计（答疑 vs 维修占比、高频故障型号）。
- gap-1：售后统计**维度全集**（是否含客户 / 季度 / 原厂维度）——默认产线 × 处理方式 × 型号三维（见 05c-1 gap-1）。
- gap-2：工单 → RMA 是**自动派生草稿**还是**仅手工关联**——默认 `ESCALATED_RMA` 推进时**派生 RMA 草稿 + 回链**（减少重复录入），由 PA/SA 在 RMA 单补全；待甲方确认是否要强制工单先行（默认两入口并存、不强制）。
- gap-3：售后**收费维修 / 报价 / 退款**是否进系统（工时费 / 维修费 / Credit Note）——本批**不做收费维修核算**（访谈未提售后收费流程，售后多为质保内答疑 / 换修）；若后续要做，扩 `repair_fee` 等字段须按 00 §8 字段防火墙处理，列待甲方。
- gap-4：**紧急度 / SLA** 是否要做倒计时硬卡（高紧急 N 小时未响应升级）——本批**只做轻量 `urgency` 标记 + 已挂天数提醒**，不做 SLA 硬卡（售后「琐碎、靠人」，访谈 10:340），列待甲方。
- gap-5：售后工单与 **04b RMA 的状态联动强度**（工单 `RESOLVED` 是否随 RMA `CLOSED` 自动触发）——默认**弱联动**（FK 关联 + 台账 / 链路对照，不强绑定双向状态机），避免两单状态机互锁；待甲方确认是否要更紧的联动。
