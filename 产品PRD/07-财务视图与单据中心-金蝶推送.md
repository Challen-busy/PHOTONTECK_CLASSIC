## 模块定位（业务价值一句话）

把「本系统业务单据 → 金蝶云星空做账」这条**跨系统写**从「人工二次录入金蝶 / 旧系统手敲 K3」（访谈 05 ERP 录入段「专人把出庫登記录入系统」）升级为**自动推送 + 留痕 + 可重推**：每张到达触发态的业务单据（采购订单 / 采购入库 / 销售订单 / 销售出库 / 销项发票 / 进项发票 / 库存调整 / 付款 / 预收预付到账）经 `@register_transition_effect` 适配器写一行 `kingdee_outbox`（**业务单号=幂等键** + `company_id`=金蝶组织 + 推送状态 RUNNING/SUCCESS/FAILED + 金蝶回执），调金蝶 WebAPI（Save→Submit→Audit），**失败可重推、绝不静默丢单**。「单据推送中心」页 = DataExplorer over `kingdee_outbox` + 重推动作。每条带 `company_id`，**6 租户各推各自金蝶组织，财务绝不合账**（总览 §6.1）。

> ★ 本模块是**全 PRD 最大的 ➕扩展点**（总览 §4.9、§6.3）：引擎当前**无任何金蝶推送能力**（引擎 00 §0.7 D7、引擎 08 §8.8）。整章为 ➕extension。**接口前提已确认（甲方 A-1，2026-06-16）**：金蝶云星空开放 **OpenAPI（appid / 应用 id 鉴权）**，各业务对象 保存(Save)/提交(Submit)/审核(Audit) 写入接口齐全，**业务单据推送在接口层面可行**——本模块推送逻辑（outbox + 适配器 + 下文 formId/字段映射）已按金蝶 OpenAPI 网关形态写定。**剩余仅为真实接入**：金蝶凭据（appid/appSecret/accesstoken）、6 公司组织码（gap 7）、各账套固定字段编码（单据类型/款项性质/业务类型等，见各单据 gap）到位后排真实对接（在此之前为已写就的设计契约 + 开关默认 OFF/dry-run）。

**与各业务模块的接力点**：本模块**不重定义任何业务单据**——单据的字段表 / 状态机在各自模块（01~05）；本模块只定义①触发哪个态推哪张金蝶单；②我方业务字段 → 金蝶字段（formId + 字段名）的映射；③ outbox 台账与重推。

### 页面 1 · 单据推送中心（outbox 台账）

**定位与使用者**：财务的「金蝶推送驾驶舱」。一张 `kingdee_outbox` 模型的只读台账网格（DataExplorer），每条推送一行，按 `active_company_id` 行级过滤（财务总监 / BOSS 看全部）。失败行高亮 + 行内「重推」动作。这是总览 §6.3「单据推送中心页 = DataExplorer over `kingdee_outbox` + 重推动作」的落地页。

**典型流程 / 场景**（端到端，引用蓝图 §7、§3.6）：
1. 某业务单推进到触发态（如出库单进 `shipped`）→ 适配器 effect 自动写一行 outbox（状态 `RUNNING`），并经命令调金蝶 WebAPI。
2. 金蝶返回成功 → 回填 `kingdee_bill_no`（金蝶单据编号）+ 状态 `SUCCESS` + 回执 JSON。失败 → 状态 `FAILED` + 错误码 / 错误信息。
3. 财务在本页看到 `FAILED` 行（工作台「推送失败待重推」卡片同源），下钻看回执 → 判断是「业务单本身错」（回原模块改单 → 改完重新推进或手动触发推送）还是「金蝶侧瞬时错 / 基础资料缺失」（直接重推）。
4. 点**重推** → 复用原幂等键（业务单号）重跑推送命令；已 `SUCCESS` 的重推自动去重返回原结果（引擎 03 §3.6 幂等回放）。

> ★ 本页**不含**新的业务硬关卡——推送是单据**已过**业务 / 财务关卡（如出库已财务放行、进项发票已审核）之后的**下游动作**；推送失败**不回滚业务单**（业务以本系统为真相源，总览 §1.2），只在 outbox 留 FAILED 行待处理。

**引擎映射 & 兼容性**（逐条 ✅/➕/❌）

- ➕ **`kingdee_outbox` 新模型** = 全新 `Base` 子类，`__queryable__ = True`（不是 doc_type，无生命周期审批，故不注册 `__doc_types__`，避免被 WorkflowDefinition 强配状态机）；一次 alembic 迁移建表。引擎无此模型 → **engineFlags**。
- ➕ **DataExplorer over `kingdee_outbox`** = 引擎 08 DataExplorer 对 `__queryable__` 模型自动出列、status 过滤、千分位、下钻可直接复用（✅ 这部分兼容）；**行级 `_company_filter` 生效**需 outbox 有 `company_id` 列（有）→ 财务限本公司、财务总监 / BOSS 看全部，符合引擎 05 §5.3。**唯一新增** = 行内「重推」动作按钮（DataExplorer 默认只读，需挂自定义行动作 → 前端扩展）。**engineFlags**。
- ➕ **重推动作** = 复用引擎 `POST /command-center/logs/{id}/retry`（仅 `FAILED` + `supports_retry`，复用原 payload + 原幂等键，引擎 03 §3.6）；outbox 行须存对应 `command_log_id` 以便一键 retry。**engineFlags**（接线 outbox↔CommandLog）。
- ✅ **幂等 / 单事务 / 留痕**天然继承：推送实现为 `@register_command`，业务单号作 `idempotency_key`，`(command_name, idempotency_key)` 部分唯一索引兜底「每键至多一个活动推送」（引擎 03 §3.4/§3.6）——**这正是「绝不静默丢单 + 重复推送去重」的引擎落点**。
- ➕ **命令模块 import 列表**：新增推送命令模块须改 `load_commands()` 硬编 import（引擎 03 D-03a）。**engineFlags**。

**表级权限（ROLE_ALLOWED_TABLES）**：`kingdee_outbox` 表对 {FINANCE, FINANCE_DIRECTOR, BOSS, ADMIN} 可见；SALES/PA/SA/LOGISTICS 默认**不可见本表**（推送是财务 / 系统域；失败经通知中心提醒相关角色回原单，而非给本表权）。是否给 PA 只读本产线 outbox = 待甲方确认。

**验收标准（可测）**
1. 业务单进触发态 → outbox 多一行 `RUNNING`，业务单号写入幂等键列。
2. 金蝶成功回执 → 该行 `SUCCESS` + `kingdee_bill_no` 回填 + 回执 JSON 留存。
3. 金蝶失败 → 该行 `FAILED` + 错误码可见；业务单**不回滚**（仍为本系统真相源）。
4. 同一业务单号重复触发推送 → 不产生第二行成功推送（幂等去重，回放原结果）。
5. 财务（本公司）能重推 `FAILED` 行；财务总监 / BOSS 重推按钮不可见 / 被拒。
6. 财务只看本 `active_company_id` 的 outbox 行；财务总监切换器 / 全公司聚合可见全部（只读）。
7. 重推成功后该行转 `SUCCESS`，`retried_from_id` 指向原行。

### 页面 2 · 业务单据推送清单 + 金蝶接口 / formId / 关键字段映射表 ⭐（本模块核心交付物）

**定位与使用者**：这是**设计契约 + 8 配置域字段映射表**，不是交互屏。供乙方实现推送适配器、供甲方与金蝶实施方对接字段时逐项签字确认。覆盖总览 §6.2 推送单据清单的全部 9（含预收 / 预付拆为 10 行）类单据。

> **接口形态总则**（金蝶云星空开放平台 WebAPI 通用约定，金蝶 API 文档各对象一致）：每个业务对象（formId）有标准操作族——`add`/`batchAdd`/`addSave`（保存新增）、`submit`/`batchSubmit`（提交）、`audit`/`batchAudit`/`billbatchAudit`（审核）、`query`（查询）、`unaudit`/`unsubmit`/`delete`（反审核 / 撤销 / 删除）。推送一张单 = 调 `add/batchAdd/addSave`（带 data 数组）→ 视需要 `Submit`→`Audit`。本系统每张单据**推一次 = 一个 `@register_command` 处理器**，内部按需链式调 新增→提交→审核，回执回填 outbox。
>
> **★ 一手源核对修正（接口形态，金蝶 API 文档「请求 URL/请求头/请求示例/响应示例」节）**：本离线文档为**金蝶云星空开放平台网关（acgw）版**，与旧 K3Cloud `ExecuteBillQuery`/`Save` WebAPI（`{"FormId","Data"}` 封装）**不同**，下列为离线文档实测形态——
> - **请求 URL（POST）= `/v2/<域>/<formId>/<操作>`**：域前缀 = `pm`（采购）/`im`（库存）/`sm`（销售）/`ar`（应收）/`ap`（应付）/`cas`（出纳）。各单据具体 URL 见 2.1 表「请求 URL」列（均从离线文档「请求 URL」字段直取）。
> - **请求体顶层 = `{"data": [ {<单据对象>}, ... ]}`**（小写 `data`，**数组**；非 `{"FormId":..,"Data":"{...}"}`）。明细子表为对象内嵌数组（如 `"billentry":[{...}]`）。
> - **请求头**（4 个，离线文档「请求头」表）：`Content-Type: application/json`、`accesstoken`（请求令牌，先换取）、`x-acgw-appid`（网关应用 id）、**`Idempotency-Key`（防重复调用，传唯一 requestId）**。
> - **★ `Idempotency-Key` 直接承载本系统幂等键**：金蝶网关原生幂等头 = 本系统**业务单号**（= 命令 `idempotency_key`），双层幂等（本系统命令层 + 金蝶网关层）天然防重复推、防静默重单（总览 §6.1 铁律 2）。
> - **响应体（回执）= `{"data": {"successCount","failCount","result":[...]}, "errorCode": "", "message": "", "status": true}`**（离线文档「响应示例」）。outbox 回填：`status=true & errorCode 空` → SUCCESS，**金蝶单据号从 `data.result[]` 解析**回填 `kingdee_bill_no`；`status=false`/`errorCode` 非空 → FAILED，错误存 `errorCode`+`message`（**非旧版 `Result.ResponseStatus.Errors`**）。

> **组织映射（company_id → org_number）**：每条推送的金蝶头字段 `org_number`（结算组织 / 库存组织 / 销售组织 / 采购组织，按单据类型不同字段名见下）由本系统 `company_id` 经「6 公司 → 金蝶组织编码」映射表解析（ADMIN 在 8 配置维护）。**6 租户各推各组织，财务绝不合账**（总览 §6.1）。组织映射表未提供 = 待甲方确认 gap 7（总览）。

**引擎映射 & 兼容性**（逐条 ✅/➕/❌）

- ➕ **每张单据一个推送命令 + 适配器 effect**：`@register_transition_effect(doc_type=<源单>, to_state=<触发态>, auto=True)` 在源单推进到触发态时 ① 写一行 `kingdee_outbox`、② 调 `@register_command("push_kingdee_<对象>")`（按 2.1「请求 URL」`POST /v2/<域>/<formId>/<操作>`，body = `{"data":[{单据对象, 子表数组}]}`，请求头带 `accesstoken`/`x-acgw-appid`/`Idempotency-Key=业务单号`，链式调 新增→提交→审核 → 回执 `status`/`errorCode`/`data.result[]` 回填 outbox）。effect 拿 `command_log_id` 挂同一审计链（引擎 04 §4.A.1）。**engineFlags**（全新适配器层）。
- ➕ **触发态语义**：入库审核 / 出库放行 / 发票审核这类**与态强绑定**用 `auto=True`（引擎 04 §4.A.2）；库存调整拆双单这类**跨单据派生**可用 EXPLICIT effect（节点 `effects:[...]` 列名）。
- ➕ **HTTP 出站 + 凭据**：引擎处理器内做 HTTP 调金蝶（appId/appSecret/账套从 8 配置凭据库取）；引擎无原生外部 HTTP 适配层 → 新增。**engineFlags**。
- ➕ **基础资料编码对齐**：金蝶按 `*_number`（编码）匹配物料 / 供应商 / 客户 / 仓库 / 单位 → 本系统主数据编码须 = 金蝶编码（2 主数据模块需标「金蝶编码」字段）；`cas_*` 单据用 `_id` 内码须先 query 换码。**engineFlags + gap**。

### 页面 3 · 推送回执详情 / 重推（单行下钻）

**定位与使用者**：财务从页面 1 某行下钻进来的只读详情 + 重推动作。展示这次推送的**请求体快照**（脱敏后的 URL + `{"data":[...]}` 体）、**金蝶返回原文**（网关响应 `status`/`errorCode`/`message`/`data.result[]`）、**重试历史**（链 `retried_from_id`），并提供「重推」「跳转源业务单」两个动作。

**典型流程 / 场景**：
1. 财务点 FAILED 行 → 进本页看金蝶错误（如「物料编码 XXX 不存在」「组织未启用」「单据类型不存在」）。
2. 判断：基础资料 / 配置缺 → 通知 ADMIN 在金蝶补基础资料后**重推**；业务单本身错（数量 / 客户错）→ 点「跳转源业务单」回原模块改单，改完重新触发推送。
3. 重推 → 调引擎 retry（复用幂等键），生成新 RUNNING，旧行留 FAILED 作历史。

**字段表**（只读下钻 + 动作）

| 字段 | 类型 | 必填 | 选择器来源 | 校验 | 默认 | 扫码兜底 | 引擎映射 |
|---|---|---|---|---|---|---|---|
| 请求体快照（URL+`{"data":[...]}`，脱敏） | json | 只读 | — | — | CommandLog.request_payload | — | 引擎命令日志（脱敏，引擎 03 §3.9） |
| 金蝶返回原文 | json | 只读 | — | — | outbox 回执列 | — | 回执存储 |
| 错误码 / 错误信息 | text | 只读 | — | — | 失败时填 | — | 金蝶网关 `errorCode`+`message` 解析（非旧版 `Errors`） |
| 重试历史 | list | 只读 | — | — | `retried_from_id` 链 | — | 引擎 retry 链 |
| 「重推」动作 | button | — | — | 仅 FAILED + supports_retry | — | — | `POST /command-center/logs/{id}/retry`（引擎 03 §3.6） |
| 「跳转源业务单」动作 | link | — | — | business_doc_type + id | — | — | 链路追踪反链（引擎 02 §2.7） |

**引擎映射 & 兼容性**
- ✅ **请求 / 响应快照** = 直接复用 CommandLog 的 `request_payload`/`result_payload`（引擎 03 §3.9 只读 API，敏感字段脱敏）；请求快照 = `{"data":[...]}` 体（含 formId 对应 URL），响应快照 = 金蝶网关 `{"status","errorCode","message","data"}` 原文。
- ➕ **金蝶回执单独留存**：CommandLog.result_payload 存命令结果，但**金蝶原始回执**（含 kingdee_bill_no / Errors）建议同时落 outbox 行的回执列，便于台账直接看（不必每次拆 CommandLog）。**engineFlags**。
- ✅ **重推** = 引擎 retry 端点（仅 FAILED + supports_retry 门控，引擎 03 §3.6）。
- ✅ **跳转源单** = 引擎链路追踪 `routers/chain.py` 沿 FK 串链（引擎 02 §2.7）。

**验收标准**：① FAILED 行能看到金蝶错误码与请求快照；② 重推按钮仅 FAILED 可见；③ 重推后旧行留痕、新行 RUNNING、retried_from 指向旧行；④ 跳转源单可达原业务单 DocEditor。

**待甲方确认 gap**：金蝶错误码 → 中文人话提示的映射字典是否需要（默认透传金蝶原文）；请求体快照里金额 / 采购价对哪些角色脱敏（本页仅财务 / ADMIN 可见，SALES 无权进入）。
