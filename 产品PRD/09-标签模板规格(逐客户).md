## 1. 定位与边界

### 1.1 一句话定位

> **本页 = 富泰出货时贴在货物上的「客户要求标签」的可配模板库**：把现在散落在各 SA Excel / 客户系统导出 PDF 里的"每家客户、每种标签（包装/外箱/内标/外标）要哪些字段、字段什么顺序、二维码怎么拼、用什么分隔符、标签多大尺寸"，收口成一套**客户 × 标签类型 × 字段映射**的元数据模板，出库时由系统据出库登记数据**自动套打**（含二维码自动拼接）。

### 1.2 三类标签来源（决定本系统要不要做模板）

回覆表 `客户标签制作方式` 列把客户标签分三类，**只有第 3 类需要本系统做模板**：

| 来源类型 | 含义 | 本系统职责 | 涉及客户（样例） |
|---|---|---|---|
| A · 客户网上系统直接打印 | 客户在自己系统里导出/打印，富泰不制作 | **不做模板**（仅出库台账记录"标签由客户系统出"） | Accelink 光迅(包装标签) |
| B · SA 同事提供（客户系统 PDF） | 客户系统出 PDF，SA 转交，富泰照打 | **不做模板**（可做"附件上传+套打位"占位，范围待定⑫） | EOPTOLINK(外箱/包装标签2)、HGO、Linktel、ZTE、SG TFC、Taclink(包装标签1) |
| **C · 自制表格配合打印机/打印程式** | **富泰自己用打印程序按客户格式制作** | **★做可配模板**（本页主体） | INNOLIGHT、XUNTE、ACCELIGHT、POTRON深圳力子、成都储翰、NB GLOBAL、SOURCE、Tri-light、HISENSE海信、FIBERHOME烽火、TACLINK(包装标签2/外箱)、各客户「公司外箱標籤」 |

> 关键：**同一客户可同时有多种来源**（如 Accelink 包装标签=A 客户系统，但「公司外箱標籤」=C 富泰自制）。模板库按 **(客户, 标签类型)** 粒度建，不按客户整体建。

### 1.3 与出库登记的关系（字段从哪来）

回覆表把每个标签表头映射到 **`出庫登記.xlsx` 字段**（`出库表字段参考` sheet，见 §3），分三种：
- **「是」对应出库表字段** → 出库时自动取数（如 `型號→P/N`、`SN/LOT#→批次`、`數量`、`供應商`、`出庫單號`、`箱號`、`原產地`、`出庫日期`、`出庫重量(Kgs)`）。
- **「否，待确认字段来源」** → 字段来源是客户系统 / Commercial Invoice / SA 邮件 / 进库包装资料 / 默认值 → **本系统需补字段或人工录入**（标 ⚠️待确认）。
- **派生/计算字段** → 如 `箱号 1/11`（本箱/总箱）、`流水号 00001`（包顺序）、`合并项 型号/品牌/COO:xxCOD:xx`（多字段拼文本）、二维码（多字段按分隔符拼接）。

## 2. 引擎映射总则（标签模板引擎 ➕extension）

> 本页所有客户模板共用同一套引擎构件。逐客户章节只列**该客户的字段子表 + 二维码拼接规则**，不重复引擎说明。

### 2.1 构件落位（对应 `00 §4` 公约）

| 业务概念 | 引擎构件 | 兼容性 | 说明 |
|---|---|---|---|
| 标签模板（一种客户标签=一条模板） | ➕ 新增 `label_template` 模型（`__queryable__=True`，带 `__doc_types__`） | ➕extension | 引擎无标签子系统；字段：`id / customer_id / label_type / company_id / size_mm / orientation / printer_profile / qr_separator / qr_field_order(JSON) / status`。 |
| 模板字段行（表头→数据源映射，多行） | ➕ 子表 `label_template_line`（`_line` 命名，FK→label_template） | ➕extension | `SubTableEditor` 网格录入（UX 公约 §1）；字段见 §2.2。 |
| 二维码拼接 | ➕ `@register_command` `build_label_qr` | ➕extension（`load_commands()` 硬编 import 须改，`00 §4.4` D-03a） | 入参=出库行+模板；按 `qr_field_order` 取值、`qr_separator` 连接，输出 QR 字符串；幂等键=`(出库单号, 箱号, 流水号)`。 |
| 出库推进→批量生成标签 | ➕ `@register_transition_effect`（EXPLICIT，挂出库「财务放行后出库」边 `effects:[gen_labels]`） | ➕extension | 按分箱明细逐箱/逐包派生标签实例，调 `build_label_qr`；派生写挂同一 `command_log_id`（`00 §4.5`）。 |
| 套打渲染 | ➕ 模板渲染器（HTML/ZPL→打印机） | ➕extension | 复用引擎 `custom_html` 逃生舱思路（引擎 08 §8.8）；尺寸/方向来自模板。 |
| 字段防火墙 | ✅ 复用引擎字段防火墙 | ✅compatible | 标签字段**只含出货信息，无成本/利润点/采购价**，故对销售可见无碍（`00 §8`）；模板配置页属 ADMIN/物流主任。 |
| 公司隔离 | ✅ `_company_filter`（模板带 `company_id`） | ✅compatible | 模板按 6 公司各自配（同一客户在不同公司可有不同抬头/默认值）。 |

### 2.2 `label_template_line` 字段表（所有客户模板通用结构）

| 字段 | 类型 | 必填 | 选择器来源 | 校验 | 默认 | 扫码兜底 | 引擎映射 |
|---|---|---|---|---|---|---|---|

| `label_header` | str | 是 | 手填 | 非空 | — | — | ✅ 子表列 |
| `source_type` | enum | 是 | cell 选择器 | ∈{出库表字段,Commercial Invoice,SA邮件,进库包装资料,默认值,客户系统,派生} | 出库表字段 | — | ✅ 子表列 |
| `source_field` | str | source_type=出库表字段时必填 | cell 选择器(出库表字段集) | ∈ §3 字段集 | — | — | ✅ FK 风格 cell 选择器 |
| `default_value` | str | source_type=默认值时必填 | 手填 | — | — | — | ✅ 子表列 |
| `in_qr` | bool | 是 | 勾选 | — | 否 | — | ✅ 子表列（是否入二维码） |
| `qr_order` | int | in_qr=是时必填 | 手填 | 唯一 | — | — | ✅ 决定二维码字段序 |
| `is_confirmed` | bool | 是 | 勾选 | — | 否 | — | ⚠️ 标"待业务确认"（回覆表全列"待确认"） |

> 二维码分隔符 `qr_separator` 放在 **模板头**（`label_template.qr_separator`），不放行（各家分隔符唯一：见 §12 汇总表）。

### 2.3 通用兼容性结论

- ✅ 现成：字段子表 = `SubTableEditor`；公司隔离 = `_company_filter`；字段防火墙无需特殊处理（标签无敏感价）。
- ➕ 扩展（必入 engineFlags）：`label_template` + `label_template_line` 模型、`build_label_qr` 命令（须改 `load_commands()` import）、出库 `gen_labels` effect、套打渲染器。
- ❌ 不兼容：无元数据化的"可视化标签布局编辑器"（字段坐标/字号/条码位置）——引擎只能到 `custom_html` 逃生舱；**首批用固定 HTML/ZPL 模板 + 字段占位符替换**，不做拖拽式布局器（列入 §13 gap）。

## 4. 入仓编号标签（富泰内部入库标签 · 全公司统一）

> 区别于「客户出货标签」：这是**入库收货环节贴在货物上的富泰内部追溯标签**（`图片5.png`），入仓编号=标签条码主键（`00 §7` 编号规则）。

| 项 | 规格 | 引擎映射 |

| 内容 | 入仓编号文本 + 一维条码（同值），如 `PR2604-048-01`（`PR{YYMM}-{seq}-{line}`，`00 §7`） | ➕ 编号规则生成（`00 扩展点`）+ 条码渲染 |
| 尺寸 | **62mm × 29mm**（来源 `图片5.png` 标签编辑器「規格 62mm x 29mm」） | ➕ `label_template.size_mm` |

| 数据源 | 入库收货单（入仓编号 effect 生成） | ✅ 出自 3 仓储 WMS 入库收货页 |
| 兼容性 | ➕extension：固定模板（单字段+条码），最简，无二维码拼接 | — |

> ⚠️ 待确认：条码码制（Code128/Code39）；是否需 QR 版（当前样图为一维条码）。

> - 流水号/包数 `00001`：按同单包顺序自增（XUNTE/ACCELIGHT/SOURCE/Tri-light）。
> - 供应商代码：取自合同号/Package ID 前缀（ACCELIGHT=Package ID 前5位；SOURCE=CONTRACT NO. 前5位；Tri-light=批次号前缀 Cxxxx）。⚠️ 提取规则各家不同，需逐家配前缀长度。
> - 失效日期 Exp = 生产日期 + 保质期（成都储翰/Tri-light/ACCELIGHT Validity Days）。⚠️ 保质期值来自 SA 邮件，本系统无源 → 需补字段。

## 14. 扫码槍 ↔ 供应商配置二维码联动（入库环节，非客户标签）

> 来源：`扫码采集补充信息.txt` + `客户二维码信息/*.pdf`（4 份 CINO FuzzyScan A680 iCode 配置码）。**这影响入库收货页的扫码工作流，非出货标签**，在此记录以便 03 仓储入库页引用。

| 项 | 内容 |
|---|---|
| 扫码槍型号 | CINO FuzzyScan **A670 / A670BT**（BT=无线，A670=有线，功能完全一致） |
| 供应商配置码 | 仅 4 家做了 iCode 配置二维码（PDF）：**Lumentum / MaxLinear / Phononic / SiFotonics**；其余供应商不需要 |
| 工作流 | 收哪家供应商货 → **先打开该供应商 PDF、扫描其 iCode 配置二维码**（设定扫码槍解析规则）→ 再扫供应商产品二维码 → 解析出字段写入入库登记 |
| 配置码性质 | iCode = FuzzyScan PowerTool3 1.00.14 导出的「Factory default + Settings」DataMatrix 配置码（非产品数据），每家解析规则不同 |

| 兼容性 | ❌ 引擎无扫码采集能力 → 前端读扫码槍键盘输入填字段；配置码切换是**线下硬件操作**，系统仅在入库页提示"按供应商切换扫码配置" |

## 15. 状态机（标签生成生命周期）

> 标签实例不是独立单据，是出库单的派生物。生命周期挂在出库流程的 effect 上（`00 决策⑥` 出库两道关之后）。

```
出库单：... → [财务放行] → [出库执行] --(effect: gen_labels)--> 逐箱/逐包生成 label_instance
   label_instance 态：DRAFT(预生成) → READY(QR已拼/可打印) → PRINTED(已套打) [终态]
```

- `gen_labels` effect（EXPLICIT，挂出库执行边 `effects:[gen_labels]`）：按分箱明细 × 客户模板，逐实例调 `build_label_qr`，写 `label_instance`（状态 READY）。
- 重打 = 幂等：幂等键 `(出库单号, 客户, 标签类型, 箱号, 流水号)`，重复生成去重（`00 §4.4` 幂等四件套）。
- 打印动作 = 前端套打 + 回写 `PRINTED`（轻量，不强制审批）。

## 16. 金蝶推送

- **标签模板/标签实例不推金蝶**：标签是物流出货附属物，无财务语义。金蝶只接收出库/发货单本身（`00 §6.2` 出库/发货单 → 收入确认源）。

## 17. 验收标准

1. **模板可配**：ADMIN/物流主任能在 8 配置「标签模板」页为 (客户, 标签类型, 公司) 建一条 `label_template` + 字段子表（`SubTableEditor` 网格），设分隔符与字段序。
2. **取数正确**：出库执行后，对一条真实出库单，§5~§12 各「是」字段能从出库登记自动填值。
3. **二维码拼接正确**：§13 表的 7 套二维码（XUNTE/ACCELIGHT/POTRON/储翰/SOURCE×2/Tri-light）按各自分隔符与字段序拼出与样例一致的串。
4. **入仓编号标签**：62×29mm，`PR{YYMM}-{seq}-{line}` 文本+条码可打印（§4）。
5. **手填兜底**：所有「待确认来源」字段在套打前可人工录入/改值（UX 公约 §4）。
6. **公司隔离**：6 公司各自的同客户模板互不串（如香港抬头 vs 内地抬头），`_company_filter` 生效。
