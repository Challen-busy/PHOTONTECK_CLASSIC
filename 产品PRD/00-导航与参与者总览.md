## 0. 开发铁律（强制，优先于一切进度；详见 13 实施）

> 本 PRD 是**业务设计**；后续**开发也必须严格落在现有引擎上**（用户 2026-06-16 反复强调）。
> 1. 严格守引擎五条：**唯一写入路径**（Command→Workflow→Domain）/注册器接入/元数据驱动/**引擎不认业务术语**/审计四本留痕。
> 2. 每个需求**先映射引擎构件**（doc_type/states/命令/effect/字段防火墙/审计）；标 **✅ 或 ➕** 的照写。
> 3. **遇 ❌**（引擎语义表达不了 或 要动核心 `execute_transition`/注册器/命令框架/审计/唯一写入路径）→ **立即停，先定"套路"**（现有积木拼 / 业务层扩展 / 等价替代），**与人确认后再动，绝不硬改引擎、绝不绕过底座**；改核心是最后手段且不得让引擎沾业务术语。
> 4. 全 PRD 每页 ✅/➕/❌ 标注即为此用：**✅/➕ 放手写，❌ 停下来。**

## 1. 产品定位与边界

### 1.1 一句话定位

> **本系统 = 富泰 6 家公司的「业务 system-of-record」**：把现在散落在 Excel / 飞书多维表 / 邮件里的 CRM、采购供应链、仓储 WMS、报关、单据/标签全流程**收口到一个有权限、有留痕、有状态机**的系统里；财务做账继续在**金蝶云星空**完成，本系统**推业务单据给金蝶 + 拉金蝶已确定的报表/视图**，不实时回填、不算成本（蓝图 §0、§1.2）。

### 1.2 系统做什么 / 不做什么（硬边界）

| 维度 | 本系统（业务 of-record） | 金蝶云星空（财务做账） |
|---|---|---|
| CRM | 线索→商机→报价（PM 定利润点）→合同/订单、跟进、认证、标书 | — |
| 采购供应链 | 询价、PO、备货、样品 SDN、RMA、在途、付款申请（**发起**） | — |
| 仓储 WMS | 入库、库存（批次/SN/LOT/状态/库位/标记）、出库、调拨、盘点、标签 | — |
| 报关 | 报关单（进/出/退运）、退运 180 天监控、报关费/证补录 | — |
| 财务 | 应收/应付**只读视图**、预收/预付**到账确认标记**、Credit Note、对账 | **凭证、应收应付总账、成本核算、收付款执行** |
| 单据流转 | 生成业务单据 + **推送给金蝶** + 链路追踪 + 推送回执 | 接收单据做账、生成凭证 |
| 报表 | 经营 KPI（转化/成单/毛利，产线×工程师）、看板、提成 | 财务报表（本系统**拉取已确定视图**展示，不重算） |

**明确不做**（蓝图 §0）：①财务做账；②历史数据迁移（只建期初）；③AI 功能（甲方明确要确定性传统工具）；④公司间联动——**一笔生意绝不跨 2 家公司**，无公司间调拨/转销。

### 1.3 与金蝶的边界图（蓝图 §1.2）

```
[本系统：业务 system-of-record]  ──单据推送(outbox)──▶  [金蝶云星空：财务做账]
  线索/商机/报价/合同/订单                                    凭证 · 应收应付总账 · 成本 · 收付款执行
  采购/入库/库存/出库/调拨/盘点                  ◀──拉取已确定报表/视图(只读展示)──
  报关/标签/单据
  应收应付「视图」(只读, 展示+提醒, 不硬拦)
```

> 关键不变量：**本系统是业务操作的真相源；金蝶只接收单据做账**。本系统不实时回填金蝶过账结果，只在需要时**拉金蝶已对外确定的报表/视图**（如已确认应收余额）做展示。详见 §6。

## 4. 引擎映射通用公约（下游模块逐页"引擎映射"按本节落位）

> 总原则（引擎 00 §0.2 六哲学）：**业务 = 状态机 + 字段变更 + 副作用**，三者各有归属互不耦合；**唯一写入路径** `Command → Workflow → Domain`（引擎 00 §0.4，INV-4）；**元数据优先于代码**（INV-3）；**写必留痕**（INV-5）。下游每页都按 ✅compatible / ➕extension / ❌incompatible 标兼容性。

### 4.1 业务单据 → `doc_type`（注册模型）

- 每个业务单据 = 一个 `Base` 子类，类属性 `__doc_types__ = ("KEY",)` 即注册为可寻址单据；子表/主数据/日志用 `__queryable__ = True`（引擎 01 §1.3）。不声明的模型默认隐身。
- 单据类型键全局唯一（冲突启动即 `RuntimeError`）；加新单据 = 一个模型类 + 一次 alembic 迁移 + 一行 `WorkflowDefinition`（引擎 09 §9.1）。
- 命名约定：`*_number` 列自动参与模糊搜索且创建时自动生成前缀唯一标识（引擎 02 §2.3 `_auto_fill_required_fields`）；显示名优先列 `name/full_name/short_name/code/sku/*_number` 用于 FK 标签解析（引擎 01 §1.4.2）。

### 4.2 流程 → `WorkflowDefinition.states`（JSONB 状态机）

- 一个单据的**整套生命周期 = 一个 JSONB 字段** `states:[StateNode]`（引擎 02 §2.2）。节点字段：`code/name/is_initial/is_terminal/allowed_roles/editable_fields/description/hard_rules/hooks/effects/next[]`；边字段：`to/label/editable_fields/hard_rules/effects/roles`。
- 唯一写入口 `execute_transition` 三模式（引擎 02 §2.3）：**创建**（doc_id 空→进唯一 `is_initial` 态）/ **编辑**（to_state 空或等于当前态→只改字段，editable=所有出边并集）/ **推进**（to_state 在 next 中→改字段+切态，editable=命中边白名单）。
- **角色闸**：节点 `allowed_roles` + 边 `roles`（ADMIN/BOSS 绕过）。⚠️引擎 02 D-02e：边级 `roles` 在 preview/枚举强制、但**直接调 execute_transition 的推进路径当前只校验节点级**——下游凡依赖边级角色的关卡，须在 PRD 标注「需引擎补齐 FR-2.7」并列入 engineFlags。

### 4.3 多行录入 → 子表 `SubTableEditor`（Excel 式网格）

- 子表 = 表名含 `_line` / `_entry` 且有指向父表 FK 的模型（引擎 01 §1.4.2、引擎 08 §8.3）。前端 `SubTableEditor` 自动渲成多行网格：脏跟踪 `_new/_modified/_delete`、自动 `line_number`、`quantity*unit_price→total_price`。
- **凡蓝图/访谈里现在用 Excel 多行登记表的**（PO 明细、SO 型号明细、入库批次行、出库拣货批次、报价阶梯数量-单价、盘点明细等），一律落为子表 + `SubTableEditor`，对应 UX 公约 §5（贴合"现在用 Excel 登记表"的习惯）。

### 4.4 跨模块写 → 注册命令 `@register_command`（execute_command）

- 任何跨模块/跨单据的写都路由到具名命令处理器（引擎 03 §3.3）。引擎自带四件套：**幂等键去重**（`(command_name, idempotency_key)` 部分唯一索引）、**单事务**（处理器写+日志一起提交/回滚，处理器不得自管事务）、**命令日志**、**统一错误信封**。
- 流程推进本身也是注册命令（`workflow_transition`/`workflow_commit_card`，以 `manage_transaction=False` 跑在命令事务里），故状态机推进天然继承幂等/事务/审计（引擎 00 §0.4）。
- ⚠️ 命令发现是 `load_commands()` 硬编 import 列表（引擎 03 D-03a）：新增命令模块须改这段 import——下游新增命令时在 engineFlags 标 ➕。

### 4.5 跨单据派生 → 注册副作用 `@register_transition_effect`（effect）

- 一单推进自动派生另一单（如 SO 推进→生成采购通知、出库推进→扣库存、入库审核→结成本）= 注册 effect（引擎 04 §4.A）。两种派发：**AUTO**（匹配 `(doc_type,to_state)` 自动跑，适合与态强绑定的扣库存/结成本）/ **EXPLICIT**（节点/边 `effects:[...]` 显式列名才跑，适合跨单据派生）。
- effect 拿得到 `command_log_id`，派生写挂同一命令审计链。**新业务流必须用 effects/commands，禁止在流程定义里写内联写钩子**（引擎 04 §4.E、§09 C-2；`hooks` 写能力默认关闭）。
- 校验同理：领域硬约束 = `@register_transition_validator`（返回失败串列表即阻断，引擎 04 §4.A.1）；声明式只读断言 = 节点/边 `hard_rules`（rules.py DSL，引擎 04 §4.B）。

### 4.6 权限 → 角色表 + 字段防火墙 + 公司隔离（三正交维度）

- **表级**（维度2）：`ROLE_ALLOWED_TABLES` 角色→可见表集；privileged 角色（`FULL_ACCESS_ROLES`）绕过；未知角色落最小默认集（引擎 05 §5.3、引擎 01 §1.5）。
- **字段级防火墙**（维度2b）：序列化时按 (表×角色) 遮蔽——**买价/卖价/成本/利润点列对无相应权角色删**（在 query 与 schema 两路一致执行，引擎 01 §1.5、引擎 05 §5.3）。本 PRD 隐藏清单见 §8。
- **行级公司隔离**（维度3）：`_company_filter(user)` privileged→看全部，否则限本租户；模型有 `company_id` 列才生效（引擎 05 §5.3、引擎 01 §1.5）。**财务绝不合账；跨公司只读汇总仅 privileged**。
- **超管闸**（维度1）：`is_admin` 门控所有元数据管理端点（引擎 05 §5.3）。

### 4.7 审批中心 → `list_user_todos` / `list_user_actions`

- 「我的审批 / 审批中心」统一收件箱 = `list_user_todos`（按角色+租户过滤、排除终态、附摘要与可触发动作）+ `list_user_actions`（可触发的状态推进），引擎 02 §2.5、引擎 08 §8.3。
- **所有 ★ 财务/业务硬关卡都作为流程的一个状态节点**（`allowed_roles=[FINANCE/PA/...]`），由对应角色在审批中心推进。决策⑤吸收的关卡：采购审批、入库审核、进项发票审核、出库财务放行、备货会审、SO 审核、预收/预付到账确认。
- 审批通过/驳回 = `execute_transition` 推进到下一态/打回态，经 `preview_transition`→ChangeCard→`commit_card` 两步确认（引擎 02 §2.4、引擎 08 §8.3）。

### 4.8 审计四本（写必留痕，INV-5）

每个写动作至少落一条对应日志（引擎 07）：**CommandLog**（命令幂等/事务/回放）· **WorkflowLog**（状态变更+字段 diff+effect 轨迹，`__queryable__` 业务可查）· **AgentLog**（本项目不做 AI，基本不产生）· **WorkflowDefAuditLog**（配置变更前后快照）。下游"业务单据推送"与"操作日志审计"页直接消费这四本。

### 4.9 通用 UI 外壳（前端落点，引擎 08）

- 列表/台账 → **DataExplorer**（按首行 introspection 自动出列、status 自动过滤、金额千分位、只读 + 下钻 + CSV 导出）。**大台账（PO 总表/SO 总表）= DataExplorer 只读网格 + 下钻 DocEditor**（引擎 08 §8.3）。
- 单据表单 → **DocEditor**（`/api/schema` 列→字段控件，fk→可搜 cell 选择器，子表→`SubTableEditor`；提交走 预览→ChangeCard→commit）。
- 流程图 → **FlowEditor / WorkflowActions**（运行态+管理态，消费 states+node_positions）。
- ⚠️ 引擎已知缺/弱（下游大概率 ➕扩展而非 ❌）：报表非元数据化（手写路由，引擎 00 D7）· 金蝶推送（引擎无→需新增 outbox+适配器命令，§6）· 标签/单据模板引擎（仅 `custom_html` 逃生舱→需新增可配模板子系统）· 扫码枪/OCR 采集（无→前端输入增强填字段）· 物流API（无→占位）· **多公司单用户切换**（`UserCompanyAccess` 当前死代码，引擎 05 D-05c→需扩展，见 §9）· 大台账跨独立单据内联编辑（DataExplorer 只读→台账=只读网格+下钻 DocEditor 通常可兼容）。

## 5. UX 公约（全 PRD 统一）

1. **表单网格（Excel 式多行）+ 行内 cell 选择器**：多行数据录入一律用网格，不逐字段填空（贴合他们现在用 Excel 登记表的习惯，访谈 08「我只拉我的数据」、访谈 09 订单型号-数量-单价多行）。对应引擎 `SubTableEditor`。
2. **「参与者视角」优先**：每个模块先写"角色→做什么/何时/数据范围/限制"（本文 §3 矩阵为模板）。
3. **采购侧进价/成本/采购价对销售端（SALES+SA）字段级隐藏；报价利润点/毛利对 SALES+SA 可见（Q18）**：见 §8（引擎字段防火墙，引擎 01 §1.5）。
4. **扫码字段一律支持手填兜底**：扫码枪只能扫部分，其余手填（蓝图 §1.3、访谈 03 OCR/扫码枪讨论）；所有 SN/LOT/入仓编号字段都要可手工录入。
5. **状态用药丸（StatusPill）、金额千分位、时间 ISO 格式化**——前端壳默认行为（引擎 08 §8.3），下游不必重复定义。
6. **预览→确认卡→提交**：所有写经 ChangeCard 二次确认（字段 diff from→to + 检查 + 建议），降低误改（引擎 08 §8.3）。

## 6. 金蝶对接总则（下游"业务单据推送"章节统一遵循）

> **接口前提已确认（甲方 A-1，2026-06-16）**：金蝶云星空开放 **OpenAPI（appid / 应用 id 鉴权）**，各业务对象 **保存(Save) / 提交(Submit) / 审核(Audit)** 写入接口齐全——**业务单据推送在接口层面可行**，§6 推送整章（outbox + 适配器 + 07b formId/字段映射）**可落地、推送逻辑已写定**。**剩余仅为真实接入**：金蝶凭据（appid/appSecret + accesstoken 换取）、6 公司金蝶组织码（company_id→org，gap 7）、各账套固定字段编码（单据类型/款项性质/业务类型等，见 07b gap）到位后即排真实对接（在此之前推送为已写就的设计契约 + 默认 OFF/dry-run）。

### 6.1 三条铁律

1. **本系统推单 + 拉已确定报表/视图，不实时回填**：本系统把业务单据推给金蝶做账；不轮询金蝶过账明细回写本系统。需要财务结果时**拉金蝶对外已确定的报表/视图**（如已确认应收余额）做只读展示（蓝图 §0、§1.2、§7）。
2. **业务单号 = 金蝶单据编号，做幂等键**：推送以本系统业务单号作为幂等键（对应引擎命令 `idempotency_key`），重复推送自动去重、失败可重推（引擎 03 §3.6、蓝图 §7）。**幂等/静默缺行**是已知坑——推送必须留 outbox 行 + 回执，不允许静默丢单。
3. **组织隔离 vs 统一视角**：每条推送带 `company_id`（金蝶组织映射）；本系统 6 租户各自推各自金蝶组织，**财务绝不合账**；管理层统一视角只在本系统侧只读汇总（§9）。

## 8. 字段级隐藏清单（成本 / 利润点 / 采购价对销售）

> 引擎字段防火墙在 query 与 schema 两路一致遮蔽（引擎 01 §1.5、引擎 05 §5.3 维度2b）。配置以 (表, 列, 可见角色) 集表达；本表为业务侧配置清单，下游凡含这些字段的页须标注。

> **★甲方 Q18（2026-06-16）：销售（SALES）与销售助理（SA）同一可见层——报价利润点/毛利对二者都可见（报价决策用）；但采购侧进价/成本明细对销售端（SALES+SA）仍隐藏。** 防火墙边界由此从"利润点对 SALES 隐藏"翻转为下表。

| 字段类 | 典型列 | 对谁隐藏 | 对谁可见 | 一手源 |
|---|---|---|---|---|
| **报价利润点**（报价决策用）| 报价单 `profit_point`、阶梯子表 `unit_profit_point` | —（**不对销售端隐藏**，Q18 翻转）| **SALES、SA**（同层，报价决策）+ PM、财务、ADMIN、BOSS | 甲方 Q18（2026-06-16）；蓝图 §3.2 |
| **报价毛利/毛利率**（销售侧 KPI/报价用）| 报价/SO 毛利、KPI 看板毛利率列 | —（**不对销售端隐藏**，Q18）| **SALES、SA** + PM、财务、ADMIN、BOSS | 甲方 Q18 |
| **采购侧进价/成本明细**（防火墙保留隐藏）| 对原厂询价价（`supplier_inquiry_line.unit_price`/`commission`）、PO 单价/采购成本（`purchase_order(_line).unit_price/total_price/total_amount`）、入库批次成本、采购在途单价 | **销售端（SALES + SA）** | **PA、PM、财务、财务总监、ADMIN、BOSS**（管理）| 甲方 Q18（采购↔销售防火墙保留）；蓝图 §2、访谈 07/08 |

> **Q18 边界要点（务必区分两类价）**：①**报价侧的利润点/毛利**=销售做报价决策要用的 → **SALES 与 SA 同层，二者都可见**（取消原"利润点对 SALES 隐藏"）；②**采购侧的进价/成本明细**（对原厂询价价、PO 单价/采购成本、入库批次成本、采购在途单价）=采购端真实买价 → **对销售端 SALES+SA 一并隐藏**，仅 PA/PM/财务/管理可见。即"防火墙"墙体是**采购买价**，不是"利润点"。
> 落点：在业务侧 `BUY_TABLES/SELL_TABLES/BUY_PRICE_FIELDS/SELL_PRICE_FIELDS/_can_view_*` 集合配置（引擎 01 §1.8、引擎 05 §5.8）——**把 `profit_point`/`unit_profit_point` 移出"对销售隐藏"集（SALES+SA 可见）**；**采购买价列（supplier_inquiry/PO 单价、入库成本、在途单价）保持对 SALES+SA 隐藏**。下游报价单/PO/在途/台账页必须按此显式标字段防火墙。卖价对客户的可见性细则仍待甲方确认。

## 9. 跨公司只读汇总与「用户-公司-角色」三元授权

### 9.1 多租户隔离（蓝图 §1.1、§5.6）

- **6 家公司 = 6 个独立租户**，业务完全相同、数据互相隔离；所有业务数据带 `company_id`。**财务/账目绝对不合并**。
- 仅 **2 类区域配置**：香港 / 内地（差异：币种、报关规则、发票抬头/模板、税率）。落点：`Company` 主数据带 `region(HK/内地)`，发票模板/编号前缀/税率按区域配置（8 配置域）。

### 9.2 跨公司只读汇总（管理层）

- **跨公司只读汇总 = {`BOSS` 管理层, `FINANCE_DIRECTOR` 财务总监}**（决策A）：BOSS 看经营汇总、财务总监看财务口径汇总。引擎落点：把这两个角色加入"行级只读 privileged"集，`_company_filter` 对其返回 `None`（看全部，引擎 05 §5.3）；聚合经 `/api/aggregate`（引擎 01 §1.4.2）。**普通 `FINANCE` 不在此集**——严格本公司账套（须从引擎默认 privileged 集移出，否则违反"财务绝不合账"§5.6）。
- ⚠️ 报表非元数据化（引擎 00 D7）：跨公司经营 KPI/汇总看板是**手写路由 + 租户作用域**，引擎只提供脚手架 → 6 报表模块的看板必入 engineFlags（➕）。

### 9.3 用户-公司-角色三元授权（多账号场景）

- **决策B（2026-06-16）**：一个用户可被授权操作**多家公司，且在各家角色相同**——用户的 `role` 单一、跨所开通公司一致，**不需要 per-company 角色**。"开通哪些公司"由**管理员按用户可配置**（授权 = 用户 × {公司集}）。
- ⚠️ **引擎现状**：`UserCompanyAccess`（多公司授权表）已建表+seed，但 `_company_filter` **从不读它**=死代码（引擎 05 §5.3 D-05c）；`UserAccount` 单 `company_id` 只支持"一人一公司"。
- → **必做 ➕扩展**（写进 9 账号管理）：① 接线 `UserCompanyAccess` 作"用户×公司"授权（角色取 `UserAccount.role`，各公司相同，**无需 per-company 角色列**）；② 管理员页配置每个用户**开通**哪些公司；③ 会话加载 `active_company_id`（当前活动公司）+ 顶部**公司切换器**，仅可在已开通公司间切换；④ `_company_filter` 改为：read-privileged(BOSS/财务总监)→看全部只读、普通用户→限 `active_company_id`。**必入 engineFlags**。
- 会话安全旁注（引擎 05 D-05e/f）：当前签名 Cookie 无服务端吊销、口令单一全局盐——上生产前整改，下游 9 账号管理须如实标注（非"实时踢人"）。
