## 模块定位（业务价值一句话）

把现在**靠翻历史 Excel 查 HS/原产地、纸面钉装报关材料、线下记忆盯 180 天退运、货代账单后到才人肉摊成本**的整条报关作业（调研证据 1-2/1-3/1-4、7-1、W-2），收口为**一张报关单（商品明细子表从出入库批次自动聚合为真相、按报关地自动选中国/香港 HS 口径）+ 一套确定性合规校验（缺 HS 双码/中文品名/ECCN → 申报态阻断 + 自动派 PA 补录）+ 退运 180 天倒排预警 + 「香港卖的退香港」公司一致性硬校验 + 报关费后补回写批次到岸成本**，做到：报关单据自己做、系统只跟踪进度（决策⑪）、全程留痕、跨公司绝不合并报关（蓝图 §5.6、调研证据 7-4）。

> 边界重申（决策⑪）：报关**动作自己做**（自己找报关行 / 自己报），系统**不对接海关单一窗口、不接政府系统**；本系统只负责建单、聚合明细、合规拦截、退运倒计时、费用回写、（占位）接顺丰 API 查货物进度。**报关单据默认不推金蝶**（仅报关费/运费随入库/出库单推送做账，HS/原产地随出入库单流转，见「金蝶推送」节）。

## 06-1 报关单（进口 / 出口 / 退运 + 商品明细子表）⭐

### 定位与使用者
物流主任在**跨境货到 / 跨境发出 / 进口货退原厂**时建立的主单据。一张报关单 = 一票报关（可合并同公司同方向的多张出/入库单，调研证据 CUS-P0-02），头部记方向/关联单据/报关行或自报/贸易条款/状态，**商品明细子表从关联出入库批次按「PO×型号×HS×原产地」自动聚合为真相**（替代调研证据 1-6「几百项…291 项…按 PO 来分」的人肉简化），不逐行手抄。

### 典型流程 / 场景（端到端，引用蓝图 P08 / P05）

> ★ = 本模块的硬关卡（合规阻断 / 退运一致性）。注意：报关域**无财务硬关卡**（区别于出库两道关，决策⑥）；★ 在此指**合规与海关规则闸**。

**A. 进口报关主路径**（调研证据 §四.1、SOP §四-1）
1. 跨境入库单产生（运单=顺丰/FedEx/DHL 等跨境快递）→（占位）系统按规则预判「需进口报关」（CUS-P0-07，本系统默认**不自动派单**，物流主任顺手建，SOP §四「系统目前不会自动提醒…全靠你自己想着来建」；自动预判列为 gap）。
2. 物流主任「报关管理」→「报关单」→「新建报关单」→ **方向=进口** → 选关联**入库单**（可多选合并一票）。
3. 点「创建并聚合明细」：系统按 PO×型号×HS×原产地从批次聚合明细行，金额取入库批次采购价（**对 SALES 隐藏**，字段防火墙）；缺申报价值的行**粉红高亮 + 黄框提醒**（先建单后补价，SOP §四-1）。
4. 逐行核对；缺 HS（按报关地选中国口径）/中文品名/原产地/ECCN 的行标红「缺」（现可建可改，**点申报才卡**）。
5. 受管制（ECCN）行挂进出口证（06-4）；「资料清单」核对 invoice/装箱单/运单缺件（06-6）。
6. ★**点「申报」**：系统一连串拦（见状态机硬规则）——空明细 / 申报价值缺失 / **合规五件套缺失（逐行红字 + 自动派 PA 补录待办）**。全过 → `已申报`（记时间）。
7. 海关：抽中查验 → `查验中`（录查验记录，CUS-P1-04）；放行 → `已放行`（记时间，**报关单号回写入库单**，替代调研证据 W-5 人工换号）；退回 → `退单`（来源行释放，改资料后「重新申报」回草拟）。
8. 货代账单后到 → 「登记费用」（**仅进口已放行单**有此动作）按比例分摊回写各入库批次到岸成本（06-3）。

**B. 出口报关主路径**（调研证据 §四.2、SOP §三+§四）
- 销售出库单（**财务已审 + 跨境**）→ 物流主任新建出口报关单 → 选**出库单** → 明细聚合（价值取发票价，贸易条款自销售订单带出，调研证据 S-2）→ ECCN 管制行确认+挂证 → 生成盖章版 invoice+PL 成套资料（调研证据 W-7）→ 申报 → 放行 → **报关单号回写出库单/装箱单**（把发票号 `I…` 换成报关单号 `CD…`/`E…`，调研证据 W-5）。

**C. 退运（RMA→退运报关）路径**（蓝图 P05、调研证据 §四.3、SOP §四-1 退运特别说明）
- RMA 单（`SALES_RETURN`）处置=退原厂 → 系统检查来源批次**是否关联进口报关单** → 是则强制走退运报关：方向=**退运出口**（香港货退香港也用此方向，无「退关进口」，SOP §四-1）。
- 退运单**必填「原进口报关单」**（下拉只列**本公司、已放行**的进口单 → 物理上保证「香港卖的退香港」，SOP §四-1/红线3）；选完自动算「退运截止日期 = 原进口放行日 + 180 天」（可手改但必填，SOP §四-1）。
- 退运单**不自动聚合明细**，明细在详情页**手工补**（型号/数量必填，HS/中文品名/原产地留空则按主数据带出，SOP §四-2）。
- 详情页显示 180 天倒排时间轴；工作台退运预警（06-2）。

**D. 免报关短路径**（调研证据 1-5、CUS-P0-07）
- 香港本地（香港寄香港 / 本地司机送）收发货 → 单据标「本地免报关」→ **不产生报关单/不进报关流**，列表灰标，报关同事零操作（调研证据 1-5「看到 local 他就不用弄了」）。

### 单据状态机 `CUSTOMS_DECLARATION`

> 基线对齐 SOP §四「草拟 → 已申报 →（查验中）→ 已放行；退单后可重新申报回草拟」。

| 状态 code | 名称 | 可操作角色 | 可编辑字段 | 硬规则 / 关卡 | 下一状态 |
|---|---|---|---|---|---|

| `DECLARED` | 已申报 | LOGISTICS_LEAD | 仅状态备注（明细只读，「仅草拟可改」，SOP §四-2） | ★**申报闸**（推进前 validator 顺序拦）：①明细非空；②退运 `return_deadline` 必填；③逐行 `declared_amount` 非空；④逐行主数据**合规五件套**（HS 报关地码/HS 原产码/原产地/中文品名/ECCN）非空，缺则阻断 + 自动派 PA 补录（调研证据 V-1/V-9，SOP §四-3） | `INSPECTING`（查验）/ `RELEASED`（直接放行）/ `RETURNED`（退单） |

## 06-2 退运监控（180 天倒排预警 + 香港卖退香港硬校验）

### 定位与使用者
退运不是独立单据，而是**报关单 `direction=REEXPORT` 的方向 + 一块倒计时监控**。物流主任天天扫工作台「退运临期」卡 + 报关单列表「退运限期」列，盯哪票货快到 180 天大限（蓝图 §5.4、调研证据 7-1、SOP §四-5）。**系统只飘红预警，不自动拦、不替办**（SOP §四-5「全靠你盯」）。

### 典型流程 / 场景
1. RMA（`SALES_RETURN`）处置=退原厂 → 系统检查来源批次是否关联进口报关单 → 是则强制发起退运报关单（06-1 路径 C）。
2. 退运单挂回原进口报关单，`return_deadline` = 原进口放行日 + 180（可改为海关批文日 + 180，调研证据 CUS-P0-03）。
3. `remaining_days` = `return_deadline` − today（计算字段，每次查询/工作台轮询实时算）。
4. 颜色预警（SOP §四-5）：绿 `剩 N 天`（>90）/ 金（30~90）/ 红（<30 或「已超期 N 天」）/ 绿「已放行」（办妥）。
5. ★**香港卖退香港**：退运单「原进口报关单」下拉物理限本公司已放行进口单 → 跨公司物理选不到（调研证据 7-2/V-4，SOP 红线3）。
6. 超 180 天未放行：单据标红 + 管理层通报，**关闭须填超期原因**，不允许静默关闭（调研证据 V-6）。

### 字段 / 计算

| 字段 | 类型 | 说明 | 引擎映射 |
|---|---|---|---|
| 退运截止日期 `return_deadline` | date | 默认原进口放行日+180（SOP §四-1） | ➕ 列（06-1 头） |
| 剩余天数 `remaining_days` | 计算 | `return_deadline` − today（不入库，查询时算） | ➕ 计算字段（前端 / `/api/query` 派生；或 effect 缓存日更） |
| 退运状态色 | 计算 | <30 红 / 30~90 金 / >90 绿 / 已放行绿 / 超期红 | 前端 StatusPill 着色（引擎 08 §8.3 状态药丸） |
| 海关批文号/日期 | string/date | 180 可改起算（CUS-P0-03） | ➕ 列（06-1 头） |

### 引擎映射 & 兼容性
- ➕ **退运为报关单方向**：复用 06-1 `CUSTOMS_DECLARATION` doc_type，`direction=REEXPORT`，不另建单据（蓝图 §3.5 方向枚举含退运）。
- ➕ **180 天计算字段 + 预警**：`remaining_days` 非持久列 → 两种落法：①前端 `DataExplorer`/工作台卡按 `return_deadline` 实时算并着色（轻量，推荐）；②若要「到期推送提醒」则 ➕ 一个日更 effect/命令写通知中心（§00 通知中心）。引擎无定时器 → 退运到期提醒走**工作台轮询 + `list_user_todos`**（引擎 08 §8.3 待办徽标轮询）而非后台 cron，标 ➕。
- ➕ **香港退香港硬校验**：`origin_declaration_id` 下拉的租户过滤天然由 `_company_filter` 物理限本公司（引擎 05 §5.3），**额外加一条 `@register_transition_validator`**（退运目的公司=原报关公司，调研证据 V-4）双保险。
- ➕ **超期关闭须填原因**：「关闭」边 `hard_rules`/validator 要求超期时 `notes` 非空（调研证据 V-6）。
- ➕（与 02 RMA 联动）**RMA→退运自动判**：RMA 处置=退原厂时查来源批次进口报关引用 → 强制生成退运待办（调研证据 CUS-P0-03）；落 02 采购/RMA 模块的 effect，本模块提供退运单入口。
- ❌ 无。

### 验收标准 + gap

## 06-3 报关费 · 进出口证后置补录（挂入库批次或报关单）

### 定位与使用者
**报关费/运费货代账单后到**（货物入库之后才来），物流主任在**已放行的进口报关单**上「登记费用」，系统按比例分摊回写各入库批次的到岸成本（蓝图 §3.4 批次「报关费/运费」列、调研证据 CUS-P0-06、SOP §四-6）。来源可来自 HK 报关网站 / PA 到货前通知（任务描述），挂入库批次或报关单。**出口方向无此动作**（SOP §四-6「费用回写只支持进口报关单」）。

### 典型流程
1. 货代/报关行账单到 → 进已放行进口报关单详情 →「登记费用」。
2. 填总报关费 / 总运费（至少一项，SOP §四-6）+ 备注（货代账单号）+ 发生日期 → 「分摊回写并推送」。
3. 系统按「申报金额占比」或「数量占比」（可选）摊到关联各入库批次「到岸附加成本」（调研证据 CUS-P0-06）。
4. 推金蝶做**增量更新**（随入库单，不做账，本系统只归集）。
5. 两张账单先后到 → 分两次登记，系统累加留痕（SOP §四-6）。

### 字段表 — 费用子表 `customs_fee_line`（网格录入）

|---|---|---|---|---|
| 费用类型 `fee_type` | enum | 是 | 报关费/关税/增值税/查验费/运费/其他（CUS-P0-06） | ➕ 列 |

| 发生日期 `incurred_date` | date | 否 | — | ➕ 列 |
| 分摊方式 `allocation_basis` | enum | 是 | 申报金额占比 / 数量占比 | ➕ 列 |

### 验收标准 + gap

### 集成形态（接好等配置，分三层）

**① 数据层 —— `shipment_tracking`（运单轨迹快照表，`__queryable__`，非 doc_type）**
一个运单号一行（头），轨迹节点为其子行（`shipment_tracking_node`）。轨迹是承运商侧事实快照（拉取写入 / webhook 回推写入），**非业务审批单**，故不挂状态机、不进审批中心（与 `kingdee_outbox` 同范式：`__queryable__=True`、不声明 `__doc_types__`，避免被 `WorkflowDefinition` 强配生命周期，引擎 01 §1.3 / 07-金蝶 §页面1 同款落法）。

**② 适配器层 —— 顺丰开放接口命令（鉴权占位，等配置）**
顺丰丰桥 / 顺丰开放平台 WebAPI 通用约定：路由服务（`EXP_RECE_SEARCH_ROUTES` 查路由 / 轨迹）+ 订阅服务（`EXP_RECE_PUSH_ROUTES` 路由推送订阅），报文经 `appKey + 时间戳 + msgData` 做 `MD5(msgData + timestamp + checkWord)` 数字签名 + Base64。**本系统把每类调用封一个 `@register_command`**，签名 / `base_url` / `customerCode` 全部从 8 配置域读（未配置 → 命令直接返回「集成未启用」软结果，不抛错）：

| 命令（`@register_command`） | 顺丰接口（服务码占位，待顺丰确认版本） | 入参 | 出参 / 副作用 | 鉴权占位 |
|---|---|---|---|---|
| `sf_query_tracking` | 路由查询 `EXP_RECE_SEARCH_ROUTES`（按运单号查全程轨迹） | `tracking_number`、`direction`、关联单据号 | 解析节点 → upsert 写 `shipment_tracking` + `shipment_tracking_node`，回填 `current_status`/`last_event_time` | 签名用 `SF_APP_KEY/SF_CHECK_WORD/SF_CUSTOMER_CODE`（8 配置域，**留空占位**） |
| `sf_subscribe_tracking` | 路由推送订阅 `EXP_RECE_PUSH_ROUTES`（注册我方回调 URL，顺丰主动推节点） | `tracking_number`、回调 URL（本系统 webhook 入口） | 登记订阅关系（`is_subscribed=true`）；顺丰后续推送由 webhook 落库 | 同上 + 回调验签 token（占位） |
| `sf_tracking_webhook`（入站，非 register_command，是路由）| 顺丰 → 我方 `POST /integrations/sf/tracking-callback` | 顺丰推送报文 | 验签 → 解析节点 → upsert 轨迹（与 query 同一 upsert 逻辑复用） | 校验顺丰回调签名（占位 token） |

> 拉取触发（三选一，配置时选，默认①）：① **按需查询**（前端「刷新轨迹」按钮 / 详情页打开时调 `sf_query_tracking`，最省额度、推荐首期）；② **订阅推送**（发货时 `sf_subscribe_tracking`，顺丰主动 webhook 回推，最实时，需公网回调）；③ **定时轮询**（在途运单批量 `sf_query_tracking`，引擎无 cron → 走工作台轮询触发或外部定时器，**占位不实现**，同 06-2 退运提醒落法）。

**③ 展示层 —— 前端「物流跟踪」卡片 / 货物进度时间轴（用户可见、可发现）**
出库单 / 入库单 / 报关单详情页**始终内嵌**「物流跟踪」卡片（见上「UI 占位态 / 开通后即用」），另在物流角色工作台有可发现入口：竖向时间轴（节点时间 + 地点 + 操作描述 + StatusPill 着色：揽收 / 运输中 / 派送中 / 已签收 / 异常），数据 = `shipment_tracking_node` 按运单号 + 时间排序。**SF 未接 / 未命中**时卡片**不消失**，退化为「占位时间轴灰骨架 + 手填进度备注输入（来源标记 `MANUAL`）」，顶部显示状态徽标「**功能已就绪 · 待顺丰 OpenAPI 配置开通**」灰条（而非简单「未启用」字样，强调功能存在、可发现）—— 不报错、不阻塞（§00-5 扫码字段手填兜底同源原则）。**关键：占位与接通共用同一张卡片、同一入口，开通当天即用，用户零学习成本。**

### 单据状态机（轨迹**非审批单**，`current_status` 由 API 节点映射驱动，无人工流转）

> `shipment_tracking` 不挂 `WorkflowDefinition`（同 `kingdee_outbox`）。`current_status` 是**最新节点态的派生展示值**，由 `sf_query_tracking`/webhook 写入，非 `execute_transition` 推进。状态枚举：`UNKNOWN`（未查 / 未配置）→ `PICKED_UP`（已揽收）→ `IN_TRANSIT`（运输中）→ `DELIVERING`（派送中）→ `DELIVERED`（已签收，终态）/ `EXCEPTION`（异常，旁路提示物流专员）。

### 引擎映射 & 兼容性（逐条；性质：➕集成，引擎无物流 API）

- ➕ **`shipment_tracking` + `shipment_tracking_node` 新模型**：两个全新 `Base` 子类，头 `__queryable__=True`（非 doc_type，不挂状态机，与 `kingdee_outbox` 同范式，07-金蝶 §页面1 印证）+ 一次 alembic 迁移。引擎现**无任何轨迹模型** → **engineFlags**。
- ➕ **顺丰适配器命令（鉴权占位）**：`@register_command("sf_query_tracking")` / `"sf_subscribe_tracking"`，`supports_retry=True`、`supports_preview=False`（查询类，非业务写）。引擎**无出站 HTTP**（核对 `command_registry.py` 无 httpx/requests）→ 适配器须引入 HTTP 客户端 + 顺丰签名工具（`MD5(msgData+timestamp+checkWord)` + Base64），全新基础设施层 → **engineFlags**。⚠️ 命令发现是 `load_commands()` 硬编 import 列表（D-03a，核对该函数仅 import finance/wms/workflow 三模块）→ 新增 `services.sf_logistics_commands` 须改这段 import，标 ➕。
- ➕ **入站 webhook 路由**：`POST /integrations/sf/tracking-callback` 为顺丰主动推送入口（非 register_command，是 FastAPI 路由 + 回调验签 + 复用 upsert），引擎无入站集成端点 → **engineFlags**。
- ➕ **鉴权 / 端点配置占位（核心「等配置」点）**：`SF_APP_KEY / SF_CHECK_WORD / SF_CUSTOMER_CODE / SF_BASE_URL / SF_INTEGRATION_ENABLED` 入 8 配置域（敏感配置，写不回读 + 脱敏；同 §00-9 会话密钥整改同级别敏感处理）。**默认 `SF_INTEGRATION_ENABLED=false`** → 命令短路返回「集成未启用」软结果，前端退化手填兜底。顺丰给到配置即填即开，**代码零改动**（这正是「接好等配置」的落点）。
- ✅ **运单号联动现成**：联动锚点 `tracking_number` 在 `goods_receipt_line`(:680)/`shipment_line`(:729) 已存在、`Shipment.logistics_tracking_number`(:748) 已存在 → `shipment_tracking.tracking_number` 直接对齐这些列做关联（沿运单号串链：出库单 / 入库单 / 报关单填了运单号 → 详情页内嵌时间轴按运单号查 `shipment_tracking`）。**报关单 06-1 头与 06-6 资料清单的运单号**亦经此联动展示在途。
- ✅ **行级公司隔离 / 字段防火墙**：`shipment_tracking.company_id` → `_company_filter` 天然限本公司（引擎 05 §5.3）；轨迹**不含价 / 成本字段**，对 SALES 无需遮蔽（运单进度本就可对销售展示，便于 SA 回报客户）。
- ➕ **前端货物进度时间轴**：DataExplorer/DocEditor 默认不渲「竖向时间轴」卡片 → 前端新增一个轨迹时间轴组件（消费 `shipment_tracking_node`），嵌出库 / 入库 / 报关详情页。属前端扩展 → **engineFlags**。
- ➕（占位不实现）**定时轮询**：引擎无 cron（同 06-2）→ 批量刷在途运单走工作台轮询触发或外部定时器，**首期不做**，按需查询 + 订阅推送已覆盖。
- ❌ **无破坏底座的点**：轨迹为旁路只读增强，与报关 / 退运 / 出入库主流程**完全解耦**；SF 不通 = 卡片退化手填，主骨架零影响（符合决策⑪「相对独立后做」）。

### 业务单据推送（金蝶）
- **不推金蝶**：货物轨迹是承运商物流事实，**与做账无关**，不进 `kingdee_outbox`、不映射任何 formId（与 06 报关单本身默认不推同理，§00-6.3）。物流时效若入经营 KPI（如签收时长）= 08 报表只读聚合，不推送。

## 06-6 报关资料清单导出（发货代/报关行，缺件提示）

### 定位与使用者
报关前把这票货资料整理发给货代/报关行。物流主任在报关单详情「一键导出资料清单」生成纯文本清单（报关单号/方向/入库表/出库单·发票·装箱单/商品明细/许可证 + **缺件提示**），复制发出（SOP §四-7、调研证据 CUS-P0-04）。报关材料包 = 报关明细汇总表 + 运单 + 商业发票 invoice + 装箱单 packing list（调研证据 1-4「最主要就是 invoice」）。

### 内容要素（对齐调研证据 1-4 / W-7 / S-3）
- 报关单号 / 方向 / 公司 / 报关行 or 自报。
- 关联入库表（基本进库简化表，按 PO×型号×数量汇总，调研证据 1-6）/ 出库单 / 发票 / 装箱单（毛/净重/箱号，调研证据 S-3）/ 运单。
- 商品明细（型号/中文品名/HS/原产地/数量/申报金额）。
- 挂接许可证。
- **缺件提示**：入库缺运单号标 ⚠；出口**缺 Invoice 标「商业发票最致命」**（调研证据 1-4/V-8/SOP §四-7）。

### 引擎映射 & 兼容性
- ➕ **导出命令**：`@register_command`（汇集报关单+关联单据+明细+许可证→生成纯文本清单 + 缺件标注）；或退化为 DataExplorer CSV（引擎 08 §8.3 客户端 BOM CSV 导出，✅ 但无缺件逻辑）。缺件提示需自定义命令。
- ➕（可选）**资料包对象**：若甲方要委托报关行模式的成套盖章件归集（调研证据 CUS-P0-04），新增「资料包」清单对象（各项状态：系统生成/已上传/缺件/不适用）；本文默认**先做资料清单导出**，资料包对象列 gap。
- ✅ 盖章扫描件 / 海关批文回传 = `wms_attachment` 归档（引擎现成）。
- ❌ 无。

### 验收标准 + gap
- ✅ 一键导出资料清单含报关单号/明细/缺件提示，可复制；出口缺 Invoice 明确标致命。
- gap-14：是否需「委托资料包」对象（成套盖章件状态管理，CUS-P0-04）vs 纯清单导出——本文默认纯清单 + 附件归档，资料包待甲方。
- gap-15：资料清单是否需直接邮件发货代（引擎无邮件，同 03 模块 gap-9）；默认导出复制，邮件外推待甲方。
