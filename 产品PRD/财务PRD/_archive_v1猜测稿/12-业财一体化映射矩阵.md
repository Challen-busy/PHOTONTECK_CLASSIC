# 12 · 业财一体化映射矩阵（自动凭证发动机）

> Layer1 核心发动机模块。对齐 Layer0 基石 digest（v1·2026-06-17）。科目码/状态码/角色/取数格式以基石为唯一来源，冲突即停。
> 本文件只写 PRD，不动代码。所有引擎接入纯扩展（`@register_command`/`@register_transition_effect`/`@register_transition_validator` + `__doc_types__` + `WorkflowDefinition.states` JSONB），core/registry.py、services/workflow.py、services/commands.py 字节级零 diff。

---

## ① 模块定位（一句话业务价值）

**把全部业务单据（销项发票/收款/出库成本/采购入库暂估/进项发票/付款/预收预付/退货红冲/库存调整/调拨/报关费分摊/工资计提/费用报销/折旧/结转损益）在业务动作发生的同一事务里，按"科目映射规则表"自动生成借贷平衡的会计凭证草稿并过账更新科目余额——实现"财务唯一写入路径=凭证过账"的业财同源底座。** 业务人员零会计知识录单，财务只审映射后的凭证，金蝶手工做账彻底下线。

本模块是 Layer1 全部交易模块的"公共发动机"：AR/AP/出纳/存货成本/收入CAS14/暂估/固资/薪酬/费用/税务各模块的"⑥效果（过账）"统一调用本模块的 `map_to_voucher_entries` + `finance.post_voucher`，本模块只负责"业务单→分录→过账"的通用机制 + 全量映射规则字典，不重复定义各业务单的状态机（各模块自管）。

## ② 本模块的财务对象清单（对象范式 8 列）

本模块自有 3 个对象 + 1 套规则数据（其余业务单状态机归各业务模块，本模块只提供"挂在它们出边上的 effect"）。

### 对象 2.1 — AccountMappingRule 科目映射规则（❌新建·A 主数据）

| 范式列 | 内容 |
|--------|------|
| ① 对象 | `AccountMappingRule`（model，❌新建表）；`__queryable__=True`；无 doc_type（纯配置主数据，不走状态机，走 DocEditor/DataExplorer 通用壳 + 专用映射维护屏） |
| ② 类型 | **A 主数据**（公司级账套化配置） |
| ③ 数据 | 见 §3.1 字段字典。一行=「某公司·某业务单类型·某触发动作·某分录行序」的取数规则 |
| ④ 动作×角色 | 新增/改/停用：`FINANCE_MANAGER`（建账期维护）+`FINANCE_DIRECTOR`（准则切换/重大改）；查看：全财务。改映射=高危，须 `WorkflowDefAuditLog` 留痕（变更前后 JSON 快照），生效走 `effective_date`（不追溯改历史凭证） |
| ⑤ 逆操作 | 不冲销（配置类）。误配→停用（`is_active=False`）+ 新增正确行（新 `effective_date`）；历史已过账凭证不受影响（凭证一旦生成即与规则解耦，存自身分录快照） |
| ⑥ 效果 | 被 `map_to_voucher_entries()` 读取，把业务单字段按 `amount_formula`/`tax_handling`/`account_source` 解析为 `VoucherEntry[]`。本身不直接产生余额变动 |
| ⑦ 回链 | `company_id` 账套；规则不回链业务单（是业务单 effect 反查规则）。审计经 `WorkflowDefAuditLog` |
| ⑧ 准则/账簿 | CAS / HKFRS 双套：同一业务单类型在不同准则下可配不同 `account_code`（如 HKFRS 用 Trade receivables 对应码）；按公司 `region`（CN→CAS / HK→HKFRS）取规则集 |
| +拓 | **方向不写死**：规则只标 `dr_cr_role`（DR/CR 增减语义），最终借贷由引擎 `f(account.balance_direction, dr_cr_role)` 推导。备抵科目（1602/1241/1471…CREDIT）减少时自动记反方向。每条规则发布前必须有"财务确认"标记位 `confirmed_by_id` |

### 对象 2.2 — Voucher 自动凭证（✅复用模型 ➕扩展逆向边·B 交易凭证）

| 范式列 | 内容 |
|--------|------|
| ① 对象 | `Voucher`（model ✅已有）+ `VoucherEntry`（✅已有）；doc_type=`VOUCHER`（状态机✅已 seed 18 态，➕需补逆向边）；`is_auto_generated=True` 标识本发动机产物 |
| ② 类型 | **B 交易凭证单据**（头 Voucher + 行 VoucherEntry） |
| ③ 数据 | Voucher 头：`voucher_number`/`voucher_date`/`period_id`/`voucher_type`/`description`/`total_debit`/`total_credit`/`status`/`is_auto_generated`/`source_doc_type`/`source_doc_id`/`posted_by_id`/`posted_at`。VoucherEntry 行：`line_number`/`account_id`/`description`/`debit`/`credit`/`currency`/`exchange_rate`。金额 Numeric(16,2)、汇率 Numeric(12,6)。详见 §3.2 |
| ④ 动作×角色 | 状态机出边见 §4。关键：制单（自动 effect，actor=触发业务单的人，凭证 `created_by_id` 落实）→提交审核（`ACCOUNTANT`）→审核过账（`FINANCE_MANAGER`/`BOSS`）。**制单≠审核≠过账**（职责分离开关 ON 时强制，见 §5） |
| ⑤ 逆操作 | **未结账蓝冲**：POSTED→AUDITED「反过账」（清 AccountBalance 增量）→AUDITED→DRAFT「反审核」→改/删→重过账（蓝字正常凭证，不留负数）。**已结账红冲**：`finance.red_reversal` 命令——按原凭证生成红字负数冲销凭证（同科目同方向取负）+蓝字重做凭证（跨年损益经 6901）。财务自判用哪种 |
| ⑥ 效果 | 过账 effect `finance.post_voucher`：逐行把 debit/credit 累加到 `AccountBalance.period_debit/period_credit`（同 company+account+period），刷新 closing；写 `posted_by_id`/`posted_at`；置 status=POSTED。反过账 `finance.unpost_voucher` 取反 |
| ⑦ 回链 | `source_doc_type`+`source_doc_id`→业务单；业务往来单 `voucher_id`→本凭证（双向）；`CommandLog`（过账命令）；红冲凭证经 `source_doc_type='VOUCHER'`+`source_doc_id`=原凭证 id 回链原凭证 |
| ⑧ 准则/账簿 | `company_id` 账套隔离；科目码按公司准则模板；调整期凭证走 `VOUCHER_ADJUSTMENT` 状态机 |
| +拓 | 业财同源：业务单 effect 与凭证草稿在同一 DB 事务原子生成（业务单失败→凭证回滚，反之亦然）。`is_auto_generated=True` 的凭证，分录行只读（要改先反过账改业务单重生成，或财务手工调整凭证另立） |

### 对象 2.3 — 业务红冲广播（❌新建命令+effect·E 期末/过程批处理 + 通知）

| 范式列 | 内容 |
|--------|------|
| ① 对象 | `finance.create_red_reversal_from_business`（❌新建 effect，挂在退货/退款/采购退货业务单的"确认/审核"出边）；不新建表，产物是红字 Voucher（is_auto_generated=True）+ Notification 广播 |
| ② 类型 | **E 期末/过程批处理**（事件触发的自动冲销） |
| ③ 数据 | 输入=退货类业务单（`SALES_RETURN`/`RMA`/采购退货）；输出=红字凭证草稿 + `Notification`（recipient_role 广播全财务）。无独立头表 |
| ④ 动作×角色 | 触发=业务单到"退货确认/审核通过"态自动跑（actor=业务审核人）；生成的红字凭证草稿由 `FINANCE_MANAGER` 审核过账（不自动过账，留人工确认闸） |
| ⑤ 逆操作 | 红字凭证本身是冲销凭证；若误红冲→对红字凭证再反过账（未过账）或再红冲（已过账）。原则上业务红冲应与业务单退货状态联动，不单独撤 |
| ⑥ 效果 | ①顺 `source_doc_type`/`source_doc_id` 回链原业务单→原凭证（`Voucher.source_doc_type=原单type AND source_doc_id=原单id AND status=POSTED`）；②按 `reversal_handling=NEGATE` 生成红字凭证（原分录每行金额取负，同科目同方向）；③**必广播**：`create_notification(recipient_role='FINANCE'/'ACCOUNTANT'/'FINANCE_MANAGER', ...)` 通知全部财务（基石不变量⑩） |
| ⑦ 回链 | 红字凭证 `source_doc_type='VOUCHER'`+`source_doc_id`=原凭证 id；同时挂退货业务单（经退货单自身 source_doc 链）；`CommandLog` 留命令；`Notification.source_doc_type/source_doc_id`=退货单 |
| ⑧ 准则/账簿 | 跨年退货（原凭证在已结账年度）→损益类红冲经 `6901 以前年度损益调整`，不直接动本年 6001/6401；同准则同账套 |
| +拓 | 退货红冲含增值税：红字凭证的销项税 `222102` 行也取负（红字专票口径，WebSearch 查证：红字发票以负数销项税入账）；CAS14 附退回条款销售另有"预计负债"处理（见模块 收入CAS14，本发动机按映射规则配置即可） |

## ④ ★科目映射取数公式（本模块核心交付：业务单→凭证分录全量矩阵）

> 全表约定：方向列写 `dr_cr_role`（DR/CR 增减语义），最终借贷由 `f(account.balance_direction, dr_cr_role)` 推导。金额取数遵基石 §6：INCLUSIVE=amount×(1+税率)；EXCLUSIVE=amount；TAX_ONLY=amount×税率。成本=Σ `InventoryTransaction.total_cost`（移动加权）。外币=amount×exchange_rate 折本位币。**每条标【财务确认】待签**。

## ⑥ 校验规则 + 边界 + 错误处理（失败精确提示，绝不静默）

| 校验点 | 规则（hard_rule） | 失败提示（精确） |
|---|---|---|
| 借贷平衡 | 过账前 Σ`debit`=Σ`credit` | 「凭证不平衡：借方合计 {x}，贷方合计 {y}，差额 {x-y}，禁止过账」 |
| 期间锁 | `AccountingPeriod.status` 须 OPEN | 「会计期间 {period} 已结账，不可过账/反过账；请用红冲或调整期凭证」 |
| 明细科目 | 每行 `account.is_leaf=True` | 「科目 {code} 为汇总科目，不能挂分录，请选下级明细」 |
| 职责分离 | 开关ON时 制单≠审核≠过账 | 「制单人与审核人/过账人不可为同一人（公司职责分离已启用）」 |
| 映射未确认 | 引用规则须 `confirmed_by_id` 非空 | 「科目映射 {source_doc_type}/{action} 未经财务确认，无法自动生成凭证」 |
| 金额非负 | `debit≥0 AND credit≥0` 且一行不可同时有借有贷 | 「分录行 {n} 借贷不可同时填值/不可为负」 |
| 红冲前置 | 原凭证须 POSTED 且能回链（source_doc 命中） | 「未找到原凭证（{source_doc_type}#{id} 无 POSTED 凭证），无法红冲」 |
| 暂估幂等 | 进项发票冲回须找到 provisional_flag=True 的原暂估凭证 | 「未找到暂估应付凭证或已冲回，跳过（幂等）」 |
| 业财同源原子 | 业务单 effect 与凭证草稿同事务；任一失败全回滚 | 事务回滚，business+voucher 都不落（CommandLog 记 FAILED） |
| 外币汇率 | 外币行须有 `exchange_rate`（取 ExchangeRate 当日） | 「币种 {cur} 缺当日汇率（{date}），请先维护 ExchangeRate」 |

边界：①跨期凭证→走调整期 `VOUCHER_ADJUSTMENT`；②暂估转实差异为负（实际<暂估）→1405 红字调减、移动加权回调；③同一业务单重复触发→`source_doc_type+source_doc_id` 幂等查重，已有凭证返回提示不重建；④红冲已被再红冲→检测原凭证已有红字配对，阻断重复红冲。
