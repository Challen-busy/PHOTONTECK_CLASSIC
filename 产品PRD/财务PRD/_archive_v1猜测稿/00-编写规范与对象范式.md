# 00 · 财务ERP PRD 编写规范与对象范式

> 本文是「自研财务ERP·完全替代金蝶」整套财务PRD的**编写与协作宪法**。所有模块PM（01～13）写作前必读，写作中逐条对齐。违反范式 = 不可开工 = 退回。
> 战略前提：**自研出全部法定报表+申报数据，金蝶下线**。上线先做期初导入+并行对账验证期，验平后金蝶退场。
>
> 引擎事实（已 grep 核实，2026-06-17）：FastAPI + SQLAlchemy2 async + PostgreSQL；React19 + Vite + AntD Pro。引擎核心三件 `core/registry.py`、`services/workflow.py`(`execute_transition`)、`services/commands.py`(`execute_command`) **字节级零 diff**；财务全靠注册扩展点 + JSONB 状态定义实现，**纯扩展不改核心**。

## 一、统一对象范式（PRD 骨架·固定 8 列 + 拓展）

每一个财务对象（doc_type / model）都必须按下表 8 列 + 拓展列写满。这是 PRD 的最小信息单元，也是 dev 的取数清单。

| # | 列名 | 写什么 | 映射的引擎构件（grep 已核实） |
|---|------|--------|------------------------------|
| ① | **对象** | doc_type 码 + model 类名 + 中文名 | `__doc_types__` 元组（被 `core/registry.py:doc_model_map()` 启动扫描）；纯子表/主数据标 `__queryable__=True` 进 `table_map()` |
| ② | **对象类型** | 5 类之一：A 主数据 / B 交易凭证单据 / C 台账余额派生只读 / D 报表派生定期 / E 期末过程批处理 | 决定有无专用状态机、是否只读派生、冲销如何处理（见 §二） |
| ③ | **对象数据** | 字段级数据字典：头 + 行子表 + **辅助核算维度**（往来 customer/supplier、部门、项目、币种 currency + exchange_rate）。每字段：类型/长度/必填/默认/枚举/精度/唯一约束/FK。★金额 `Numeric(16,2)`、单位成本 `Numeric(16,4)`、汇率 `Numeric(12,6)` | SQLAlchemy `Column`；公司隔离用 `AuditMixin.company_id`（NOT NULL, indexed）；审计字段 created_by_id/updated_by_id/created_at/updated_at 来自 `AuditMixin` |
| ④ | **动作 ×（角色，时机/前置）** | 状态机每条出边：`{label, from, to, roles[], hard_rules[], editable_fields[], effects[]}` | `WorkflowDefinition.states` JSONB 的 `next[]`；`execute_transition` 校验 `allowed_roles` / `editable_fields` / `hard_rules`，再跑 `effects[]`（`_collect_effect_names` 从 current_state+next_entry+entered_state 收集，按名 resolve 到 `@register_transition_effect`） |
| ⑤ | **逆操作** | 冲销路径：蓝冲（反过账/反审核→改/删→重过账，未结账）/ 红冲（红字负数冲销凭证+蓝字重做，已结账）/ 反审核 / 反结账 / 作废。财务自行判断选 | 逆向出边（POSTED→AUDITED 反过账、AUDITED→DRAFT 反审核）+ 红冲 `@register_command` 生成红字 Voucher 草稿 |
| ⑥ | **效果** | 过账更新 `AccountBalance`（period_debit/credit→closing）；生成往来 AR/AP；推台账/报表数据 | `@register_transition_effect("finance.xxx")`；effect 签名固定 `(db, doc_type, doc, to_state, user, command_log_id) -> list[str]` |
| ⑦ | **关联回链** | `source_doc_type` + `source_doc_id`（凭证回链业务单）、`voucher_id`（往来回链凭证）、`command_log_id`、FK 链 | `Voucher.source_doc_type/source_doc_id`；AR/AP/InventoryTransaction.`voucher_id`；`routers/chain.py /api/order-chains`、`routers/data.py /api/related/{table}/{id}` |
| ⑧ | **准则 / 账簿** | CAS（内地）/ HKFRS（香港）差异；`company_id` 账套（公司行级隔离，绝不合账） | `Account.company_id` + `_company_filter(user)`（`routers/reports.py` 已用） |
| +拓 | **对象特有规则 / 口径 / 钩子 / 准则差异** | 该对象独有的：暂估口径、红冲回链规则、收入五步法时点、调汇范围、坏账计提钩子等 | `@register_transition_validator` 硬拦 + effect 内联逻辑 |

**逐列详解要点：**
- ③ 必须给到「可直接建表/可直接 seed」的精度。金额一律 2 位、单位成本 4 位、汇率 6 位。枚举值给全集 + 默认。
- ④ 一条边 = 状态机一行。`roles[]` 留空 `[]` = 系统/自动触发边。`effects[]` 是**字符串名数组**（引擎按名查 `_EFFECTS`），不是函数引用。
- ⑤ 必须明确「未结账走蓝冲、已结账走红冲」，并标谁有权选（默认 ACCOUNTANT 提、FINANCE_MANAGER 过）。
- ⑥ **财务唯一写入路径 = 凭证过账**。AccountBalance / 报表 / 台账永远是派生只读，禁止任何旁路直写。

## 二、对象类型 5 分类（行为 + 引擎落点 + 冲销处理）

| 类 | 名称 | 行为特征 | 有无专用屏 | 引擎落点 | 冲销如何处理 |
|----|------|----------|-----------|----------|--------------|
| **A** | 主数据 | 长期存在、被引用、少变更（科目/期间/币种/汇率/科目映射模板/角色配置） | 配置类用通用壳 `DocEditor`；科目表用专用「左树右明细」 | `__queryable__=True`；变更走 Admin 审计（`WorkflowDefAuditLog` 同源） | 不冲销，只「停用/版本化」。生效日切换准则模板。 |
| **B** | 交易凭证单据 | 有完整生命周期状态机、可过账、强审计（Voucher、AR、AP、预收/预付、付款申请、票据、发票、未来的折旧凭证/计提凭证/费用报销/红冲凭证） | **有专用屏**（凭证录入密集网格；往来核销左右栏） | `__doc_types__` + `WorkflowDefinition.states`；effect 过账 | **核心冲销对象**：未结账蓝冲（反过账→改→重过账）；已结账红冲（红字 Voucher + 蓝字重做）。业务退货红冲自动生成草稿+广播全财务。 |
| **C** | 台账 / 余额派生只读 | **不可手工写**，由 B 过账派生（AccountBalance、InventoryValuation、往来余额、信用占用、累计折旧台账） | 无专用屏，用 `DataExplorer` 下钻 | effect 计算后写入；查询走 `routers/reports.py` + `_company_filter` | 不直接冲销。冲销发生在源 B 凭证，台账随过账/反过账自动重算。 |
| **D** | 报表派生定期 | 期间内可反复出、口径固定（三大表+四账簿+试算+科目余额+账龄+现金流量间接法+申报数据） | 复用现有 reports 壳 + 期间/科目范围筛选 | `routers/reports.py`（已有 trial_balance/account_balance/aging_analysis）；JSONB `report` 节点 | 不冲销。源数据改了重新出表（口径决定快照 or 实时）。 |
| **E** | 期末过程批处理 | 一次性向导式批量动作（调汇、结转损益、期末结账、坏账计提、暂估转实、折旧计提、增值税计提） | **向导式**（分步 + 前置校验清单，UX 律 14 唯一例外） | `@register_transition_effect` 批处理 + `@register_transition_validator` 前置清单；走 VOUCHER 主流程 FX_ADJUSTED→PL_TRANSFERRED→CLOSED | 反结账（CLOSED→PL_TRANSFERRED→...）解锁期间；调汇/结转生成的凭证按 B 类冲销规则处理。 |

## 三、dev-ready 开发可开工 8 件套（硬标准）

凡有专用状态机/专用屏的对象（B 类、部分 E 类），PRD 必须给齐以下 8 件，缺一不可开工：

| # | 件套 | 「可开工」硬标准 |
|---|------|------------------|
| 1 | **字段级数据字典** | 每字段有类型/长度/必填/默认/枚举/精度/唯一/FK，可直接建 SQLAlchemy `Column` |
| 2 | **精确状态机表** | `state码 × 节点allowed_roles × 边editable_fields × hard_rules精确条件 × 触发effect名`，可直接 seed `WorkflowDefinition.states` JSONB |
| 3 | **★科目映射取数公式** | `业务单×动作 → 借/贷科目代码 + 金额取数公式(含税/不含税) + 税率来源 + 暂估/红冲处理`，每条标「财务确认」。方向由 `Account.balance_direction` + 增减推导，不写死借贷 |
| 4 | **权限矩阵** | `角色 × 对象 × 动作 × 字段`，含字段防火墙（成本/买价对 SALES+SA 隐藏）。可直接生成 `allowed_roles` + 字段级 mask |
| 5 | **校验规则+边界+错误处理** | 借贷平衡 / 期间锁 / 超额 / 必输 / 勾稽 / 冲销前置；每条失败给**精确中文提示**，绝不静默缺行 |
| 6 | **接缝/接口契约** | 哪个业务 doc 哪态触发哪 effect；复用现有还是新增哪个 effect（标 ✅复用/➕扩展/❌新建） |
| 7 | **验收用例** | Given-When-Then，**必含边界**：冲销/暂估转实/跨期/调汇/坏账/红冲回链/借贷不平拒过账 |
| 8 | **种子数据 + UI 交互稿** | 可 seed 的样例数据 + 对齐行业惯例的屏交互稿（密集网格 or 向导）+ 关键交互清单 |

纯配置主数据（A）/ 纯派生台账（C）/ 报表（D）**不强制 8 件**，但必须给①②③⑥⑧的对应子集（数据字典 + 取数口径 + 接缝 + 复用哪个壳）。

## 五、分层依赖与并行规则

```
Layer0 基石（串行·先封板）            ← 本文 + 01-基石.md
  科目表 / 会计年度期间 / 币种本位币 / 汇率
  ★凭证内核（录入→审核→过账 + 蓝冲/红冲）
  ★科目映射与凭证模板框架（业财翻译规则库格式）
  财务角色 + 职责分离可配开关
        │ 封板（核心契约冻结、digest 下发）
        ▼
Layer1 交易模块（可并行，各自只依赖 Layer0 契约）
  总账手工凭证 · 应收AR · 应付AP · 出纳资金/票据/核销
  存货成本(移动加权+暂估) · 收入确认(CAS14五步法) · 暂估转实
  固定资产折旧 · 薪酬人力成本 · 费用报销 · 税务(进销项+计提+申报)
        │ 各模块过账 effect 写 AccountBalance（同一契约）
        ▼
Layer2 台账/报表/期末（收口·报表最后）
  调汇 · 结转损益 · 期末结账(向导) · 坏账计提
  四账簿 · 试算/科目余额 · 三大表 · 现金流量(间接法) · 申报数据 · 管理看板
```

**规则**：
1. **Layer0 必须先封板**（本文 + 01-基石.md 完成、digest 下发），Layer1 任何模块不得提前定义自己的科目码/状态码/过账口径——一律引用 Layer0 契约。
2. Layer1 各模块**并行**，互不依赖；唯一共享 = Layer0 的科目集 + 凭证过账 effect 契约 + AccountBalance 写法。
3. **报表（D）排最后**——它消费所有 Layer1 的过账结果，口径在 Layer1 全部落地后才能定稿。
4. 期末过程（E）依赖全部 Layer1 凭证都已过账，故归 Layer2。

## 六、全局不变量清单（每模块逐条遵守）

1. **借贷平衡**：过账前 `Σ借 = Σ贷`（hard_rule），不平禁过账，精确提示差额。
2. **过账不可改**：`Voucher.status=POSTED` 只能蓝冲/红冲；AccountBalance / 账簿 / 报表 / 科目余额是**只读派生**。
3. **期间锁**：已结账期（`AccountingPeriod.status=CLOSED`）不可过账；跨期用调整凭证（VOUCHER_ADJUSTMENT 第 13 期）。
4. **业财同源**：业务单与凭证**同事务原子生成**，要么都成要么都回滚。
5. **财务唯一写入路径 = 凭证过账**：任何科目余额变动必经 Voucher→VoucherEntry→AccountBalance。
6. **审计四本**：`CommandLog` / `WorkflowLog` / `AgentLog` / `WorkflowDefAuditLog` 全程留痕。
7. **字段防火墙**：成本/买价对 SALES + SALES_ASSISTANT 角色隐藏（字段级 mask）。
8. **公司行级隔离**：`company_id` NOT NULL，所有查询走 `_company_filter(user)`，**绝不合账**（合并报表仅占位不做）。
9. **冲销不留脏数**：未结账蓝冲不留负数（直接改/删原蓝字）；已结账红冲留红字+蓝字配对（发生额净额为 0，余额不受影响）。
10. **业务红冲必广播**：退货/退款/采购退货自动生成红字冲销凭证草稿 + **站内通知全部财务**（`Notification.recipient_role`）。

## 七、甲方 14 条已拍板决策（全 PRD 绑定前提）

1. **战略**：完全替代金蝶——自研全部法定报表+申报数据；上线先期初导入+并行对账验证期，验平后金蝶下线。
2. **范围**：总账/往来/出纳资金/存货成本（必做）+ 固定资产折旧 + 薪酬人力成本 + 费用报销 + 税务（进销项+计提+申报表数据）+ 现金流量表（间接法）+ 三大表+四账簿+轻量管理看板；**合并报表仅占位不做**。
3. **报表**：法定三大报表+四账簿必做；管理看板轻量复用现有 reports 壳。
4. **成本**：移动加权平均 + 出库暂估成本 + 期末按实际成本差异冲回调整。
5. **暂估**：采购入库即暂估应付；进项发票审核时冲回暂估。
6. **★收入**：严格按新收入准则 **CAS14 五步法**（①识别合同 ②识别履约义务 ③确定交易价格 ④分摊 ⑤控制权转移时确认）；**开票与收入确认解耦**，用合同资产/合同负债过渡；控制权转移时点按销售条款配置（默认发货/客户签收 `CUSTOMER_RECEIVED`），标财务确认。
7. **增值税税率**：行级可选——物料主数据带默认税率 → 单据行带出可改；公司层设纳税人身份（一般/小规模）。
8. **本位币**：公司级可配（内地 CNY、海外默认 USD）；外币业务入账即时汇率、期末货币性项目按期末汇率调汇、差额进汇兑损益。
9. **会计期间**：可配，默认截止 27 号、会计年度 12 月 27 号分割（对齐业务盘点）；启用日对齐期间起点。
10. **角色**：新增 出纳 `CASHIER` / 会计 `ACCOUNTANT` / 财务主管 `FINANCE_MANAGER`（现有 `FINANCE`/`FINANCE_DIRECTOR`/`BOSS`）；制单/审核/过账**强制不同人**，但职责分离做成**可配开关**（一人可多职责，小公司放宽）。
11. **科目表**：CAS（内地）/ HKFRS（香港）双标准模板，公司可切换；各公司模板下增设明细；含累计折旧/坏账准备等备抵科目（方向独立标 DEBIT/CREDIT）。
12. **★冲销**：未结账（能改）用**蓝冲**（反过账/反审核→直接改或删→重过账，蓝字正常凭证，不留负数）；已结账（不能改）用**红冲**（红字负数冲销凭证+蓝字重做），由财务自行判断选择。
13. **业务红冲**：客户退货/退款/采购退货**自动生成红字冲销凭证草稿**（顺 source_doc 回链找原凭证做反向）+ 财务审核过账；**★必须通知所有财务（广播站内通知）**。
14. **期初**：导入科目期初余额 + 未结应收应付 + 存货成本，启用日一刀切，与金蝶并行对账验证后金蝶下线。
