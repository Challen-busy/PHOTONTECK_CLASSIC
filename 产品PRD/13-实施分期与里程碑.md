## 0. 开发铁律（强制，优先于一切进度）

1. **严格遵守引擎五条**：①唯一写入路径（Command→Workflow→Domain）②注册器接入 ③元数据驱动 ④引擎不认业务术语 ⑤全程留痕（审计四本）。详见引擎 PRD `技术文档/00 §0.2/§0.4`、`09`。
2. **每个业务需求先映射到引擎构件**（doc_type / `WorkflowDefinition.states` / 命令 `@register_command` / 副作用 `@register_transition_effect` / 校验 `hard_rules`·validator / 字段防火墙 / 审计四本）；本 PRD 各页标 **✅ 或 ➕** 的——照写。
3. **一旦遇到 ❌**（在引擎语义内表达不了，或要动**引擎核心**：`execute_transition` / 注册器 / 命令框架 / 审计四本 / 唯一写入路径）——**立即停**，先定"套路"：能用现有积木拼就拼、该加业务层扩展就加、找等价替代；**与人确认后再动，绝不硬改引擎、绝不绕过底座**。改引擎核心是最后手段，且**不得让引擎沾上业务术语**。
4. 这正是 PRD 每页 ✅/➕/❌ 标注的用途：**✅/➕ 放手写，❌ 停下来。**

> 落地自检（每写一块都过一遍）：跨模块写是否只走 Command/Workflow？字段变更是否只在 `editable_fields` 白名单内？副作用是否注册成 effect（而非内联写钩子）？是否每个写动作都进了审计四本？引擎层文件里能否 grep 到业务名词（应为 0）？任一条"否"→停下来按铁律定套路。

## 1. 开发顺序（底座 → WMS → 采购 → 销售 → 报关/财务/报表 → 真实接入）

> 排序唯一依据：**谁被依赖谁先建**（先建地基/被引用的真相源，避免返工）+ **素材越硬越先建**（字段越确定，编码越快、返工越少）。

| 序 | 建什么 | 为什么排这（依赖 / 素材） |
|---|---|---|
| **0 横切底座** | 多租户 + 用户-公司-角色三元 + 字段防火墙 + **会话服务端吊销(强制下线/封号)** · 主数据(客户/供应商/产品/产品代码/产线/库位/HS/单位) · **编号规则引擎(月度连号)** · 审批中心 + **会签标准件(子表+校验器)** + 待办 + 通知 · **金蝶 outbox 适配器(壳)** · **标签/单据模板引擎** · 审计四本接线 | **被所有模块引用的地基**。先建，后面每个业务模块直接挂；不先建 = 每个模块都要返工权限/编号/会签/推送/标签。 |
| **1 WMS 仓储** | 入库收货 · 库存(批次/SN/LOT/状态/库位/来源标记) · 库存流水(事件溯源) · 出库发货(两道关) · 调拨 · 盘点 · 库存调整单 · 库位 · 标签打印 · 委外加工入库(薄) | ①**库存是采购入库、销售出库、报关明细、发货 全都引用的"真相源"**，先立住下游直接用真实库存；②**WMS 素材最全**(真实 Excel 登记表)，字段最硬，编码最快、返工最少。 |
| **2 采购供应链** | 询价(内部/对原厂) · PO · PO 总表台账 · 采购通知 · 进项发票 · 备货(会签) · 样品 SDN · RMA 统一单 · 采购在途 · 付款申请 | 依赖 主数据 + 库存；采购入库已在 WMS 立住，进项发票/付款挂金蝶 outbox。 |
| **3 销售 / CRM** | 客户·联系人·线索·商机·跟进 · 报价(利润点) · 合同/SO · SO 签单台账 · 发货申请 · 销项发票 · 客户认证(并行会签) · Forecast 接单 · 售后技术工单 · 特批发货(可隐藏) | 业务上游没错，但**代码上其发货/开票依赖 库存 + 采购派生**；库存/采购就绪后再建，发货/出库/对账**一次成型**（先建销售会卡在"库存还没有"挖坑返工）。 |
| **4 报关 · 财务视图/单据中心 · 报表看板** | 报关单/退运180/顺丰UI · 应收应付视图/预收预付到账/Credit Note/对账/链路追踪/单据推送中心 · 经营KPI/商机看板/应收看板/业绩/提成(空) · 跨公司汇总 | 都是"**消费前面单据**"的，放最后拼；推送中心消费各模块写出的 outbox，链路追踪沿已建 FK 串。 |
| **5 真实外部接入（配置类，最后）** | 金蝶真实对接 + 顺丰真实对接 + 期初数据导入 + 三方对账试运行 | 逻辑前面都写好了，这步只接凭据/配组织码/灌期初——**卡的是外部(账号申请/购买)，不卡写代码**。 |

> **回应"销售应先做"的直觉**：销售是业务流程上游，但在"谁依赖谁"的**代码视角**里，**库存(WMS) 是被销售/采购/报关共同引用的底层真相**。先把库存与其硬字段立住，销售的"发货扣库存 / 开票 / 对账"才能一次写对。所以**底层先、上游后**，是返工最少的顺序。

## 4. 风险与缓解

- **引擎被破坏** → 开发铁律(§0)：遇 ❌ 停下定套路，绝不硬改核心。
- **数据正确性(推金蝶)** → 业务单号=金蝶单据编号做幂等、失败可重推不静默丢单、三方对账兜底(段5)。
- **API 过载(529/限流)** → 退避重试、分段跑、不丢数据(本 PRD 编写期已实证)。
- **外部前提未到** → 逻辑先写、真实接入后排，UI 做成"功能已就绪·待配置"可见占位(顺丰/金蝶)，不阻塞主体。
- **静默部分成功(推金蝶/导入)** → 强校验 + 逐行回执 + 对账，杜绝缺行(参见金蝶坑研究)。
