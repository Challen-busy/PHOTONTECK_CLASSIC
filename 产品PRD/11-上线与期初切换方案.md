## 模块定位（业务价值一句话）

**把「6 家公司从 Excel / 飞书多维表 / 旧 K3 切到本系统」这件一次性的高风险动作，收口成一套有先后依赖、有逐行校验、有金蝶三方对账兜底、有回滚预案的上线脚本**——不做历史数据迁移（蓝图 §0 明确边界），**只在切换日把"现在还活着的状态"建成期初**：仓里还有多少货（库存期初）、还有哪些 PO 没到齐货（未完成 PO）、还有哪些 SO 没发完 / 没收完款（未结销售订单）。

落到蓝图三处硬约束：
- **只建期初、不迁历史**（蓝图 §0「不做历史数据迁移（只建期初数据）」、§1.2「期初数据（非迁移）」）：本系统只录"切换基准日 0 点的存量状态"，**不**把过去三年的已结订单 / 已出库流水 / 已关闭 PO 搬进来。
- **期初也要走金蝶**（蓝图 §7「上线建期初：库存期初、未完成 PO、未结销售订单」+「试运行三方对账」）：期初库存 / 未结单据同样要让金蝶做账（金蝶有 `isinitbill` 期初单标记，07b §2.1 注 `isinitbill`「期初单标记，正常推送=false」→ **期初推送时置 true**）；上线前要"推送→金蝶做账→拉金蝶已确定视图核对"三方跑通。
- **实施分期最后一环**（蓝图 §9「1 底座 → 2 WMS → 3 采购 → 4 CRM → 5 金蝶对接+期初+试运行三方对账」）：本模块就是第 5 期的落地手册——前 4 期把功能建好，第 5 期把数据灌进去、对平、切过去。

> 边界提醒（本模块性质）：本模块**不新增任何业务 doc_type / 状态机**，全部复用 01~09 已定义的单据模型（`inventory`/`goods_receipt`/`purchase_order`/`sales_order`/`kingdee_outbox` 等）；新增的只有**一批"期初导入"命令**（`@register_command`，引擎无原生导入，引擎 08 无导入端点 → ➕）与一套**对账校验路由**（手写聚合，引擎报表非元数据化 D7 → ➕）。本模块是**实施动作 + 校验脚本**，不是日常业务屏。
> **本模块最大引擎性质 = ➕extension（导入命令 + 三方对账路由），且强依赖第 1 大扩展点「金蝶推送 outbox」🚫前提**：期初也要推金蝶（带 `isinitbill=true`），故金蝶推送接口能否落地（总览 GAP·金蝶前提）**直接决定本模块"三方对账 / 并行运行"能否做**——若金蝶接口未通，期初只能落本系统侧、金蝶侧由甲方人工初始化（降级方案，见 §3.5）。

## §1 · 期初数据（非历史迁移，只建期初）：导入方式与校验

> **总原则（务必先于一切灌数动作建立共识）**：
> 1. **基准时点单一**：所有期初锁定到一个**切换基准日 0 点**（建议=某月盘点 27 号当期，**【待甲方】具体日历日**，见 GAP-11-1）。基准日之后的一切业务在系统里正常发生，基准日之前的已结业务**不进系统**。
> 2. **只录"还活着的状态"**：库存=基准日仓里实有；PO=基准日**未到齐货**的（已全到全消的不录）；SO=基准日**未发完或未收完款**的（已全发全收全开票的不录）。
> 3. **盘点为准、非旧表照搬**：库存期初以基准日**实物盘点**核定数为准（账实一次对齐），不是把旧 Excel/K3 的账面数原样导入（旧账可能不准）。
> 4. **导入即留痕、可重跑**：每条导入命令幂等（幂等键=业务单号/入仓编号，引擎 03 §3.6），失败逐行回执、可纠正后重跑，**绝不静默丢行**（对齐总览 §6.1 铁律②"不允许静默丢单"）。

### 1.1 库存期初（批次 / SN / LOT / 库位 / 状态 / 来源标记）

**定位**：6 公司各仓在基准日实有的全部库存，按**批次（=入仓编号）**粒度建期初（与 03a-3 库存真相台账同粒度、同模型 `inventory`）。这是期初里**数据量最大、字段最全、最易错**的一块（批次/SN/LOT/状态/来源标记/串货报备客户都要逐批次定）。

**导入方式（两条，扫码兜底）**：
1. **Excel 模板批量导入**（主路径）：ADMIN 用统一 Excel 模板（列=03a-3 `inventory` 字段表全集）按公司分表录入 → 跑 `@register_command("import_opening_inventory")` 批量建批次。模板列**严格对齐 03a-3 字段**：
2. **现场盘点扫码补录 + 手填兜底**（兜底路径）：批次缺 SN/LOT/库位时，物流现场扫码枪扫入仓编号/SN 补，扫不到的手填（UX 公约 §5.4 扫码字段一律手填兜底）。

**库存期初字段表（导入模板列 = `inventory` 模型字段，权威对齐 03a-3）**：

| 期初列 | 字段 | 必填 | 期初取值来源 | 校验 |
|---|---|---|---|---|
| 入仓编号 `inbound_number` | string（批次主键=标签条码） | 是 | 旧 Excel 入仓号 / 期初统一补号 `PR{YYMM}-OPEN-{seq}` | 唯一（本公司内不重）；缺则按期初规则补号（见下） |
| 批次号 `batch_number` | string | 是 | 旧表批次 | 非空 |
| 型号 `material_id` | fk→`material.sku` | 是 | 旧表型号 → 匹配产品主数据 | **FK 必须命中**（主数据先行，§2）；不命中=阻断行 |
| SN/LOT `serial_lot_number` | string | 是 | 旧表 SN/LOT | SN（单件）/LOT（批）；无不管控货（蓝图 §5.3） |
| 供应商 `supplier_id` | fk | 是 | 旧表供应商 | FK 命中供应商主数据 |
| 性质 `goods_nature` | enum(GOODS/SAMPLE/RMA…) | 是 | 旧表性质 | 样品/RMA 货进对应仓 |
| 数量 `quantity` | number（结存） | 是 | **盘点核定数**（非旧账面） | >0；包/盘实际数可变（几十~2000，蓝图 §5.3）|
| 已预留 `reserved_quantity` | number | 否 | 被未结 SO 占用的（与 §1.3 联动）| 期初一般=0，除非有未结 SO 已锁货 |
| 单位 `uom` | enum(包/盘/PCS) | 是 | 旧表单位 | 芯片=包/盘、器件=PCS（蓝图 §5.3）|
| **库存状态** `status` | enum(7:可售/已预留/待处理待检/NG/样品/原厂暂存/报废) | 是 | 旧表状态 → 映射 7 态 | 枚举扩 7 态（03a 兼容性，引擎现仅 AVAILABLE/RESERVED → ➕）|
| **来源/品质标记** `source_marker` | json/string | 否 | 旧表 RMA/PCN 标记 | RMA 好货=可售保标记（蓝图 §5.4）；PCN 限售标记 |
| 原厂报备客户 `reported_customer_id` | fk | 否 | 旧表报备客户（串货隔离）| 命中客户主数据；驱动出库串货校验（§00·蓝图 §5.2）|
| 库位 `location_id`/`location_code` | fk/string | 上架后 | 旧表库位 | 命中库位主数据；流转仓可空 |
| Date Code/生产日期 `date_code`/`production_date` | string/date | 否 | 旧表 | — |
| 原产地/HS `origin_country`/`hs_code` | string | 否 | 旧表 | 报关用，缺可后补 |
| 收货日期 `received_date` | date | 是 | 旧表入库日（期初取原始入库日，非基准日）| 用于库龄；index |
| 单位成本 `unit_cost`/批次成本 `total_cost` | number | 是 | 旧表批次成本 | **对 SALES 隐藏**（防火墙照常）；期初库存金额=Σ total_cost，对金蝶期初核对 |

**期初补号规则**（旧表缺规范入仓号时）：本公司本月连号取号命令（§00-7 编号规则）派生 `PR{YYMM}-{seq}-{line}`；或统一期初前缀 `PR{YYMM}-OPEN-{seq}`（**【待甲方】是否要可识别的期初专属前缀**，见 GAP-11-2）。

**校验（导入命令逐行执行，命中即标错不入库，回执导出）**：
- **FK 完整性**：型号/供应商/库位/报备客户必须命中主数据（主数据先行未做全 → 整批阻断，提示先补主数据）。
- **唯一性**：`inbound_number` 本公司内不重；重复=标错行。
- **枚举合法**：`status`/`uom`/`goods_nature` 必须落在合法枚举集（7 态/包盘PCS/性质集）。
- **数量正向**：`quantity>0`；`reserved_quantity≤quantity`。
- **盘点签字闭环**：导入后生成"期初库存清单"供 LOGISTICS_LEAD 抽核签字（业务确认，非系统强制）。

**引擎映射 & 兼容性**：
- ✅compatible：写入目标 = 现有 `inventory` 模型（03a-3 已定，`__queryable__`），导入后 DataExplorer 库存台账即可见；行级 `_company_filter` 天然按 `company_id` 隔离（引擎 05 §5.3）。
- ➕extension（**EXT-11-A 期初导入命令**）：新增 `@register_command("import_opening_inventory")`——读 Excel 行（或 JSON 数组）→ 逐行校验 → 命中即建 `inventory` 批次 + 写 1 条 `InventoryMovement(IN, movement_type=OPENING)` + 失败行入回执。引擎**无原生批量导入**（引擎 08 无导入端点，已知扩展 03a-8）；命令幂等键=`inbound_number`（重跑不重复建批次，引擎 03 §3.6）；须改 `load_commands()` import（引擎 03 D-03a）。**engineFlags**。
- ➕extension（**EXT-11-B movement_type 补 OPENING**）：库存流水 `movement_type` 枚举补 `OPENING`（期初入），与正常 `IN`/`OUT`/`COUNT_ADJUST` 区分，便于对账/审计辨识"哪些是期初灌的"。**engineFlags**。
- ➕extension（金蝶期初推送）：期初批次推金蝶**其他入库单/库存初始化单**，`isinitbill=true`（07b §2.1，正常单=false）→ 复用 07b 推送适配器但置期初标记（见 §3.2）。**engineFlags**。

**验收**：
- 6 公司期初库存全部建成 `inventory` 批次，DataExplorer 台账可见、可下钻；批次数 = 盘点核定批次数。
- 期初库存金额（Σ `total_cost`，对 SALES 隐藏）与盘点核定总额一致，且与金蝶期初库存视图对平（§3）。
- RMA 好货=可售保来源标记、PCN 货带限售标记、报备客户串货绑定齐全（出库串货校验可触发）。
- 重跑 `import_opening_inventory` 不产生重复批次（幂等验证）。

### 1.2 未完成 PO（在途 / 未到齐货）

**定位**：基准日**还没到齐货**的采购订单——已下单但未全部入库的（已全到全消的不录）。建成现有 `purchase_order`（+`purchase_order_line`）模型，**直接进"已下单/部分到货"态**，让 PA 在 PO 总表（04a-4）能接着跟单消单。

**导入方式**：Excel 模板（列=04a-3 PO 头 + 04a-3 行明细 `purchase_order_line`）批量导入 → `@register_command("import_opening_po")` 建 PO + 子表行，并用 `execute_transition` **创建模式**落初态后、再**推进模式**直推到"已下单/部分到货"态（引擎 02 §2.3 三模式；期初 PO 不重走"建单→★财务采购审批"——审批是基准日前已在线下做过的，期初直接置后置态，**审批留痕标注"期初导入、原审批已线下完成"**）。

**未完成 PO 期初字段（关键列，对齐 04a-3）**：
- 头：PO 号 `po_number`（旧 PO 号原样保留，**不重新连号**——保持与原厂/金蝶可追溯）、供应商 `supplier_id`、报备 end-customer、原厂 SO#、是否备货 `is_stock_order`、关联 SO（备货=空）、币种/付款条件。
- 行 `purchase_order_line`：型号（产品代码，按供应商）、订单数量 `quantity`、**已收数量 `received_quantity`**（基准日前已入库部分）、**未交在途 `=quantity-received_quantity`**（消单基线）、单价/金额（**对 SALES 隐藏**）。

**校验**：
- 只导**未完成**（`received_quantity < quantity` 至少一行，全到的 PO 不导）。
- 已收数量 ≤ 订单数量；在途 = 订单 − 已收 ≥ 0（消单基线正确）。
- FK 命中（供应商/产品代码/报备客户/关联 SO）；产品代码须按供应商维度命中（一型号多 code，04a）。
- 已收部分若已建库存批次（§1.1），**与该 PO 关联回填**（避免重复计在途）——即 §1.1 库存里"来自这张 PO 的已到货批次"应能挂到 PO 行（PA 核对台账消单基线一致）。

**引擎映射 & 兼容性**：
- ✅compatible：目标=现有 `purchase_order`/`purchase_order_line`（04a 已定，含 `received_quantity` 入库 effect 回填列）；台账 04a-4 直接可见。
- ➕extension（EXT-11-A 同族）：`@register_command("import_opening_po")` 建单 + 直推后置态（`execute_transition` 推进模式，跳过★财务采购审批节点，标注期初）；幂等键=`po_number`。**engineFlags**。
- ⚠️ 注意（引擎 02 D-02e 边级角色）：期初直推到后置态由实施高权账号经命令执行，**不走审批中心的角色闸**（期初是数据初始化非业务审批）；须在命令内显式置态 + WorkflowLog 标注"OPENING_IMPORT"。**engineFlags**（与缓解策略一致：不裸调 `execute_transition`，经命令封装 + 留痕）。
- ➕extension（金蝶期初推送）：未完成 PO 推金蝶采购订单，`isinitbill=true`（若金蝶要求期初采购单单独初始化；**【待甲方】金蝶侧未完成 PO 是否需作期初单还是仅本系统留底**，见 GAP-11-3）。**engineFlags**。

**验收**：
- 所有未完成 PO 建成、进"部分到货/已下单"态，PO 总表（04a-4）按 PA 产线可见、在途数量正确。
- 在途消单基线 = 订单 − 已收，与 PA 旧账本一致；后续到货入库正常消单（received_quantity 累加）。
- 已到货批次与 PO 行关联，不重复计在途。

### 1.3 未结销售订单（未发完 / 未收完款）

**定位**：基准日**还没履约完**的销售订单——已签但未全部发货 / 未全部收款的（已全发全收全开票的不录）。建成现有 `sales_order`（决策①合同与 SO 合一）模型，进"执行中"态，让 SA 在签单大表（05b-3）接着跟发货/收款。

**导入方式**：Excel 模板（列=05b SO 头 + 型号-数量-单价明细行）批量导入 → `@register_command("import_opening_so")` 建 SO + 子表行，`execute_transition` 创建→直推到 `executing`（执行中）态（跳过 `draft→leader_review→signing→已签→★预付到账→下采购通知` 前置链——这些基准日前已线下完成，期初直接置执行中态 + 留痕标注）。

**未结 SO 期初字段（关键列，对齐 05b）**：
- 头：合同号=**内部订单号**（旧月度连号 `202603-022` 原样保留，**不重新连号**）、编号=**客户订单号**（黄字只读）、客户/区域/供应商/PM、币种/税率、付款方式（★预付/账期）、双签合同附件（期初可挂旧合同 PDF）、电子签章状态（期初=已回传）。
- 行：型号-数量-单价明细、**已交付数量**、**未交付在途 `=数量-已交付`**（销售在途基线，05b-6，决策②）。
- 收款：**收款/到账状态**（期初取基准日已收/应收口径；本系统只记到账确认状态，做账在金蝶，决策④）——未收部分=期初应收（推金蝶应收期初余额，与 §3 对账）。

**校验**：
- 只导**未结**（未发完 OR 未收完，至少一项未完成）；全结的不导。
- 已交付 ≤ 订单数量；未交付在途 = 订单 − 已交付 ≥ 0（销售在途基线正确）。
- FK 命中（客户/产品代码/PM/供应商/关联报价占位）。
- **与 §1.1 库存联动**：未结 SO 若已锁货（已预留），对应 `inventory.reserved_quantity` 应一致（期初预留与未结 SO 在途对得上）。
- **与未完成 PO 联动**（备货/接单）：未结 SO 若有对应未完成 PO（接单采购），两者关联回填（消单/在途账对齐）。

**引擎映射 & 兼容性**：
- ✅compatible：目标=现有 `sales_order`（05b 已定，含内部订单号/客户订单号/在途列）；签单大表 05b-3 直接可见。
- ➕extension（EXT-11-A 同族）：`@register_command("import_opening_so")` 建单 + 直推 `executing` 态（跳过签约/预付到账前置链，标注期初）；幂等键=内部订单号。**engineFlags**。
- ➕extension（金蝶期初推送）：未结 SO 的**应收期初余额**推金蝶应收期初（`isinitbill=true`），与 §3 应收对账对平；SO 单本身是否推金蝶=**【待甲方】**（金蝶侧未结 SO 作期初销售单还是仅推应收期初余额，GAP-11-3）。**engineFlags**。

**验收**：
- 所有未结 SO 建成、进"执行中"态，签单大表按 SA 部门可见、未交付在途正确。
- 销售在途基线 = 订单 − 已交付，与 SA 旧账本一致；后续发货正常减在途（05b-6）。
- 期初预留库存（§1.1）与未结 SO 锁货一致；期初应收余额与金蝶应收期初对平（§3）。

## §3 · 与金蝶三方对账（试运行 / 并行运行）：推送→金蝶做账→拉已确定视图核对

> **"三方"= 本系统业务侧 ↔ 金蝶财务侧 ↔ 实物/旧账基线**（蓝图 §7「试运行三方对账」、§9 第 5 期）。核心动作链固定为总览 §6.1 铁律①：**本系统推单 → 金蝶做账 → 拉金蝶已确定视图核对**（不实时回填，需要时拉已对外确定的报表/视图只读展示）。

### 3.1 试运行 vs 并行运行（两阶段）

| 阶段 | 做什么 | 时长 | 退出判据 |
|---|---|---|---|
| **试运行（dry-run）** | 用**1 家灰度公司**（建议先 1 家内地 + 1 家香港各跑，覆盖两区域，见 §4 灰度）跑通"期初推金蝶→金蝶做账→拉视图对平"；少量真实业务单据走完整链路（PO/入库/SO/出库/发票各 1~2 张）推金蝶核对 | **【待甲方】**（建议 1~2 周/家，GAP-11-5）| 期初对平 + 样本单据推送回执全 SUCCESS + 拉金蝶视图与本系统核对差异=0（或差异可解释闭环）|
| **并行运行（parallel run）** | 全 6 公司**新系统与旧方式（Excel/飞书/旧 K3）并行**记一段时间，每日/每周双轨对账，差异定位闭环；金蝶侧正式做账 | **【待甲方】并行期长度**（建议 1 个月度账期，覆盖一次月结，GAP-11-5）| 连续 N 个对账周期差异=0、业务角色验收签字、BOSS Go 决策 |

### 3.2 期初推金蝶（`isinitbill=true`）

- 期初三类（库存/未完成 PO/未结 SO 应收余额）经 07b 推送适配器推金蝶**对应期初单**，**`isinitbill` 置 true**（07b §2.1 注：正常推送=false → 期初=true，标记金蝶按"初始化单"做账，不重复生成业务凭证）。
- 推送同样走 `kingdee_outbox`（带 `company_id`=金蝶组织 + 业务单号幂等键 + RUNNING/SUCCESS/FAILED + 回执），失败可重推（07b 推送中心）。
- 期初推送单据 → 金蝶 formId 映射（裁定，对齐 07b §2.1，**【待甲方】确认金蝶期初单 formId**）：
  - 库存期初 → 金蝶**其他入库单 / 库存初始化单**（`isinitbill=true`）；
  - 未完成 PO → 金蝶采购订单期初（或仅本系统留底，GAP-11-3）；
  - 未结 SO 应收期初 → 金蝶**应收期初余额**（`isinitbill=true`）。

### 3.3 拉金蝶已确定视图核对（对账校验路由）

- 期初推完、金蝶做账后，**拉金蝶已确定视图**（只读，不回填）：期初库存金额 / 应收应付期初余额（复用 07a「拉金蝶已确定应收/应付视图」适配器 + `kingdee_ar_snapshot`/`kingdee_ap_snapshot`/`kingdee_inv_snapshot`，方向与推送相反，引擎无金蝶拉取 → ➕）。
- **对账校验路由**（手写聚合，本系统侧 vs 金蝶视图侧）：

| 对账项 | 本系统侧（聚合源）| 金蝶侧（拉视图）| 差异判据 | 路由 |
|---|---|---|---|---|
| 期初库存金额 | Σ `inventory.total_cost`（按 company_id，对 SALES 隐藏）| 金蝶期初库存余额视图 | 差异=0 或可解释 | ➕`/api/recon/opening_inventory` 手写聚合 |
| 期初应收余额 | 未结 SO 应收口径 Σ | 金蝶应收期初余额视图 | 同上 | ➕`/api/recon/opening_ar` |
| 期初应付余额 | 未完成 PO/未付进项 Σ | 金蝶应付期初余额视图 | 同上 | ➕`/api/recon/opening_ap` |
| 并行期业务单据 | 本系统已推 outbox SUCCESS 单据 Σ | 金蝶对应已做账单据视图 | 逐单据类型差异 | ➕`/api/recon/parallel_daily` |

**引擎映射 & 兼容性（§3）**：
- 🚫前提（**全模块最大阻塞，强依赖第 1 大扩展点**）：金蝶推送 outbox + 适配器（07b）须落地；金蝶云星空须**①支持业务单据接收（含 `isinitbill` 期初单）②能提供已确定的期初库存/应收应付余额视图供拉取**（总览 GAP·金蝶前提、07a 拉视图前提）。**前提不通 → §3 三方对账无法做（降级见 §3.5）**。
- ➕extension（拉金蝶视图）：复用 07a「拉金蝶已确定应收/应付视图」适配器（`kingdee_*_snapshot` 只读模型）+ 新增期初库存 snapshot；不回填、不重算（总览 §6.1 铁律①）。**engineFlags**。
- ➕extension（对账路由）：手写聚合路由 `/api/recon/*`（引擎报表非元数据化 D7，同 08 模式：`_company_filter` 租户作用域 + `/api/aggregate`）；**对账聚合含成本/应收，须按角色裁剪**（财务=本公司、财务总监=跨公司只读、SALES 不可见成本，手写路由自裁剪同 EXT-08-B）。**engineFlags**。
- ✅compatible：推送台账/回执/重推=07b `kingdee_outbox` + DataExplorer（已定）；推送审计=CommandLog 请求/响应快照（07b §复用）。

**验收（§3）**：
- 灰度公司期初推金蝶（`isinitbill=true`）回执全 SUCCESS；拉金蝶期初库存/应收/应付视图与本系统侧对账差异=0（或差异逐项可解释、闭环签字）。
- 并行期每个对账周期差异=0（或闭环）；财务/财务总监按口径签字。
- 推送失败行 100% 在推送中心可见、可重推，**无静默丢单**（总览 §6.1 铁律②）。

### 3.5 降级方案（金蝶接口前提未通时）

> **若金蝶推送/拉视图接口最终不可用（总览 GAP·金蝶前提未通），§3 三方对账降级为**：
- 期初只灌**本系统侧**（§1 照常）；金蝶侧期初由**甲方财务在金蝶内人工初始化**（金蝶自带期初导入工具）。
- 对账降为**本系统侧 vs 金蝶侧人工核对**（导出本系统期初清单 → 甲方在金蝶比对），不做自动拉视图对账路由（`/api/recon/*` 仅出本系统侧聚合供人工比对）。
- 日常业务单据**人工二次录入金蝶**（回到现状的"专人录金蝶"，访谈 05），本系统推送中心降级为"待推清单导出"。
- **此降级须 BOSS/财务总监知情接受**（增人工成本），并在 Go/No-Go 评审明确。

### 4.3 回滚预案（Rollback）

> **回滚=切换日 Go/No-Go 判 No-Go 时，安全退回旧方式**（本系统不做历史迁移，回滚相对简单：期初是"建"出来的，可整批撤销）。

| 触发条件 | 回滚动作 | 引擎落点 |
|---|---|---|
| 期初导入大面积失败（FK/校验错过多）| 不切：补主数据/纠数据，择日重灌（期初命令幂等，重跑覆盖）| ✅ 命令幂等 + CommandLog 可回放 |
| 期初推金蝶失败/对账差异大且不可解释 | 不切：暂缓金蝶侧、定位差异；或转降级方案（§3.5）| ✅ outbox 可重推；金蝶期初单可在金蝶侧反审/删（`unaudit/delete`，07b 接口族）|
| 业务冒烟发现阻断性缺陷 | 回滚：① 本系统期初批数据**整批撤销**（按 OPENING_IMPORT 标记 + CommandLog 批次反向命令，或 DB 快照恢复）；② 金蝶侧期初单反审/删；③ 业务退回旧 Excel/飞书/K3 继续 | ➕回滚撤销命令（按 command_log 批次 + OPENING movement 反冲）/ ✅ DB 切换前快照恢复（运维层）|

**回滚必备（T-1 前就绪）**：
- **切换前全库 DB 快照**（运维层备份，回滚最硬兜底——直接恢复到切换前状态）。
- **期初导入按"批次"可识别**（CommandLog 同一导入批 + `movement_type=OPENING` + WorkflowLog `OPENING_IMPORT` 标记）→ 可按批反向撤销。
- **金蝶侧期初单可反审/删**（07b 接口族含 `unaudit/unsubmit/delete`）→ 金蝶侧期初可撤。
- **旧方式不立即停**（并行期内旧 Excel/飞书/K3 仍可用）→ 回滚即"继续用旧的"。
