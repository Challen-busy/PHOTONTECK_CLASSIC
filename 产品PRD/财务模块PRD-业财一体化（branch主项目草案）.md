# 财务模块 PRD · 自研业财一体化（替代/退居金蝶）

> 本文是会话分支「主项目」的**草案**，由本分支创建、仅本分支维护；**不修改其他分支/agent 创建的任何文件**。定位：评估同一引擎能否承接完整财务、对照金蝶财务功能覆盖、并给出「金蝶推送 → 自研财务模块」的实现 PRD。
> 一手依据：引擎实勘（见 §1 证据）+ 金蝶坑研究 + 中国企业会计准则(CAS) + 香港 HKFRS（6 公司双准则）。
> 状态：草案待评审。日期：2026-06-17。
> ⚠️ 不动代码——本文只起 PRD。

---

## 0. 范围与原则
- **目标**：在本引擎内自研一套**业财一体化**财务模块，符合 **CAS（内地 3 家）+ HKFRS（香港 3 家）双准则**，覆盖**金蝶云星空财务的大部分功能**（总账/往来/出纳资金/存货成本/固资/薪酬费用/税务/报表/期末）。
- **两条铁律继承**：① 引擎五条（唯一写入路径 / 注册器 / 元数据驱动 / 引擎不认业务术语 / 审计四本）；② UX 律 14。每个子模块标 **✅复用 / ➕扩展 / ❌需新建**；遇 ❌ 停下定套路，不硬改引擎核心。
- **财务层的"唯一写入路径" = 凭证过账**：账簿、报表、科目余额、应收应付全是凭证的只读派生；任何修正只能"红冲+重做"，永不直接改账簿/报表。

## 1. ★引擎适配性评估（结论先行）

### 1.1 后端：**高度适配（约 85%）** —— 引擎本就按完整 ERP 设计、内置复式记账内核
| 已具备（实勘证据） | 模型/流程 |
|---|---|
| 会计期间体系 | `FiscalYear` 会计年度 · `AccountingPeriod` 会计期间(OPEN/CLOSED) |
| 科目表 | `Account`：`code/name/parent_id/level/is_leaf` + **`account_type`(ASSET/LIABILITY/EQUITY/REVENUE/EXPENSE/COGS)** + **`balance_direction`(DEBIT/CREDIT)**（含备抵科目方向独立存，设计正确） |
| 凭证与分录 | `Voucher`(含 `total_debit/total_credit/status/period_id/source_doc_type/source_doc_id/posted_by/posted_at/is_auto_generated`) + `VoucherEntry`(account_id/debit/credit/currency/exchange_rate) |
| 科目余额 | `AccountBalance`（公司×科目×期间：期初/本期发生/期末，借贷双栏） |
| 往来 | `AccountsReceivable`(voucher_id/paid_amount/status) · `AccountsPayable` · `AdvanceReceipt` · `AdvancePayment` · `SupplierCredit` |
| 多币种 | `ExchangeRate`（汇率表）+ 分录级 currency/exchange_rate |
| 存货成本 | `InventoryValuation`(`cost_method=WEIGHTED_AVG`/current_unit_cost/total_value) + `InventoryTransaction`(unit_cost/total_cost/**voucher_id**) |
| 财务流程 | `VOUCHER`(18态 K3 镜像：科目→录入→审核→**过账**→期末调汇→结转损益→期末结账 + 账簿/报表查询节点) · `VOUCHER_ADJUSTMENT` · `ACCOUNTS_RECEIVABLE`(23态) · `ACCOUNTS_PAYABLE` · `ADVANCE_RECEIPT/PAYMENT` · `INVENTORY_COSTING`(12态) |
| 业财过账 effect(已起头) | `finance.create_sales_invoice_from_shipment` · `create_accounts_receivable_from_sales_invoice` · `create_accounts_payable_from_purchase_invoice` · `create_advance_receipt/payment` · `confirm_payment_request_settlement` |

> **唯一真正的后端缺口**：① 上述 effect 现在只建**往来单(AR/AP)**，**还没真正生成凭证(Voucher+VoucherEntry)+ 过账更新 AccountBalance**——要补"**凭证过账层**"；② 缺**固定资产/薪酬费用/税务计提/资金出纳/报表生成**这几个纯财务模块的模型与流程。这是"补齐"，不是"从零"。

### 1.2 前端：**部分适配（约 30%）** —— 通用壳能兜，但财务专用屏要定制
- ✅ 通用壳可直接用：`DataExplorer`(台账：往来/凭证列表)、`DocEditor`(单据壳)、`ReportDrawer`/FlowEditor 报表节点(账簿/报表云朵)。
- ❌ **无任何财务专用页**（实勘：`frontend/src/pages/` 下无 voucher/account/ledger/report 财务页）。财务有强专用交互，通用 ProTable 兜不漂亮，需**定制前端**：
  - **凭证录入屏**：多借多贷分录网格 + **实时借贷平衡校验** + 连续录/复制凭证/红冲 + 科目可搜选择 + 辅助核算(往来/部门/项目)。
  - **科目表树**维护、**账簿**(总分类账/明细分类账/科目余额表/多栏账/试算平衡)、**三大报表**、**期末结账向导**。

### 1.3 结论
**同一引擎能承接完整财务，且大部分内核已在（它本就是按 ERP 设计的，只为"推金蝶"而封存）。** 工作 = 「激活财务内核 + 补凭证过账层 + 补缺失财务模块 + 做财务专用前端」，而非从零造财务。**金蝶推送可平滑改为/并行为自研财务过账**（见 §3）。

## 3. 战略建议：金蝶推送 → 自研财务模块（架构与切换）

### 3.1 现状机制
业务单状态机的 effect 现在做两件事之一：① `kingdee.enqueue_push`（推金蝶 outbox，默认 OFF）；② `finance.create_AR/AP_*`（只建往来单，不过账）。

### 3.2 目标机制（业财一体化）
在同一个**事务**里，业务单到达"入账态"时由 effect：
1. 生成 `Voucher` + `VoucherEntry`（按**科目映射/凭证模板**算借贷分录）；
2. 同步建/更 `AccountsReceivable/Payable`（往来明细，带 voucher_id）；
3. 过账时更新 `AccountBalance`。
→ `kingdee.enqueue_push` **退居为"法定申报/对账"可选通道**（或完全停用）。

### 3.3 三种过渡关系（建议走②再到③）
- ① **并行双写**（灰度期）：业务过账到自研财务，**同时**仍 enqueue 金蝶；两边按月对账验平，建立信任。
- ② **自研为主，金蝶退居申报口**：自研出三大报表/账簿；金蝶只接受最终申报/税务口径。
- ③ **完全替代**：金蝶下线。

### 3.4 切换前提（必须）
- 一位**懂 CAS+HKFRS 的资深财务**定：科目表、凭证模板/科目映射、成本方法、税制、结账规则、报表项目映射。**技术由引擎实现，会计政策对不对由财务背书**——这是成败关键依赖。

## 4. 财务模块 PRD（分子模块 · 每个：定位/参与者/状态机/引擎映射/验收/gap）

### 4.1 会计科目表与基础设置
- **定位**：六公司各自独立账套；内地 CAS、香港 HKFRS 两套标准科目表；科目层级(一级国标+企业自定义明细)。
- **参与者**：ADMIN/财务 维护科目表、会计年度、期间、本位币、汇率。
- **引擎映射**：✅ `Account`(含 type/direction/parent_id) + `FiscalYear/AccountingPeriod` + `ExchangeRate` 全在；➕ seed 两套标准科目表(CAS+HKFRS，含累计折旧/坏账准备等**备抵科目方向独立标**)；➕ 科目表树维护前端。
- **验收**：内地公司能按 CAS 建账、香港按 HKFRS；科目带正确余额方向；备抵科目方向正确。
- **gap**：双准则科目表的最终版本需财务确认；HKFRS 与 CAS 差异科目逐一核。

### 4.2 凭证与总账（核心）
- **定位**：所有财务事件的统一落点；手工凭证 + 业务自动凭证。
- **参与者**：财务录入(DRAFT)→ 复核人**审核**(AUDITED)→ **过账**(POSTED，更新科目余额、锁定)→ 期末调汇/结转损益/结账。**审核≠过账，两步、可不同人**(职责分离)。
- **状态机**：✅ 引擎 `VOUCHER` 18 态已是 K3 镜像。
- **引擎映射**：✅ 流程+模型在；➕ **借贷平衡 hard_rule**(Σdebit=Σcredit 才能过账)、**过账更新 AccountBalance 的 effect**、**过账后不可改/红冲机制**、**期间锁**(已结账期不可过账)；➕ **凭证录入专用前端**(多借多贷网格+实时平衡)。
- **验收**：不平不能过账；过账后改不动只能红冲；过账即进科目余额；红冲留痕可溯(source_doc 链)。
- **gap**：凭证字号规则、必输辅助核算项(往来/部门/项目)由财务定。

### 4.3 ★业财一体化映射矩阵（各业务单 → 凭证分录）——本模块发动机
> 业务单到"入账态"时，按下表自动生成凭证(借贷由科目 `balance_direction`+增减推导)。**这是把第二轮枚举的"触财务单据"接上凭证的核心配置**。

| 业务单(触发态) | 自动凭证分录(借/贷) | 同步往来 |
|---|---|---|
| 销项发票确认(SALES_INVOICE) | 借 应收账款 / 贷 主营业务收入 + 贷 应交税费-销项税 | 建 AR(客户) |
| 收款到账 | 借 银行存款 / 贷 应收账款 | 核销 AR |
| 出库审核出库(SHIPMENT→SALES_OUTBOUND) | 借 主营业务成本 / 贷 库存商品 | — |
| 采购入库(GOODS_RECEIPT→STOCKED_IN) | 借 库存商品 / 贷 应付账款-暂估 | — |
| 进项发票审核(PURCHASE_INVOICE→AP_CREATED) | 借 应付账款-暂估冲回 + 借 应交税费-进项税 / 贷 应付账款 | 建/转 AP(供应商) |
| 付款(PAYMENT_REQUEST→PAID) | 借 应付账款 / 贷 银行存款 | 核销 AP |
| 预收/预付 | 借银行/贷预收收入；借预付/贷银行 | 预收预付→核销转 AR/AP |
| 库存调整单过账(STOCK_ADJUSTMENT→POSTED) | 借/贷 待处理财产损溢 ↔ 库存商品 | — |
| RMA 货回入库/退货 | 红字冲收入+成本 / 存货回 | Credit Note |
| 样品(免费)领用 | 借 销售费用-样品 / 贷 库存商品 | — |
| 工资计提(薪酬单) | 借 管理/销售费用-工资 / 贷 应付职工薪酬 | — |
| 费用报销 | 借 管理/销售费用 / 贷 银行/其他应付 | — |
| 折旧计提(固定资产) | 借 管理/销售费用-折旧 / 贷 累计折旧 | — |
| 期末结转损益 | 借 收入类 / 贷 本年利润；借 本年利润 / 贷 费用类 | — |

- **引擎映射**：➕ **凭证模板/科目映射引擎**(每"业务单×动作"配借贷分录模板：科目+方向+取数公式)，财务可配不改代码(引擎 effect/DSL 正好承载)；现有 `finance.create_*` effect 在此基础上补"写凭证"。
- **验收**：每类业务过账后凭证借贷平衡、科目余额对、往来明细生成；业务红冲→自动红字凭证。
- **gap**：暂估冲回时点、税率/税目、各场景借贷科目由财务逐条确认(这是最需要财务背书的表)。

### 4.4 应收管理
- **定位**：开票生成应收→收款核销→账龄→坏账。
- **参与者**：财务/SA 看账龄；财务做收款核销、坏账计提。
- **引擎映射**：✅ `AccountsReceivable`(voucher_id/paid_amount/status) 在；➕ 收款核销(部分核销/一收多销)、账龄表、坏账计提凭证。
- **验收**：开票生应收→收款核销→账龄准；超额提醒(只读不硬拦，承接现有口径)。
- **gap**：坏账政策(账龄计提比例)由财务定。

### 4.5 应付管理
- 对称于 4.4：进项/入库→应付→付款核销→账龄。✅ `AccountsPayable` 在；➕ 暂估转实、付款核销、账龄。`SupplierCredit` 已有信用额度。

### 4.6 出纳与资金（❌新增）
- **定位**：银行/现金日记账、银行对账单、资金调拨、票据。
- **参与者**：出纳。
- **引擎映射**：❌ 新增 `bank_journal/cash_journal/bank_reconciliation/fund_transfer` doc_type + 流程；与收付款凭证联动。
- **验收**：日记账与银行存款科目余额一致；银行对账勾对差异可查。
- **gap**：是否接银行回单/银企直连(排后)。

### 4.7 存货核算与成本
- **定位**：移动加权实时成本 + 期末成本调整(平销返利/月加权)。
- **引擎映射**：✅ `InventoryValuation(cost_method=WEIGHTED_AVG)`/`InventoryTransaction(voucher_id)` 在；➕ 出库按暂估/移动加权入成本、**期末按实际成本差异冲回**、成本调整凭证。
- **验收**：出库当下有暂估成本可看利润、期末调整后成本准；存货科目余额=存货明细。
- **gap**：★成本方法(移动加权 vs 月加权)、返利成本确认时点由财务定（金蝶坑研究已揭示此为最难点）。

### 4.8 固定资产（❌新增）
- 卡片(购置)→月折旧计提→处置/减值。新增 `fixed_asset` + `depreciation` doc_type + 折旧计提定时/期末 effect → 折旧凭证。gap：折旧方法(年限平均/加速)、残值率。

### 4.9 薪酬与费用报销（❌新增，★人力成本）
- **工资单**：HR/财务录→计提→发放→凭证(借费用/贷应付职工薪酬→借应付职工薪酬/贷银行)。
- **费用报销**：员工提→审批→财务付→凭证(借费用/贷银行/其他应付)。
- **为什么必须**：人力成本+期间费用是利润表大头，缺了利润表算不出净利润。

### 4.10 税务（➕扩展）
- 增值税：进销项已由发票带；➕ 月末增值税计提/认证/申报表、附加税；企业所得税计提。HK 利得税另算(HKFRS/香港税制)。gap：税率/税目/申报口径、两地税制由财务定。

### 4.11 财务报表与账簿（➕报表引擎）
- **账簿**：总分类账/明细分类账/科目余额表/多栏账/试算平衡——✅ 引擎 VOUCHER 流程已有这些 report 节点骨架；➕ 从 `AccountBalance`+`VoucherEntry` 聚合生成。
- **三大报表**：资产负债表/利润表 = 科目余额按报表项目重分类映射；**现金流量表**特殊(需现金流量项目辅助核算/工作底稿，❌较复杂)。
- **引擎映射**：➕ 报表非元数据化→手写聚合路由 + 报表项目↔科目映射配置 + 按公司/准则两套模板。
- **gap**：现金流量表编制方法(直接/间接法)、报表项目映射由财务定。

### 4.12 期末结账（➕落地）
- **流程**：期末调汇(外币科目按期末汇率，汇兑损益)→结转损益(收入费用结转本年利润)→**结账前校验**(凭证全过账/成本已核/银行已对/往来已核)→期间锁→(反结账留痕)。
- **引擎映射**：✅ `AccountingPeriod` OPEN/CLOSED + VOUCHER 流程含调汇/结转/结账态；➕ 结账校验 hard_rules、调汇/结转计算 effect、期间锁(锁后该期不可过账)。

### 4.13 多组织（6 公司双准则）
- 每公司**独立账套**(独立科目表实例/期间/凭证序列/报表)；本位币 HK=HKD、内地=CNY；外币业务折算+期末调汇。✅ 公司行级隔离天然保证**绝不合账**；BOSS/财务总监跨公司**只读**汇总。❌ 法定合并报表占位(不做或后排)。

## 6. 不变量与铁律（财务层）
- **借贷平衡**：过账前 Σ借=Σ贷（hard_rule）。
- **过账不可改**：POSTED 凭证只能红冲；账簿/报表/科目余额只读派生。**财务唯一写入路径=凭证过账**。
- **期间锁**：已结账期不可过账，跨期用 `VOUCHER_ADJUSTMENT`。
- **业财同源**：业务单与凭证同事务原子生成（引擎单事务命令四件套保证），根治"两张皮/静默缺行"。
- **审计四本** + **字段防火墙(成本/买价对销售隐藏)** + **公司隔离(绝不合账)** 对财务同样生效。

## 7. 实施分期建议
P1 科目表+期间+本位币+汇率(双准则 seed) → P2 凭证过账层+借贷平衡+红冲+期间锁(总账跑通) → P3 业财映射(开票→应收→收款、入库/进项→应付→付款、出库成本) → P4 存货成本(暂估+期末调整) → P5 出纳资金+固资+薪酬费用 → P6 税务+期末(调汇/结转/结账) → P7 账簿+三大报表 → P8 财务前端专用屏 → P9 与金蝶并行对账→切换。
