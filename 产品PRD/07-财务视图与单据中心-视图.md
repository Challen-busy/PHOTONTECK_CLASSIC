## 模块定位（业务价值一句话）

**本系统的财务模块 = 只读视图 + 到账确认标记 + 提醒，绝不做账（账在金蝶云星空）。** 把现在散落在飞书多维表（财务整理的「应收回款统计表」、SA 每月回的「预计回款日/金额」、各公司各币种各一张表）、邮件（财务月初发表→SA 月初回→月底财务自己对差异）里的应收/应付/对账/退货冲销，**收口成一组按公司隔离的只读视图 + 一个驱动 P01「预付到账★」流程的到账确认标记 + 一条沿 FK 串全链的链路追踪**。

四条不变量（贯穿全模块）：
1. **不做账**：应收/应付总账、凭证、成本核算、收付款执行**都在金蝶**；本系统**拉金蝶已对外确定的报表/视图只读展示**，不实时回填、不重算（总览 §6.1、蓝图 §1.2/§3.6）。
2. **不硬拦**：应收到期/超额只**提醒**（分派给该看的角色），**不硬拦下单/发货**（蓝图 §3.6「展示+到期/超额提醒，不硬拦下单/发货」）。唯一硬关卡是**出库前财务放行**（在 03b 出库，不在本模块）与本模块的**预付到账确认★**（账期客户则跳过）。
3. **本公司账套**（决策 A）：普通 `FINANCE`（应收/应付/出纳）**严格本公司账套，绝不跨公司**；`FINANCE_DIRECTOR` 财务总监 = 跨公司**只读汇总**（不写单据）；`BOSS` 跨公司只读。
4. **到账只记状态**：预收/预付/收款/付款，本系统只打「到账确认状态 + 台账」，**做账在金蝶**（决策④）。

**与上游模块的接力点**：应收源 = 销项发票（01 销售）+ 出库/发货（03b WMS）；应付源 = 进项发票（02 采购）+ PO（02）；到账确认驱动 P01 销售订单的「预付到账」节点（01 销售）放行采购通知；Credit Note 上游 = RMA（02 采购，统一单共用号，决策⑨）+ 退货入库（03a WMS）。

### 页面 1 · 应收视图（只读 + 提醒，不硬拦）

**定位与使用者**：财务/SA/销售的应收总览。**数据本质 = 拉金蝶已确定的应收余额 + 本系统业务台账（`Shipping2026` 回款列）合成的只读视图**，叠加「预计回款日/金额」（SA 月初回填的本系统字段）与「到期/超额提醒」。**对销售=收款责任视角、对财务=对账口径视角、对 SA=月度回款计划视角**。

**典型流程/场景**（端到端，引用蓝图 §3.6、访谈 09 应收段；本页**无财务硬关卡**，提醒不硬拦）：
1. **月初**：财务把本公司应收（按客户）整理出来（访谈 09 L694），系统从金蝶拉「已确认应收余额 + 账期」+ 本系统业务侧带出客户/订单/发票/发货编号（`Shipping2026` 列）→ 应收视图按客户列出。财务把表（本系统视图导出或多维表）发给 SA。
2. **月初 SA 回填**：SA 在本系统对每个客户行填**预计回款日 + 预计回款金额**（访谈 09 L766「他会统计好6月应收到多少，这三格空着让我们去回」、L790「给时间和计划」），给不出的写**原因 Memo**（访谈 09 L791「客户给不出的写在这边」「什么原因这个月付不出来」）。每周更新（访谈 09 L817「我们会每周都要更新」）。
3. **账期计算（展示用）**：按付款方式（月结 60 天/月结 30/月结 15/预付）+ 月结日（多数客户当月 30/31 号）推算**到期日**（访谈 09 L424「月结60天…6/9发货→6/30月结→再推60天→8/30收款」）。
4. **月底对差异**：财务把**实际回款日/金额**（金蝶口径或财务录）与 SA 的预计值比对，差异大就再找一轮客户并通知销售（访谈 09 L694、L834「没有付的他会来找我们」）。
5. **到期/超额提醒**：到期前预警（访谈 09 L427「8/15或8/20就去问采购这笔款会不会付」）+ 超信用额度提醒（蓝图 §3.6），**分派给该看的角色**（财务 + 对应销售）。**纯提醒，绝不硬拦下单/发货**（蓝图 §3.6）。

> 关键口径：**应收余额/已收的「账」在金蝶**——本系统不重算余额，拉金蝶已确定值展示；本系统**写的只有**「预计回款日/金额/原因 Memo」这三个业务协作字段（替代飞书多维表那三个空格）。**实际回款日/金额**默认拉金蝶口径只读（若金蝶未即时回填，则财务在本系统手录确认值，见 gap）。

### 页面 3 · 预收/预付到账确认 ★（财务关卡，驱动 P01）

**定位与使用者**：**财务的硬关卡载体页**。决策④：**本系统对预收/预付只打「到账确认标记 + 台账」，做账在金蝶**。这是 P01 销售订单流程里**唯一在财务模块内的★关卡**——`★预付到账（账期则跳过）`（蓝图 P01）：预付客户必须财务确认款到账，才放行「下采购通知」；**有账期的客户跳过此关，合同成立即可下采购通知**（访谈 09 L238「如果存在账期，就可以立刻把采购通知下达给 PA；如果是货前100%，那就要等客户付款」）。

> 关键区分（访谈 09 L238）：
> - **预付/货前 100% 客户** → **必须**预付到账确认★ → 才下采购通知/才发货。
> - **账期客户**（月结 N 天）→ **跳过**预付到账，合同成立即下采购通知（款在发货后按账期回，走应收页面 1 催收）。

**典型流程/场景**（端到端，引用蓝图 P01，★标财务硬关卡）：

`待到账 → ★财务到账确认（核对实收=应预收）→ 已到账（驱动 P01 放行采购通知/发货）`

1. SO 录入时付款方式=预付/货前 100% → P01 在「下采购通知」前卡一个**待到账**节点（账期客户此节点自动跳过/不生成）。
2. 客户款到 → 财务在本系统**到账确认页**核对：实收金额 ≥ 应预收金额、币种、对应订单/客户 → 打**到账确认标记**。
3. 到账确认 = **状态推进**：触发 effect → ① 推金蝶「预收到账确认」单（做账在金蝶）；② **解锁 P01**（SO 可继续推进到「下采购通知」/「发货」）。
4. **预付（对原厂）**对称：PO 付款条件=预付 → 财务确认预付款到账（已付给原厂）→ 解锁原厂发货跟踪（应付侧，对应 P02 付款）。

### 页面 5 · 对账（客户对账 / 内部对账）

**定位与使用者**：SA + 财务的月度对账页。两类：
- **客户对账（SA 主责）**：每月月底，SA 与客户对账——「客户自己怎么登记 vs 我方内部怎么登记」达成一致，**出对账单双方签章**（访谈 09 L399-L412「双方达成一致，签，就是签章，那这个账就完成了」）。
- **内部对账（财务口径）**：财务的「应收回款统计表」（按客户、预计/实际回款日+金额、差异）月底自对——即页面 1 应收视图的回款列在月底的对账快照（访谈 09 L694、L1043）。

**典型流程/场景**（端到端，引用访谈 09 对账段）：
1. **月底客户对账**（SA）：拉本月本客户的发货/发票/收款明细（`Shipping2026` 行）→ 生成对账单 → 与客户核对一致 → **双签**（电子签章/盖章扫描）→ 账确认完成（访谈 09 L412）。
2. **查客户付款计划**（SA）：到客户系统查当月付款计划，按客户回款规律看节点是否已付，未付则找销售（访谈 09 L412-L418）。
3. **内部对账**（财务）：月底比对应收视图「预计 vs 实际」差异（页面 1），差异大→再找客户+通知销售（访谈 09 L694、L834）。

**业务单据推送（金蝶）**：对账是**业务核对**，**默认不推金蝶**（账在金蝶，对账结果用于催收与差异调查，不产生金蝶单据）。若双签确认的应收金额需回写金蝶，则推「对账确认」单（待确认，默认否）。

**验收标准**
1. 客户对账拉本月明细与业务台账一致；差异列自动算。
2. 客户对账须有双签附件才能进 signed（访谈 09 L412）。
3. 各币种各一张对账单，不混币（访谈 09 L700）。

### 页面 6 · 单据链路追踪（一笔生意全链，只读）

**定位与使用者**：财务/SA/PA/BOSS 的**一笔生意全链只读视图**。沿声明的 FK 把一笔生意从头串到尾：**询价 → 报价 → SO → 采购通知/PO → 入库 → 出库/发货 → 销项发票 → 应收**（+ 退货分支 RMA → CN）。**复用引擎 `routers/chain.py`**（引擎 02 §2.7「沿声明 FK 串链，状态码映射中文，套租户作用域+表权限闸」）。**财务用它核对「单据齐不齐、账对不对」、SA 用它看订单执行到哪一步、BOSS 用它追一笔生意全貌。**

**典型流程/场景**：
1. 财务从应收视图某行下钻 → 链路追踪 → 看这笔订单的发货/发票/收款单据是否齐全、状态。
2. SA 查某订单：已发几次货（在途数量=未发货数量，访谈 09 L364「一个订单分好几次发，每发一次扣减在途」）、发票开了没、款收了没。
3. BOSS 追一笔大单全链（只读，跨公司）。

**链路节点（沿 FK，只读）**

| 链路阶段 | 上游单据 | 关联键（FK） | 来源模块 | 状态映射 |
|---|---|---|---|---|
| 询价 | 内部询价/对原厂询价 | 型号+客户 | 02 采购 | chain.py 状态码→中文 |
| 报价 | 报价单 | 型号+客户 | 01 销售 | 同上 |
| SO | 销售订单 | 内部订单号（决策①） | 01 销售 | 同上 |
| 采购 | 采购通知/PO | SO→PO（备货时 SO 为空） | 02 采购 | 同上 |
| 入库 | 入库单/库存批次 | PO→入库 | 03a WMS | 同上 |
| 出库 | 出库/发货单 | SO→出库（发货编号） | 03b WMS | 同上 |
| 发票 | 销项发票（INV#） | SO→发票 | 01 销售 | 同上 |
| 应收 | 应收对象 | 发票→应收 | 07a（本模块页面 1） | 拉金蝶口径 |
| 退货分支 | RMA → CN | RMA 号（共用，决策⑨） | 02/07a | 同上 |

**引擎映射 & 兼容性**
- ✅ **链路追踪 = 复用 `routers/chain.py`**：`GET /api/order-chains`，沿声明 FK 遍历单据、按激活流程把状态码映射中文、套租户作用域+表权限闸（引擎 02 §2.7）。✅（机制现成）。
- ➕ **具体 FK 链/分组/终态集是业务配置**：`chain.py` 的 `GROUPS/STAGE_LABELS/TERMINAL_OK/TERMINAL_BAD`、具体 FK 遍历、阶段分类**需业务侧配置**（引擎 02 §2.9 明确「具体单据图是业务」）。本模块负责定义上表的 FK 链与阶段分组。**engineFlags**。
- ➕ **应收阶段含金蝶拉取口径**：链路末端「应收」状态来自金蝶快照（页面 1），非本系统状态机单据 → chain.py 需兼容「拉取视图节点」（非 doc_type 状态机）的展示。**engineFlags**。
- ✅ **正/反向关联遍历** = `/api/related/{table}/{id}`（正向 FK + 反向引用，剔除审计/租户 FK，引擎 01 §1.4.2）辅助下钻。✅。
- ✅ **行级隔离** = `_select_visible` 套租户作用域（FINANCE 本公司、FINANCE_DIRECTOR/BOSS 跨公司只读，引擎 02 §2.7 + 决策 A）。✅。
- ❌ **无**：链路追踪是只读，不写、不做账。

**业务单据推送（金蝶）**：链路追踪**只读，不推送**。

**验收标准**
1. 从一个 SO 能串出 报价→采购→入库→出库→发票→应收 全链，状态显示中文。
2. 退货分支：从 SO 能串到 RMA→CN（共用号）。
3. 多次发货：一个 SO 多条出库，在途数量=未发货（访谈 09 L364）。
4. FINANCE 只看本公司链路；FINANCE_DIRECTOR/BOSS 跨公司只读。
5. 链路只读，无任何写入/做账动作。

**待甲方确认 gap**
- 备货场景 SO 为空时（PO 先于 SO，蓝图 §3.3「备货时关联销售订单为空」），链路如何从 PO 反挂后补的 SO（FK 后补，待确认显示规则）。
