## 模块定位（业务价值一句话）

把现在 PA 散落在 **K3（建供应商/产品代码/录 PO/审核入库/录进项发票）+ 飞书多维表（PO total / Shipping total / 询价表三张巨表手工维护）+ 邮件/微信（询价、催货期、催发货、分货预留）+ PO 微信群人工"叫号"** 里的整条采购主链，收口为：**一套询价登记（内部+对原厂）→ 一张 PO（明细子表为真相源，含报备 end-customer/是否备货/原厂 SO#）→ 一张「PO 总表/采购台账」网格（下单+发货+付款+在途一张表跟单消单，PA 自己拉自己的数据）→ ★采购审批 → 采购在途人工跟踪+到货提醒 → 进项发票录入+★审核 → 付款申请发起（财务执行）**，做到：全程留痕、PO 编号月度连号自动叫号（替代微信群人工叫号，访谈 07:739）、台账=单据聚合视图（不再手工 copy 三张 Excel，访谈 08:62）、到货/货期/备货消单**有系统提醒**（替代"人离职就跟丢"，访谈 07:856）、采购价/成本对销售字段级隐藏（采购↔销售防火墙，§00-8）、进项发票/PO 审核通过自动推金蝶。

> 边界提醒（务必区分）：本系统是**业务 of-record，不做账、不算成本**。PA 录的进项发票/PO 推金蝶后，**应付总账、凭证、付款执行、成本核算在金蝶侧**（蓝图 §0、访谈 07:1335「库存是给财务算账的…这个系统我不看库存/账款」、访谈 07:1470「后面的都是财务的了」）。本模块的"付款申请""进项发票审核"只到**"提交/审核通过 + 台账状态"**为止，做账在金蝶（决策④）。

## 04a-1 内部询价（销售 → PA 内部询价单）➕

### 定位与使用者
**销售/SA** 把客户问价需求传给 **PA**（现状：销售直接邮件给 PA，或填《内部询价表.doc》，但"他未必会填，邮件给我们就好了"，访谈 08:229）。本系统提供**轻量内部询价单**：销售录一次需求 → 自动落到 PA 待办 → PA 据此对原厂询价（04a-2）→ PM 定利润点 → 出报价单（模块 01 销售）。**容忍邮件兜底**：单可由 PA 代录，销售只补 end-customer 等关键字段。

### 典型流程 / 场景
1. 销售收到客户询价（型号/数量/项目阶段/目标价/竞品价），填**内部询价单**或发邮件给 PA（访谈 07:1587「销售给我们发一个邮件，他想知道这个价格」）。
2. PA 收到 → 据此对原厂询价（04a-2）；询价时**额外问原厂品名/ECCN/HS code/原产地/MOQ/MPQ/质保期/是否含电池**（新原厂必问，老原厂已知则免，访谈 08:580/598/604）——这些字段同时进**产品主数据**（模块 02），询价表只引用。
3. PM 定利润点 → SA 出对客户报价单（模块 01），与本询价单勾稽。

## 04a-2 对原厂询价登记（PA → 原厂询价 / 原厂报价记录）➕

### 定位与使用者
**PA** 把"找原厂/供应商询价 + 原厂回的报价"登记下来（现状：每原厂一个文件夹 + 一张《（找原厂）询价登记表.xls》，"后面生成采购订单时会用到"，访谈 07:1587、访谈 08:424/531）。这是**采购侧成本来源**，**对销售严格隐藏**（§00-8 防火墙）。同一型号可向多家供应商询价、得多个报价，PA 据此 + PM 利润点出对客户报价。

### 典型流程 / 场景
1. PA 据内部询价（04a-1）/销售邮件，向 1~N 家供应商/原厂发询价（提供型号/数量/项目/竞品价等，访谈 07:1587）。
2. 原厂回报价单或 list → PA 登记**对原厂价（U/P）+ 货期 Lead time + 付款/贸易条件 + MOQ**（访谈 08:436「原厂给我个报价单或 list，我自己去选/提取最新」）。
3. 价格特性：**单价手录、可一季度改一次、发货前还可调**（平销返利产线"成本是发货后返差价才定"，访谈 07:643/649、访谈 08:670「每一次要手输…不同客户不同价、不同供应商不同价」）——故对原厂价**按"询价记录行"存，不写死在产品主数据**。
4. 该价 + PM 利润点 → 对客户报价单（模块 01，《（给客户）对外报价登记表.xls》14 列，含阶梯数量-单价，访谈 08:436/442 阶梯价示例）。

## 04a-3 采购订单 PO ⭐

### 定位与使用者
**PA** 的核心录单动作。一张 PO = 一次对某供应商的采购，含**报备 end-customer（可≠销售客户）、原厂 SO#、关联 SO（备货时为空）、是否备货、付款/贸易条件、ship-to/bill-to**。**PO 号月度连号自动叫号**（替代微信群人工"叫号"，访谈 07:739「这个月叫到 041」、§00-7）。录完发给原厂（导出 PDF/邮件）；★**采购审批**（财务，蓝图 §5.1）；审批通过推金蝶（§00-6.2）。

> 关键业务事实（访谈 07/08）：
> - PA 录 PO 前，**前期询价/谈价/SO 全在线下完成**，系统是"录入+导出"（访谈 07:784/793「对我来说是一个录入和导出的过程」）。
> - **产品代码不能混**：建 PO 必须用"这次采买供应商对应的产品代码"（一型号多 code 按供应商分，访谈 07:409/472「采购订单建立时要使用正确的代码」）——故 PO 行的型号选择器要带**供应商维度的产品代码**。
> - **用哪个客户名义下给原厂由 PM 定**：销售用 A 客户给 SA，PA 可能用 B 客户名义下给原厂（访谈 07:1722「都是由产品经理来定的」）→ `end_user`（报备 end-customer）可≠销售客户。
> - **下单后系统就"截止"**：货期、催货、调货全在邮件/微信线下（访谈 07:823/829），**但本系统补"采购在途跟踪+提醒"**（04a-6）填这个洞（访谈 07:856「人离职就跟丢」是痛点）。
> - **是否备货**：备货=PO 早于客户 SO（related_sales_order 为空，访谈 08:16/166），需后续消单（PO 总表跟）。

### 典型流程 / 场景（端到端，引用蓝图 P02，★=财务关卡）
1. **触发**：① 正常单——SA 在 SO 审核后发**采购通知**（型号/数量/客户/付款方式/签单公司，访谈 07:1722）→ PA 据通知 + PM 确认下 PO；② 备货单——PM/销售发**备货申请**（04b）批准后 → PA 下 PO（is_stock_order=true，related_sales_order 空）。
2. **取 PO 号**：系统按本公司本月连号自动分配 `PO {YYMMSSS}`（访谈 07:739、§00-7），替代微信群人工叫号。
3. **选供应商**：肉眼/通知确定找谁买，选供应商主数据（访谈 07:436）。
4. **录明细**（网格）：选**该供应商维度的产品代码/型号** → 数量、单价（**手录、可改**，访谈 07:667/670）、交期；统一带出付款/贸易条件、ship-to/bill-to（模板自带，访谈 07:436）。
5. **填报备信息**：原厂 SO#（原厂回的销售单号）、报备 end-customer（PM 定，可≠销售客户）、关联 SO（正常单填、备货空）、是否备货、是否需预付（requires_advance_payment）。
6. **预览 → ★采购审批**（财务，蓝图 §5.1「凡涉及钱或货进出公司都要财务审批」）：审批通过 → PO 生效 + 推金蝶（应付源，§00-6.2）。
7. **导出/发原厂**：生成 PO PDF（含唯一 PO 号、供应商代码、ship-to/bill-to）发给原厂（访谈 07:436）；之后货期跟踪进 04a-6 在途台账。

> ★ 财务硬关卡：**采购审批**（蓝图 §5.1，决策⑤吸收进审批中心，`allowed_roles=[FINANCE]`）。
> ⚠️ 引擎现状偏差：seed 的 `PURCHASE_ORDER` K3 镜像大流程把"采购审批"做成 PA/OPERATIONS 推进（无独立 FINANCE 审批节点），且把入库/质检/发票/应付全塞进同一条 PO 流程。本 PRD **重构** PO 流程为聚焦"建单→★财务采购审批→已下单→（在途/入库在各自单据）"，并把入库/发票拆到独立 doc_type（决策③/⑥）。**必入 engineFlags**。

## 04a-4 PO 总表 / 采购台账（下单+发货+付款+在途一张网格）⭐

### 定位与使用者
**PA 核心工作台**。把现在飞书三张巨表（`PO total` 32 列 + `Shipping total` 33 列 + 询价表）合成的"跟单消单"全景，落为**一张只读网格 + 下钻**：PA"只拉我的数据"（按 PA/产线过滤，访谈 08:35「我只拉我的数据」）、跟下单/发货/付款/在途、消备货单。**决策⑩**：台账=只读网格 + cell 下钻 DocEditor，不在大表里跨独立单据内联编辑（编辑回各自单据）。

> 业务事实（访谈 08）：
> - 三张表 = **同一批业务数据的不同阶段切面**："下了单就记一笔（PO total），发了货又记一笔（Shipping total）"（访谈 08:67）。系统化后**不再手抄**，由 PO/在途/发货/付款单据自动聚合成台账行。
> - PA"最晚每周全部登记"是为给财务对账（访谈 08:97）——系统化后实时聚合，消除"线上线下两张皮"（访谈 08:145）。
> - **消单**：备货 PO 等客户 SO 来了就消（备货数量递减、related_SO 回填），"一直挂备货就一直提醒 PM 还没卖出去"（访谈 07:2369、08:16）。
> - **预留/分货**：到货后 PA 按 PM 决策预留若干包给小客户/产品部，剩下可发（访谈 08:919/973）——预留落在库存（模块 03）+ 此台账可见在途/可发数。

### 典型流程 / 场景
1. PA 进台账 → 默认过滤"我的产线/我负责的供应商"（back-up 可看同组，访谈 08:1042）。
2. 看每张 PO 的：订单数量 / 已发货 / 在途（订单-已到货）/ 付款状态 / 应付余额 / 是否备货 / 备货消单进度。
3. 备货 PO 未消 → 系统提醒（"还没卖出去"，到货/货期/消单提醒见 04a-6）；点行下钻看 PO 详情 / 在途 / 发货 / 付款单。
4. 透视：按原厂/月份看发货量（访谈 08:22「产品经理想知道某原厂一月发了多少，用透视表」）→ 台账提供分组聚合视图（DataExplorer 聚合 / 导出 CSV）。

### 台账列（只读网格；🔒=**采购进价/成本，对销售端 SALES+SA 隐藏**，Q18）

**PO 段**（源 `PO total`）：下单日期 · PO号/合同号 · 原厂SO# · 报备客户 · 供应商 · PM · PD · PA · 型号 · 订单数量 · 单价🔒 · 订单金额🔒 · 货币 · 付款条件 · 原厂发货日期 · 实际发货日期 · 是否备货 · 备货金额(原始/最新)🔒 · 备货数量 · 采购通知日期 · 备货原因及待跟进 · **在途数量** · 备注。

**发货/付款段**（源 `Shipping total`，聚合发货单 + 付款台账）：发货日期 · 到库日期 · 发票号 · 发货数量 · 发货金额🔒 · 货款到期日期 · 付款状态 · 应付余额🔒 · 付款日1/2/3 · 付款金额1/2/3🔒 · 付款1/2/3状态。

> 字段权威 = `PO total`(32列)+`Shipping total`(33列) 两 sheet 共 65 列（访谈给定）。系统化后**计算列**（在途=订单-已到货、应付余额=发货额-已付、备货最新=原始-已消）由聚合得出，不再手填公式（访谈 08:40「飞书带的，可能差小数点」是手工痛点）。

### 引擎映射 & 兼容性
- ✅compatible：DataExplorer（首行 introspection 自动出列、status 过滤、金额千分位、只读 + 下钻 + CSV，引擎 08 §8.3）；下钻 DocEditor 打开 PO/发票/付款单。
- ➕extension：① **台账是跨单据聚合视图**（PO + 在途 + 发货 + 付款），引擎 DataExplorer 默认 over 单表 → 需一个**聚合视图/物化查询**（沿 FK：PO→PO_line→received / PO→advance_payment / PO→shipment）= ➕只读聚合端点（类 `/api/aggregate`，引擎 01 §1.4.2 有聚合脚手架）；② **PA 数据范围过滤**（按 purchase_assistant_id/产线，含 back-up 授权）= 行级过滤扩展（引擎 `_company_filter` 之外加 PA 维度）。
- 🔒字段防火墙（Q18）：单价/采购金额/付款金额/应付余额（**采购进价/成本**）对**销售端（SALES + SA）**隐藏；台账即便被销售端打开也无这些列。
- ❌→➕：决策⑩明确"不在大表内联编辑跨独立单据"，故只读网格 + 下钻**可兼容**（不需要引擎支持大表内联跨单据写）。
- 验收：PA 进台账只见自己产线 PO；备货未消单的行有"待消单"标记；可按原厂×月聚合发货量并导出；**SALES/SA 若有读权也看不到采购进价列（Q18）**。
- 待甲方 gap：back-up 授权颗粒度（看同组哪些 PA 的数据）——默认按产线组，待甲方。

## 04a-7 进项发票录入 / 审核（原厂 invoice → ★审核 → 应付）⭐

### 定位与使用者
**PA** 收到原厂正式 invoice 后，在系统录进项发票（发票号/金额/发票日/到期日，关联 PO + 入库单），提交 → ★**进项发票审核**（财务参与，核发票号/金额/与入库一致 → 形成应付，访谈 07:1509/1521「我录入发票号…提交到财务…他就会形成他的应付」）。审核通过推金蝶（应付/进项源，决策③、§00-6.2）。**这是 PA 在采购端的最后一步**（访谈 07:1581「弄完这个发票我就没了，后面都是财务的」）。

> 业务事实（访谈 07）：
> - **发票号原厂给、唯一、财务用它对账**（访谈 07:1515「发票号是原厂给的…财务对账用这个发票号」）。
> - **无入库就无发票**（访谈 07:1509「他没有入库就没有发票」）——录发票前须有关联入库单（外购入库）。
> - 发票一般**发货后第二天**到（访谈 07:1539）；**预付场景**下单即需 invoice（访谈 07:1551「预付，财务也需要 invoice」）。
> - 录时**改发票时间/到期时间**（入库时间是香港的），带**发票附件**给财务审（访谈 07:1509/1515）。

### 典型流程 / 场景（★=财务关卡）
1. 原厂开来正式 invoice（发货后第二天，访谈 07:1539）→ PA 录进项发票：发票号、金额、发票日、到期日，关联 **PO + 入库单**（无入库不能录，访谈 07:1509）。
2. 上传**发票附件**（访谈 07:1515）；勾稽核对（发票金额 vs 入库实收数量×单价，访谈 07:895「他入了 8160 只…和供应商给我的吻合，我就审核」）。
3. 提交 → ★**进项发票审核**（财务参与；指定下级审核人 Tracy，访谈 07:1515）→ 财务核发票号/金额/与入库一致。
4. 审核通过 → **形成应付**（`accounts_payable`，访谈 07:1521）+ **推金蝶**（进项/应付源，决策③、§00-6.2）。

> ★ 财务硬关卡：**进项发票审核**（蓝图 §5.1 五关卡之一，决策⑤吸收审批中心，`allowed_roles` 含 FINANCE；PA 录、财务审）。

### 字段表 — 进项发票头（引擎 `purchase_invoice` 已有；🔒=**采购金额/成本，对销售端 SALES+SA 隐藏**，Q18）

| 关联 PO `purchase_order_id` | fk | 是 | PO 选择器 | 存在 | — | `purchase_invoice.purchase_order_id`（✅）|

## 04a-8 付款申请（发起在采购，执行在财务，决策④）

### 定位与使用者
**PA 发起付款申请**（预付 / 货后付款），**财务执行**（决策④：付款发起在采购、执行在财务；访谈 07:2528「付款也要我们发起…我和领导觉得放到财务那」、07:2546「付了款把付款单给我，我发给供应商」）。本系统**只记到账确认状态 + 台账**，做账/付款执行在金蝶（决策④、§00-6.2）。两种来源：① **预付**（PO requires_advance_payment，下单即付，访谈 07:1551）；② **货后付款**（按付款条件 Net 30，发票审核后到期付，访谈 07:1557）。

> 业务事实：
> - 付款发起前 PA"会考虑时间点、MA、前期情况再申请"（访谈 07:2528）。
> - PA 每月给财务**发货金额预测 + 应付/付款计划**（访谈 07:2375「每月给发货金额预测和应付计划」）——属财务规划，台账可导出供财务用。
> - 付款条件多在**供应商/PO 上已固定**（访谈 07:1563/1569），付款申请带出。

### 典型流程 / 场景（决策④，做账在金蝶）
1. **预付**：PO requires_advance_payment=true → PA 发起预付申请（关联 PO + 供应商 + 金额）→ 财务审批/执行 → 打**到账/付款确认标记** + 推金蝶（付款执行做账在金蝶）。
2. **货后付款**：进项发票审核通过形成应付 → 到付款到期日 → PA 发起付款申请（关联发票/PO）→ 财务执行 → 确认标记 + 台账更新（应付余额递减，对应 `Shipping total` 付款日1/2/3 + 付款状态）。
3. 财务付款后把**付款单给 PA，PA 转发供应商**（访谈 07:2546，传达单据是 PA）。
