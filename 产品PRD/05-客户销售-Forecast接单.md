# 05 · 客户/销售 — Forecast 接单对接（模块 05d）

> 本文是 PHOTONTECK（富泰）CRM+WMS **商业交付级业务 PRD** 的模块 05d，覆盖**客户 Forecast（滚动需求预测）接单对接**这条「签单之前、客户在其自有系统里发预测/我们去接单回货期」的轻量手录留痕流程：客户在**自己的供应商管理系统**上发布**滚动需求预测（Forecast）** + 我们在客户系统上**接单 / 回货期** → SA 把这些**录进我们系统留痕**，当**备货 / 采购的输入依据**，并把**接到的客户订单回填转成 SO**。**SA 主操作、SALES 看本人客户的预测趋势。**
>
> 边界与归属（**首期不直连客户异构系统**，只做轻量手录）：
> - **客户 Forecast 与接单动作发生在外部客户系统**（如烽火多套系统、其他客户的「通用一套系统」），本系统**首期不直连**这些异构客户系统——客户系统五花八门、各家一套，直连是后期 ➕扩展（决策⑫范围待定，访谈 09 L25「进入客户供应商管理系统：接单、货期回复、看 forecast、交付申请」、L64-70）。本系统只提供一张**有结构、有留痕、可统计、可联动备货/SO** 的录入网格，把现在散在客户系统截图 / Excel / 邮件里的预测与接单**抄进来留痕**。
> - 本模块**只做 4 件事**：①登记客户**滚动预测**（客户×型号×月份×预测量，多月滚动网格）；②登记我们在客户系统上的**接单/回货期应答**（客户订单号、型号、数量、承诺货期、状态）；③把预测当**备货建议**的输入依据（联动 04b 备货申请，**默认提醒不自动建单**）；④把接到的客户订单**转 SO**（一键带字段到 05b 销售订单草稿）。**不做**：直连客户系统拉/推、EDI、自动应答、预测算法/预测准确率智能分析、AI（甲方明确不做 AI，访谈 10:394）。
> - 与 05b《订单与履约》的关系：05b 页面 7 原是**单页占位**；本模块 05d 把它**升格为独立子模块 + 两张轻量单据**（`customer_forecast` 滚动预测单 + `forecast_acceptance` 接单应答单），05b 页面 7 收敛为「见 05d」指针。**Forecast→备货建议（04b）**、**接单→转 SO（05b）** 两条接力线在本模块定义。
>
> 公约一律继承 `00-导航与参与者总览.md`（下称「00 §x」）：角色中文名（SALES/SA/PM(PD)/FAE/PA/LOGISTICS/FINANCE/FINANCE_DIRECTOR/BOSS/ADMIN）、编号规则总表（00 §7）、字段隐藏清单（00 §8：采购成本 / 买价对销售端隐藏；报价利润点/毛利对 SALES+SA 可见）、金蝶推送总则（00 §6）、引擎映射通用约定（00 §4：单据=`__doc_types__`、多行=`_line` 子表 + SubTableEditor、流程=`WorkflowDefinition.states`、跨单据派生=`@register_transition_effect`、权限=角色表+字段防火墙+`_company_filter`、待办=`list_user_todos`）、UX 公约（00 §5：表单网格 + 行内 cell 选择器、扫码兜底、预览→ChangeCard→commit）。本文不重述，只在落点处引用。

## 模块定位（业务价值一句话）

把现在 SA「登客户供应商系统接单、回货期、看 forecast、做交叉应答」（访谈 09 L25/L64-70）但**只存在客户系统里、我方无留痕、无法当备货依据、无法统计**的这条工作，收口为**一张有结构的滚动预测网格 + 一张接单应答单**，让客户的**滚动需求预测**与我们的**接单/回货期**在本系统里**留痕、可查、可联动备货与 SO**：

- **看得见预测、对得上货期**：客户在其系统发的滚动预测（未来 N 个月，型号×月份×预测量）抄录进来 → SA 拿**我方内部货期（PA 回的）vs 客户需求**做**交叉应答**，系统算 **gap（应答货期−客户需求货期 / 应答量−需求量）**，明确「该提拉 / 该推迟 / 正常」（访谈 09 L66-70）。
- **当备货 / 采购的输入依据**：滚动预测是「该不该提前囤货、囤多少」的事实底稿 → 一条预测行可**一键起一张 04b 备货申请草稿**（带型号/意向客户/预测量预填，**默认提醒不自动建单**，避免凭预测盲目囤货，呼应 04b「谁提议谁担风险」）。
- **接单一键转 SO**：在客户系统真接到客户订单（客户订单号 + 型号 + 数量 + 承诺货期）后，把接单应答**登记留痕** → 客户正式下单时**一键转 05b SO 草稿**（带客户/型号/数量/客户订单号预填），减少 SA 重复录入（访谈 09 L25「接单」与 L51「编号字段=客户订单号」衔接）。
- **留痕 + 统计**：替代「只在客户系统里、我方无台账」的现状——按客户/型号/月份看滚动预测趋势、接单履约率、gap 分布，为 PM 备货决策、SA 履约跟进提供底稿。

> 轻量边界（再次强调，避免过度设计与误期待）：本模块**首期不直连客户系统**（不拉不推不 EDI），是**手录抄写**；**不做预测算法 / 预测准确率 AI 分析**（甲方明确不做 AI，访谈 10:394）；**不强制**预测→备货→SO 自动建单（全部默认提醒/草稿、人工确认）。系统只提供**录入 + 结构 + 留痕 + gap 计算 + 联动入口**，判断动作（提拉/推迟/是否囤货/是否接单）仍由 SA/PM 人工完成。

## 05d-1 客户 Forecast 滚动预测台账 ⭐

### 定位与使用者
SA / SALES / PM 用来看「本部门 / 本人客户 / 本产线当前有哪些客户在发滚动预测、各型号未来几个月预测量多少、最新版本是哪版、交叉应答 gap 多大、哪些该提前备货」。本质是 `customer_forecast` 的**台账视图 + 备货前瞻面**（决策⑦：明细为真相、汇总=同库视图），替代「只在客户系统里、我方无台账」的现状（访谈 09 L64-70）。

### 典型场景
1. SA 月初打开台账，看各客户最新预测版本：客户、型号、滚动月份预测量、版本号、登记时间、应答状态、最大 gap。
2. PM 看本产线全部客户预测：哪些型号被多客户/持续预测、累计预测量趋势 → 判断是否提前向 PA 提备货建议（04b）。
3. SALES 看本人客户预测趋势，把「客户后续要的多」反馈推进备货意向。
4. 点任一预测单**下钻**进 05d-2 详情 DocEditor 做交叉应答 / 起备货草稿。

### 字段表（`customer_forecast` 台账投影，**只读网格 + 下钻**）

| 字段 | 类型 | 说明 | 引擎映射 |
|---|---|---|---|
| 预测单号 `forecast_number` | string | 下钻入单 | ✅ DataExplorer 自动出列；`*_number` 模糊搜索 |
| 客户 `customer_id` | fk 标签 | — | ✅ FK 标签解析（引擎 01 §1.4.2）|
| 客户系统名 `source_portal` | enum 药丸 | 烽火/通用一套系统/邮件/其他（预测来源外部系统）| ✅（访谈 09 L25 多系统）|
| 版本/期次 `forecast_version` | string | 滚动版本（如 `2026-06 滚动`）| ✅ |
| 产线/事业部 `product_line` | enum/fk | 数据范围与统计维度 | ✅（与 SO/商机一致）|
| 覆盖月份范围 `month_range` | string | 如 `2026-07 ~ 2026-12`（滚动窗口）| ➕ 派生显示列（子表 min/max 月份）|
| 型号数 / 行数 `line_count` | number | 本版含多少型号行 | ➕ 派生列 |
| 最大 gap 动作 `top_gap_action` | enum 药丸 | 提拉/推迟/正常（取最显著行）| ➕ 派生列（交叉应答结果摘要）|
| 登记人 `created_by_id` | fk 标签 | SA | ✅ 系统列、FK 标签 |
| 登记时间 `created_at` | datetime | — | ✅ 系统列、ISO 格式化 |
| 状态 `status` | pill | DRAFT/ANSWERED/ARCHIVED | ✅ status 药丸自动过滤 |

### 引擎映射 & 兼容性
- ✅ **DataExplorer 只读台账 + 下钻 DocEditor**：决策⑩「大台账=只读网格 + 下钻」（引擎 08 §8.3）；status 自动过滤、FK 标签解析、CSV 导出（无金额主线）。
- ➕ **覆盖月份范围 / 行数 / 最大 gap 动作派生列**：引擎序列化按列出列、无原生计算列 → 派生只读列 / 前端按子表聚合即时算（同 04b 超期天数派生机制）。**必入 engineFlags**。
- ➕ **客户预测趋势看板**（按客户/型号累计预测量、滚动趋势、gap 分布）：属「报表非元数据化」（引擎 00 D7）→ 手写路由 + 租户作用域 + 产线作用域。**必入 engineFlags**。
- ✅ **本部门 / 本人客户 / 本产线数据范围**：SA 看本部门、SALES 看本人客户、PM 看本产线（行级 `company_id` + 产线/客户归属过滤）；BOSS / 财务总监跨公司只读（read-privileged，00 §9）。

### 验收标准 + gap
- ✅ 台账按 `customer_id` / `product_line` / `status` / `source_portal` 过滤；下钻进详情。
- ✅ SA 只看本部门、SALES 只看本人客户、PM 只看本产线预测。
- ✅ 同客户同型号多版本可按 `forecast_version` 区分、最新版可置顶。
- gap-1：预测**版本/滚动管理颗粒度**（每月一张新单 vs 同单内多版本子记录、是否要保留历史版本对比）——默认**每个滚动期次一张新 `customer_forecast` 单 + `forecast_version` 标识**，历史单不删（可对比），待甲方（呼应访谈 09 L64-70 滚动应答）。

## 05d-2 客户 Forecast 滚动预测录入（客户×型号×月份×预测量 + 交叉应答）⭐

### 定位与使用者
SA 把客户在其系统发布的滚动预测**抄录进来 + 做交叉应答**的主界面。建单/录入/应答主用 **SA**；PM/SALES 主要在台账看、消费为备货决策。是本模块预测侧唯一的单据编辑页。

### 典型流程 / 场景（端到端，引用访谈 09 L25/L64-70）
1. **登记预测**（访谈 09 L64「客户在上面发预测」）：SA 登客户系统看到新滚动预测 → 在本系统建 `customer_forecast` 单：选**客户**、**客户系统名**（来源）、**版本/期次** → 在**滚动月份网格**里逐行录「型号 × 各月预测量」（客户×型号×月份×预测量，子表 `customer_forecast_line`）。客户可现场新建型号/客户（访谈 09 L64「客户可以现场新建」）。
2. **交叉应答**（访谈 09 L66-70「内部货期 vs 客户需求做交叉应答，看 gap」）：SA 取**我方内部货期 / 可供量**（PA 回的货期、库存+在途）对每行/每月填「我方应答货期 / 应答量」→ 系统算 **gap**（应答货期−客户需求货期、应答量−客户需求量）→ SA 标**动作**：`提拉`（客户要得急/量大，提前备货拉单）/ `推迟`（库存够/客户可缓，推迟减库存存量）/ `正常`（访谈 09 L70「该提拉提拉、该推迟推迟，减少库存存量」）。
3. **当备货依据 → 起备货草稿**（联动 04b，**默认提醒不自动**）：某行预测量大、gap 显示「提拉」且无足够库存/在途 → SA/PM 可对该行**一键起一张 04b 备货申请草稿**（带型号/意向客户=本预测客户/建议数量预填、`source_forecast_line_id` 回链），由发起人在 04b 补全风险点/金额走审批（**不绕过 04b 谁提议谁担风险/会审关**）。
4. **应答 / 归档**：交叉应答填完、动作标完 → 单进 `ANSWERED`（已应答、可作为备货/采购输入）；下一滚动期客户发新版 → 本版可 `ARCHIVED`（归档留痕、不删，新建下一版单）。

> 与接单的边界：本单是**预测**（客户「打算要」），不是**订单**（客户「正式下」）。客户由预测真正下单时走 05d-3 接单应答 + 转 SO。预测行可与后续接单行靠型号/客户对照（非强绑定）。

### 引擎映射 & 兼容性（逐条）
- ➕ **新增 doc_type `customer_forecast`（+ 子表 `customer_forecast_line`）**：一个 `Base` 子类 `__doc_types__=("CUSTOMER_FORECAST",)` + 一次 alembic 迁移 + 一行 `WorkflowDefinition`（引擎 09 §9.1）；子表名含 `_line` + 指向父表 FK → 引擎自动识别为子表、DocEditor 自动渲 SubTableEditor 网格（引擎 01 §1.4.2、00 §4.3）。引擎无现成预测模型 → 全新增，标准加单据动作，不破坏底座。
- ✅ **滚动月份子表网格（方案①逐行）**：`SubTableEditor` 脏跟踪 `_new/_modified/_delete`、自动 `line_number`（引擎 08 §8.3、00 §4.3）；型号×月份逐行排布完全在现有能力内。
- ➕ **横向月份列组（方案②，可选）**：动态月份列超出引擎按固定列 introspection 的默认渲染 → 需扩展 SubTableEditor 横向列组。**默认不做（方案①），列 engineFlags（gap-2）**。
- ➕ **gap 量 / gap 天派生列**：`gap_qty=answer_qty−forecast_qty`、`gap_days=answer_date−demand_date` 为派生只读列 / 前端即时算（同 04b 超期天数机制）。引擎无原生计算列。**必入 engineFlags**。

- ✅ **轻量三态状态机 = 节点级 `allowed_roles`**：`DRAFT`(SA/SALES)→`ANSWERED`(SA)→`ARCHIVED`；用**节点级 `allowed_roles`** 控权（不依赖边级 `roles`），**规避引擎 02 D-02e 边级角色推进未校验坑**（FR-2.7）。
- ➕ **直连客户系统拉预测**（后期）：从客户供应商系统自动拉取 forecast = 外部异构系统集成（烽火多套 vs 通用一套，各家不同），**引擎无原生集成**（蓝图 §1.3、引擎 08 §8.8）→ 占位，开关 `CUSTOMER_PORTAL_SYNC_ENABLED` 默认 OFF；首期手录抄写。**必入 engineFlags / 占位**。

- ✅ **留痕**：每次状态推进 + 字段变更落 WorkflowLog（字段 diff + effect 轨迹，`__queryable__` 业务可查，引擎 07）——把现在「只在客户系统、我方无留痕」的预测沉淀为可查资产。

### 业务单据推送（金蝶）
客户 Forecast 滚动预测单是**签单前的销售前瞻 / 内部决策底稿**，**不推金蝶**（无财务凭证含义，00 §6.2 推送清单不含预测单）。预测**起的 04b 备货申请**本身也不推金蝶（内部决策单，04b）；备货批准后下的 PO、客户真正下单转的 SO/出库各自按主流程推金蝶。

## 05d-3 接单 / 回货期应答（客户订单号/型号/数量/承诺货期/状态 → 转 SO）⭐

### 定位与使用者
SA 把在客户系统**接到的订单 / 回的货期**登记留痕、并在客户正式下单时**一键转 05b SO** 的主界面。主操作 **SA**；PA 供承诺货期输入；PM/SALES 看进度。

### 典型流程 / 场景（端到端，引用访谈 09 L25/L51）
1. **登记接单 / 回货期**（访谈 09 L25「接单、货期回复」）：SA 在客户系统接到客户订单 / 客户要我们回货期 → 在本系统建 `forecast_acceptance` 单：选**客户**、录**客户订单号**（=后续 SO 的「编号」字段，访谈 09 L51「编号=客户订单号」）、**型号 / 数量 / 客户要求货期**、SA 据 PA 内部货期填**我方承诺货期**、标**应答状态**（已接受 / 部分接受 / 议价中 / 拒接）。
2. **承诺货期交叉**：承诺货期来自 PA 内部货期 / 库存+在途（与 05d-2 同源）；与客户要求货期对照、必要时回到 05d-2 预测层做提拉/推迟。
3. **转 SO**（关键接力，联动 05b）：客户正式下单 / 接单确认 → SA 对该应答单**一键转 05b 销售订单草稿**：带**客户**、**客户订单号→SO「编号」字段**、**型号/数量/承诺货期**预填、`source_acceptance_id` 回链 → SA 在 05b SO 草稿补全单价/付款方式/双签等走 SO 主流程（05b 状态机）。**不在本单定价/不走财务关**（定价/审核全在 05b）。
4. **闭环**：应答单转出 SO 后进 `CONVERTED`（已转 SO、`sales_order_id` 回链）；拒接 / 客户取消 → `CLOSED`。

> 与 SO 的边界（明确）：本单是**接单留痕 + 货期应答 + 转单入口**，**不是 SO 本体**。凡定价、双签、付款关、下采购通知、发货、推金蝶全在 05b SO（决策①订单即合同）。本单 = SO 的**前置接单底稿 / 触发源之一**；客户也可不经本单直接下正式订单（SA 在 05b 直接建 SO），两入口并存、靠 `source_acceptance_id` 回链即可。

### 状态机 `forecast_acceptance`（RECEIVED→ANSWERED→CONVERTED/CLOSED，轻量）

| 状态 | 可操作角色 | 可编辑字段 | 硬规则 / 关卡 | 下一状态 |
|---|---|---|---|---|
| `RECEIVED`（已接单，初始）| SA/SALES | 头全部 + 接单型号子表 | 客户 / 客户订单号 / ≥1 型号行（型号+数量必填）才能提交 | →`ANSWERED`（货期回复完成）/ →`CLOSED`（拒接/客户取消）|
| `ANSWERED`（已回货期）| SA | 承诺货期 / 应答状态 | 各行承诺货期可补；应答状态=已接受才宜转 SO | →`CONVERTED`（转 SO）/ →`CLOSED`（拒接/取消）|
| `CONVERTED`（已转 SO，终态）| 系统 | sales_order_id | 派生/关联 05b SO 草稿；`sales_order_id` 回链 | —（终态，后续在 SO）|
| `CLOSED`（关闭/拒接，终态）| SA | 关闭原因 | — | —（终态）|

> 轻量说明：四态，**无财务硬关卡**（定价/审核全在 05b SO）。`CONVERTED` 后业务在 05b SO 推进（双签/付款关/下采购通知/发货/推金蝶）。`CLOSED` 用于客户拒接/取消。状态推进经 `execute_transition` 三模式（引擎 02 §2.3），节点级 `allowed_roles` 控权（规避 D-02e）。

### 引擎映射 & 兼容性（逐条）
- ➕ **新增 doc_type `forecast_acceptance`（+ 可选子表 `forecast_acceptance_line`）**：`Base` 子类 `__doc_types__=("FORECAST_ACCEPTANCE",)` + alembic 迁移 + 一行 `WorkflowDefinition`（引擎 09 §9.1）；子表 `_line` + 父 FK → SubTableEditor 网格（引擎 01 §1.4.2、00 §4.3）。引擎无现成 → 全新增，不破坏底座。
- ✅ **轻量四态状态机 = 节点级 `allowed_roles`**：`RECEIVED`(SA/SALES)→`ANSWERED`(SA)→`CONVERTED`/`CLOSED`；节点级控权规避引擎 02 D-02e 边级未校验坑（FR-2.7）。
- ➕ **接单 → 05b SO 草稿派生（跨单据，联动 05b）**：`ANSWERED→CONVERTED` 推进 = `@register_transition_effect`（EXPLICIT 显式触发）派生一张 05b `SALES_ORDER` 草稿（带客户 / **客户订单号→SO「编号」字段**（访谈 09 L51）/ 型号 / 数量 / 承诺货期预填、`source_acceptance_id` 回链），SA 在 05b 补单价/付款方式/双签走 SO 主流程；或仅手工关联已有 SO（填 `sales_order_id`）。**两入口并存**（接单转 SO / SO 独立建），靠 FK 互链。跨单据派生继承命令幂等/单事务/审计四件套（引擎 03/04）。**必入 engineFlags**（跨单据派生 + 与 05b 联动）。
- ➕ **货期 gap 天派生列**：`commit_gap_days=committed_date−required_date` 派生只读列 / 前端即时算（同 05d-2 / 04b 机制）。引擎无原生计算列。**必入 engineFlags**。
- ➕ **客户订单号→SO「编号」字段对接**：接单的 `customer_po_number` 映射到 05b SO 的「编号=客户订单号」字段（决策①、00 §7、访谈 09 L51）；转单 effect 预填该字段。引擎按字段映射可表达，属派生预填。**入 engineFlags（联动 05b 编号语义）**。
- ➕ **直连客户系统接单/回货期**（后期）：自动拉客户订单 / 推回货期 = 外部异构系统集成（同 05d-2），引擎无原生 → 占位，开关 `CUSTOMER_PORTAL_SYNC_ENABLED` 默认 OFF；首期手录。**必入 engineFlags / 占位**。
- ✅ **行级隔离 + 本部门/本人客户范围**：`forecast_acceptance` 带 `company_id`（一笔生意不跨公司，00 §1.2）；SA 看本部门、SALES 看本人客户、PA 供承诺货期输入但不接客户单（00 §3.2）；read-privileged 跨公司只读（00 §9）。
- ✅ **扫码兜底**：`customer_po_number`、子表 `material_id`/`product_code_id` 支持扫码 + 手填兜底（00 §5.4）。
- ✅ **留痕**：状态推进 + 字段变更落 WorkflowLog（引擎 07）——接单/回货期沉淀为可查资产。

### 业务单据推送（金蝶）
接单 / 回货期应答单是**签单前接单留痕**，**不推金蝶**（无财务凭证含义，00 §6.2 推送清单不含接单单）。**转出的 05b SO** 按 05b 主流程在 SO 审核/已签态推金蝶（销售/应收源，00 §6.2）。本单只触发转单标记 + 回链，不参与金蝶。

### 验收标准（可测）+ 待甲方确认 gap
- ✅ 建单必填客户 / 客户订单号 / ≥1 型号行（型号+数量）；缺任一不能提交（`RECEIVED` 硬规则）。
- ✅ 接单子表可多型号录入（型号/接单数量/客户要求货期/我方承诺货期）；`commit_gap_days` 自动算。
- ✅ `ANSWERED` 应答状态=已接受时可**一键转 05b SO 草稿**（带客户订单号→SO「编号」/型号/数量/承诺货期预填、`source_acceptance_id` 回链）。
- ✅ `RECEIVED→ANSWERED→CONVERTED/CLOSED` 四态正确；拒接/取消可 `CLOSED`。
- ✅ SA 只看本部门、SALES 只看本人客户接单；无成本/买价列。
- gap-4：接单 → SO 是**自动派生草稿**还是**仅手工关联**——默认 `CONVERTED` 派生 SO 草稿 + 回链（减重复录入），两入口并存不强制（同 05c 工单→RMA gap）。
- gap-5：接单应答与 05b SO 的**状态联动强度**（应答 `CONVERTED` 是否随 SO 推进自动更新）——默认**弱联动**（FK + 台账/链路对照，不双向状态机互锁），待甲方。

## 待甲方确认 gap 汇总

1. **Forecast 接单是否纳入系统范围 / 颗粒度**（决策⑫范围待定）——本模块按「轻量手录留痕 + 联动备货/SO」设计；若甲方要更薄（仅单页登记）或更厚（直连），按 gap-7 调整。
2. **是否（后期）直连客户供应商系统**（拉 Forecast / 接单 / 推回货期）——各客户系统异构（烽火多套 vs 通用一套），**首期手录不直连**，直连列后期 ➕（开关默认 OFF）。
3. **预测版本/滚动管理颗粒度**（每滚动期一新单 vs 同单多版本、是否保留历史对比）——默认每期一新单 + `forecast_version`、历史不删（gap-1）。
4. **滚动月份排布**（型号×月份逐行 ✅ vs 横向月份列组 ➕）——默认逐行（纯 ✅），横向列组待 UX 优化（gap-2）。
5. **预测是否带预估金额**（含税报价口径，不引采购买价）——本批不带（gap-3）。
6. **Forecast → 04b 备货** 联动方式（手工起草草稿 vs 自动建单）——默认**人工起草、提醒不自动建单、不绕过 04b 审批**（gap，避免凭预测盲目囤货）。
