## 模块定位（业务价值一句话）

把现在散落在 **K3/金蝶 ERP「销售订单 + 发货通知」录入** + **飞书多维表「客户管理部数据」大表（SO/Shipping/报关三 sheet 靠公式互算在途）** + **客户供应商系统（接单/回货期/查 Forecast）** + **邮件（询价、合同双签、发货通知、月度对账、催收）** 的整条销售订单与履约链，**收口成一套以「销售订单 SO（=合同）」为枢纽的状态机 + 一张网格化签单大表**：

`报价确认 → 合同/订单录入（SA leader 审 + 一签宝电子章）→ ★预付到账 / 账期判断 → 下采购通知 → 货到 → 发货申请（关联 SO 选单号）→ ★财务审核（货不能出仓）→ 出库发货 → 销项发票 → 收款到账标记 → 对账催收`

替代旧痛点：① 在途数量靠 Excel 公式手算、跨公司汇总靠人拼大表（访谈 09 L330-352）→ 系统按订单号自动减在途；② 合同/订单两件事手工对（访谈 09 L243-245「我们其实没有合同，都是订单」）→ 决策①合同与 SO 合一；③「财务不审先出货」靠纪律守（访谈 09 L142-163「坚决不可以，这是违例」）→ 状态机硬卡 ★财务放行；④ 收款记录散在飞书回款表（访谈 09 L685-694）→ 系统只记到账确认状态，做账推金蝶（决策④）。

**与上下游接力点**：上游 = 05a 报价单（含 PM 利润点）确认后落 SO；下游 = SO 推「采购通知」给 02 采购域（PA 接单下 PO，决策②采购在途）、发货申请消费 03b 出库单（SA 指定入仓编号→出库两道关）、销项发票配对出库 PL（决策③）；在途分二：本册管**销售在途（我方→客户·未交付）**，采购在途（原厂→我方）在 02 采购域（决策②）。

### 页面 1 · 合同（订单内签约信息 + 电子签章）

**定位与使用者**：SA 与 SA leader 使用。**决策①：无独立合同实体——订单即合同**（访谈 09 L243-245「我们其实没有合同，我们都是订单。订单跟合同都是一起录的，一件事情」）。合同降为**销售订单 SO 上的一组签约信息字段 + 双签合同附件 + 电子签章（一签宝）状态**，不单建表、不单列状态机。本页只描述这组信息及签章动作；承载它的状态机见页面 2。

**典型流程/场景**（引用蓝图 P01）：
1. 报价确认后，SA 制作合同 **或** 接收客户发来的合同（访谈 09 L232-235「要不然是他做出来的合同，要不然是客户发出来的合同」）。
2. **SA leader 下级审核 / 修正**（访谈 09 L46「录入之后，下级审核」、L232「审核、修正」）——这是**业务审核关卡（非财务关卡）**。
3. 审核通过 → 走**一签宝**申请印章（访谈 09 L232-235「最近上线了一个一签宝，申请印章」）：谁做合同谁带章；客户做来的带客户章，我方做去的带我方章。
4. 盖章合同回传客户**一个来回**，合同成立（访谈 09 L235「双方基本上一个来回，合同也就成立了」）。
5. 合同成立当天即录 SO（页面 2）；客户若用我方模板才用我方模板，否则迁就客户制式（访谈 09 L136-142）。

**字段表**（合同信息 = SO 上的字段组；不单独成表）

| 字段 | 类型 | 必填 | 选择器来源 | 校验 | 默认 | 扫码兜底 | 引擎映射 |

| 编号（=客户订单号） | str | 是 | — | 主单号、双方一致、不可改 | — | 可扫/可手填 | SO `external_order_no`（访谈 09 L46「编号是客户的订单编号，主单号，不可修改」） |

**状态机**：合同审核 + 盖章作为 **SO 状态机的前置节点**（见页面 2 `draft → leader_review → signing → 已签`），不单列。

**引擎映射 & 兼容性**
- ✅ 合同信息=SO 模型上的字段组（不单建 doc_type）；下级审核 + 签章 = SO 状态机前置节点（引擎 02 §2.2）。
- ➕ **双签合同附件 / invoice 附件**：引擎无附件/对象存储原生支持，需扩展「附件引用」字段 + 上传端点（与 03b 出库照片留证同一扩展）。**engineFlags**。
- ➕ **一签宝电子签章**：引擎无原生集成，落**外部签章状态字段 + 占位集成**（点「申请印章」调一签宝/回填盖章 PDF）；首期可人工上传盖章 PDF + 手工置状态。**engineFlags**（占位，待甲方确认一签宝是否有 API）。
- ✅ **下级审核（SA leader）**=状态节点 allowed_roles=[SA_LEADER]（业务审核，非 ADMIN/BOSS 绕过的财务关卡）。

**验收标准**：① 合同号自动=内部订单号且只读；② 编号（客户订单号）必填且全链只读不可改；③ 缺双签附件无法进「已签」；④ 签章状态可追踪。
**待甲方确认 gap**：一签宝是否提供 API（默认占位/人工上传盖章 PDF）；合同模板是否纳入 8 配置模板子系统（默认是）；客户制式合同的关键字段是否需结构化抽取（默认仅存附件 + 关键字段手录）。

### 页面 2 · 销售订单 SO ⭐（核心单据）

**定位与使用者**：SA 的核心录单单据 + 全链枢纽。**一张 SO = 订单 = 合同**（决策①）。承载：合同号（内部订单号 `FT-{YYMMDD}-{seq}` 或月度连号 `202603-022`）、编号（客户订单号）、客户/区域/供应商/PM、型号-数量-单价明细（网格）、币种/税率、付款方式（★预付 or 账期）、双签合同附件、电子签章状态、下采购通知、状态。下游派生：采购通知（→02）、发货申请（页面 4）、销项发票（页面 5）、销售在途（页面 6）。

> **内部订单号双格式并存**：签单大表 `SO-2024` sheet 实样含两种内部订单号——`FT-240118-109`（FT-年月日-流水）与客户订单号风格 `CG202401000216`。访谈 09 L51 强调的是**月度连号 `202603-022`（3 月第 22 单）用于统计当月签单数量**。本 PRD：**内部订单号采用月度连号规则**（总览 §7、决策①），`FT-{YYMMDD}-{seq}` 作为兼容显示别名待甲方确认统一口径。

**典型流程/场景**（端到端，引用蓝图 P01，★标财务硬关卡）：

`draft（SA 录单）→ leader_review（SA leader 下级审核）→ signing（一签宝盖章双签回传）→ 已签/待付款关 → [★预付到账（账期则跳过）] → 下采购通知（给 PA）→ executing（执行中，分批发货）→ closed（全部交付）`

1. **录单**（访谈 09 L38-40、L61）：SA 在主数据找/建客户（访谈 09 L64「新签单客户是新的就自己建」）；录单位/收款方式/合同号基本信息；**型号-数量-单价明细网格**逐行录（访谈 09 L40「客户下单的型号、数量、单价」）；带业务员（销售）、主管；附双签合同附件（访谈 09 L40、L99-106「一定是双签」）。
2. **产品代码渠道锁定**（访谈 09 L55-62）：一型号多代码时（PA 那边按供应商建，决策见 02 主数据），SA 录单时**找 PA 问本单用哪个代码**，把销售↔采购通过同一代码关联（「从这个环节把渠道锁定」）。→ 字段「产品代码」cell 选择器，建议带「本订单供应商」过滤。
3. **币种/税率联动**（访谈 09 L52-57）：美金不含税（单价=含税单价）；人民币带 13%/6%/9%/1% 税点（视具体而定）；税率为 0 则含税=不含税自动带出。
4. **保存 + 附件校验**（访谈 09 L61「审核时一定看是不是有上传附件」）：缺附件提醒（旧 ERP 不强制，新系统建议必填，gap）。
5. **下级审核**（页面 1）→ **一签宝盖章**（页面 1）。
6. **付款方式卡点**（访谈 09 L238「下面卡着的是付款方式」）：
   - **账期**（如 NET 30 / 月结 60 天）→ 合同成立即可**立刻下采购通知给 PA**（访谈 09 L238「存在账期，可以立刻把采购通知下达给 PA」）。
   - **预付（货前 100%）**→ **等客户付款（★预付到账）**才下采购通知（访谈 09 L238「货前 100% 就要等客户付款」）。★预付到账=本系统打到账确认标记驱动（决策④、总览 §6.2），到账确认单据/动作在 05 财务域，本 SO 状态机消费其「已到账」状态作为推进闸。
7. **执行中分批发货**（访谈 09 L364-394）：一个 SO 可分多次发货，每次发货由发货申请（页面 4）→出库扣减 SO 的**在途数量（未交付）**；全部交付后 closed。

**验收标准（可测）**

### 页面 4 · 发货申请 / 发货通知 ⭐

**定位与使用者**：SA 的发货入口单 + **★财务审核关卡载体**。货到后 SA 据客户/销售发货通知建单，**核心硬规则 = 必须关联 SO 选单号、不能凭空新增**（访谈 09 L73-79「发货通知不能凭空新增，一定从关联单据里选；锁死这个订单往外导」）。SA 提交 → **★财务审核（审发票内容 vs 发货内容一致，货不能出仓）**（访谈 09 L142-163、L255-259）→ 通知仓库出库（消费 03b 出库单）+ 通知客户（发货邮件）。

> **与 03b 出库单的关系**：发货申请=销售侧发起 + 财务放行的**业务关卡单**；03b 出库单=仓库侧作业单（分箱/拣货/标签/互检）。两者可合一为一条出库链——本册发货申请承载「关联 SO + 财务审核（货不能出仓）」，03b 承载「仓库互检 + 出库扣库存」。两道关（财务放行=本页、仓库互检=03b）合计=出库两道关（决策⑥）。**SA 在发货邮件中必须带出具体入仓编号**（访谈 09 L274「为确保仓库拿指定货，邮件带库存表入仓编码」、03b 串货隔离），即发货申请明细行=指定入仓编号（与 03b 拣货明细同源）。

**典型流程/场景**（端到端，引用蓝图 P01/P03，★标财务硬关卡）：

`draft（SA 关联 SO 选单号 + 指定入仓编号 + 挂 invoice）→ ★finance_review（财务审发票 vs 发货一致）→ released（财务放行/审核标志）→ [仓库出库=03b：互检★→出库扣库存]→ shipped（关闭标志=已完成出库）→ 通知客户 + 月底对账催收`

1. **关联 SO 选单号**（访谈 09 L73-79）：SA 新增发货通知，从关联单据（SO 订单号）选出 → 自动带出客户/型号/数量；**不允许凭空新增型号**（删掉选单自己加=没有勾稽关系，违内部要求）。
2. **指定入仓编号 + 明细网格**：逐行指定要发的入仓编号（精确到批次，串货隔离，访谈 09 L274、蓝图 §5.2）→ 反查 03b 库存批次带出型号/SN/性质/可用结存。
3. **挂 invoice 附件**（访谈 09 L92-94「添加附件=他们制作的一张 invoice」）：SA 制作销项发票（页面 5）挂到发货通知。
4. **★财务审核**（访谈 09 L255-259「财务对发票内容和发货内容审核，两者一致时审核发货通知单据」、L142-163「财务不点头货不能出仓，违例」）：财务比对 invoice vs 发货明细一致 → 置**审核标志**（财务已审）；不一致退回。**财务未审仓库只能准备、不能出货**（访谈 09 L150-163）。
5. **放行 → 仓库出库**（→03b 出库单互检 + 扣库存）：出库完成置**关闭标志**（访谈 09 L139「关闭标志=订单是否完成出库；物流在 ERP 完成勾稽」）。
6. **运输方式按贸易条件**（访谈 09 L295「依 price term：CFR 我方出运费自指定运输；FCA/EXW 客户指定」）：CFR→我方指定运输方式；FCA/EXW→客户指定（费用客户出）。送货单随箱**贴箱外**（国际件给海关看，访谈 09 L298-313）。
7. **通知客户 + 后续**（访谈 09 L167-169）：发货后邮件通知客户查收；物流回 packing list（PL，invoice 基础上加每箱毛/净重，访谈 09 L289）；进入月度对账/催收（→05 财务域）。

### 页面 4b · 特批发货（先发后补单）⭐（可隐藏模块，决策⑫）

**定位与使用者**：客户**尚未正式下单**、但因紧急/口头确认/战略客户等原因需**先发货**的极特殊场景下，SALES/SA 发起 → ★**FINANCE 财务特批审**（取代正常的「必关联 SO」前置）→ LOGISTICS 出货 → 客户正式下单后 SA **补录 SO 并把本特批出库勾稽到该 SO、补推金蝶**的闭环单。直接落地访谈 09 L82-91「极特殊情况，邮件特批，暂不上 ERP，后补」与 **L1147「客户未下单先发货走邮件特批，后补订单后补 ERP」**——把原来散在**邮件特批 + 事后手工补录 ERP**的灰色流程，收口成系统内**有审批关卡、有留痕、有强制补单勾稽**的受控单据，替代「不上系统靠纪律」。

> **可隐藏模块（硬要求）**：本页是**配置开关控制的可隐藏模块**——`feature.special_batch_shipment`（8 配置域 / per-company 开关），**上线默认 OFF（隐藏入口 + 隐藏 doc_type 创建权）**，按甲方上线后实际反馈再 per-company 启用。关闭时：导航不出此入口、发起动作不可达、已存在的历史特批单只读可查。**它是「发货申请必须关联 SO」硬规则（页面 4）的唯一受控例外**：正常发货零容忍凭空发货；特批发货以**财务特批审 + 强制事后补单勾稽**换取例外，二者互不冲突（特批走独立 doc_type，不污染正常发货申请的必关联硬规则）。

> **与页面 4「发货申请」的边界**：发货申请 = 有 SO 的正常出库链财务关；特批发货 = **无 SO 的例外出库链**，多一道**特批审（理由 + 风险 + 预计补单期限）**、且出库后挂**「待补单」债务**直到补录 SO 勾稽。两者都消费 03b 出库单作业（互检 + 扣库存），但特批发货的金蝶推送**先以「特批来源标记」出库、补单后回链 SO 补推/改推销售源**（见金蝶推送）。

**典型流程/场景**（端到端，★标财务硬关卡）：

`draft（SALES/SA 发起：客户+特批理由+风险承诺+预计补单期限+入仓编号明细，无 SO）→ ★finance_special_approval（财务特批审：核风险/授信/资质）→ approved（特批放行）→ [仓库出库=03b：互检★→出库扣库存，挂"特批来源/待补单"标记]→ shipped_pending_so（已发货·待补单：金蝶以"特批出库来源标记"先推)→ [客户正式下单→SA 补录 SO（页面 2）]→ reconciled（已补单勾稽：回填 SO 号、抵减 SO 在途、补推/改推金蝶销售源）→ closed`

1. **发起（无 SO）**（访谈 09 L82-91、L1147）：SALES/SA 在本模块新建特批发货——录**客户**（主数据，可现场建）、**特批理由**（紧急订单/口头确认/战略客户/样机急发…）、**风险承诺/授权人**、**预计补单期限**（默认 +30 天，逾期升级催办）、**入仓编号明细网格**（逐行指定要发的批次，串货隔离同页面 4）。**本单不关联 SO（无 SO 可选）**——这是与页面 4 的根本区别。
2. **★财务特批审**（取代「必关联 SO」前置）：财务在审批中心审**未下单先发货的风险**（授信额度/客户资质/特批理由是否成立/预计补单期限是否合理），通过 → 置特批放行标志；不通过 → 退回（货坚决不出仓，同页面 4 铁律）。**这是本模块存在的核心关卡**——把邮件特批收口成系统审批留痕。
3. **仓库出库**（→03b 出库单）：与正常发货同一作业链（互检★ + 扣库存 + 标签），出库单挂**「特批来源 / 待补单」标记**（`source_type=SPECIAL_BATCH`、`pending_so=true`）。出库后本单进 `shipped_pending_so`，库存照常扣减（货实物已出）。
4. **金蝶先推（特批来源标记）**：出库 shipped 时按总览 §6 推金蝶**出库/发货单**，但带**「特批出库来源标记 + 待补 SO」**字段——金蝶侧做账先记出库（存货减少），销售收入/应收源**待补单后回链**（避免无源单据；具体记账口径以金蝶接口为准，见 gap）。
5. **客户正式下单 → 补录 SO**（访谈 09 L91「后补订单」）：SA 按页面 2 录正式 SO（客户订单号/型号/数量/单价），**把本特批出库勾稽进该 SO**——回填本单 `补单SO号`、把已发数量**抵减该 SO 的在途**（视同该 SO 的一次发货）、勾稽明细行一致性校验（特批已发型号/数量 ⊆ 补录 SO 明细）。
6. **补推/改推金蝶（销售源）**（访谈 09 L91「后补 ERP」）：勾稽完成 → **补推金蝶销售订单 SO（应收/销售源）+ 把先前出库的「特批来源」回链到该 SO**（改推或补推，幂等键=补单 SO 号 / 原出库单号，见金蝶推送）。进 `reconciled` → `closed`。
7. **逾期未补单升级**：超过预计补单期限仍 `shipped_pending_so` 未勾稽 → 工作台红色待办 + 升级 SA leader / 财务（「待补单」债务可视化，避免发了货永远不补单的灰色挂账）。

**业务单据推送（金蝶）**（总览 §6；两段式：先发出库、后补销售源）

| 触发态 | 推送单据 | 幂等键 | 备注 |
|---|---|---|---|
| `shipped_pending_so`（财务特批放行 + 仓库出库后） | 出库 / 发货单（**带「特批来源 / 待补 SO」标记**） | 出库单号 `PD{YYMM}-{seq}`（03b） | 先记存货减少；销售/应收源**待补单后回链**；带 company_id（6 租户各推各组织） |
| `reconciled`（补录 SO 勾稽后） | **销售订单 SO（应收/销售源）** + 回链原出库单 | 补单 SO 内部订单号（合同号） | 补推销售源 + 把先前「特批来源」出库回链该 SO（改推/补推）；与页面 2 SO 推送同口径，幂等去重避免重复 |

> 落点：① `shipped_pending_so` 时 AUTO effect 写 `kingdee_outbox`（出库单，带来源标记，幂等键=出库单号 + company_id + 回执）；② `reconciled` 时 AUTO effect 写 `kingdee_outbox`（补单 SO，幂等键=补单 SO 号 + company_id），并发「回链」推送把原出库销售源挂到该 SO（总览 §6.3）。**业务单号=金蝶单据编号做幂等**（总览 §6.1），两段推送各自幂等、可重推、不静默丢单。金蝶侧「无源出库先做账、补单后补销售源」的**精确记账口径以 `/home/thinkbook/桌面/CRM+WMS/PHOTONTECK/金蝶供应链财务API离线文档.html` 为准**（见 gap）。

**验收标准（可测）**
1. 模块开关 OFF（默认）时：导航无「特批发货」入口、不能创建特批单；ON（per-company）时入口出现且可发起。
2. 特批发货发起**无需关联 SO**（与页面 4 相反），但必填特批理由+风险承诺+预计补单期限。
3. **财务未特批审（finance_special_approval）→ 无法进 approved → 仓库无法出库**（货不能出仓硬卡，同页面 4 铁律）。
4. 特批放行 + 仓库出库后 → 库存按入仓编号明细扣减；出库单挂「特批来源 / 待补 SO」标记；outbox 多一行待推金蝶出库单。
5. `shipped_pending_so` 进入「待补单」债务台账（`pending_so=true`）；超预计补单期限未勾稽 → 工作台红标升级催办。
6. 补录 SO 后勾稽：补单 SO 号必填、特批已发明细 ⊆ 补录 SO 明细（型号/数量勾稽一致）才能进 `reconciled`；勾稽后抵减补单 SO 在途、补推金蝶销售源、`pending_so=false`。
7. 全链留痕：发起人 / 特批理由 / 财务审批人 / 出库 / 补单勾稽四审计本可查（总览 §4.8）。

### 页面 5 · 销项发票管理 ⭐（开客户，决策③销项）

**定位与使用者**：SA 制作（访谈 09 L255「这个发票是 SA 这边制作的」），财务审核（发货通知关卡，页面 4）。一张销项发票（invoice）= 发票号 `I{YYMM}-{seq}`（**与 PL 同尾号配对**，总览 §7）+ 增值税发票号（人民币单需要）+ 开票时间 + bill to（客户）/ ship to（接货地址）/ price term / 付款方式 / 货物明细。决策③：发票拆销项（本页）/ 进项（02 采购域），都推金蝶。

**典型流程/场景**（引用蓝图 P01）：
1. 货入库 → 触发 SA 下一步（访谈 09 L255「货入库后触发 SA，及时发给客户，制作一张发票」）。
2. SA 按公司模板制 invoice（访谈 09 L266-268「这是我们其中一个公司的模板，不是所有模板都长这样」）：含 invoice 号、invoice date、bill to（客户）、ship to（接货地址）、联系人、发货日期、price term、付款方式、货物基本信息（访谈 09 L268）；中文品名给海关参考（memo，非正文，访谈 09 L274）。
3. invoice 挂发货通知（页面 4）→ 财务审 invoice vs 发货一致（页面 4 ★关卡）。
4. 物流发货后回 packing list（PL，invoice 加每箱毛/净重，访谈 09 L289）；报关单号 i→e 替换（出库 03b L489-491）。
5. **增值税发票号**：人民币单含 13%/6%/9%/1% 税点（访谈 09 L52-57）；增值税发票号在开票时回填（签单大表「增值税发票号」「开票时间」列）。
6. 财务审核通过 → **推金蝶（应收/开票源，决策③）**。

### 页面 6 · 销售在途（我方→客户·未交付）

**定位与使用者**：SA / 销售 / SA leader / 财务的「未交付」视图。**决策②：在途拆二——本页 = 销售在途（我方与客户签的订单中尚未发货的部分，客户视角）**；采购在途（原厂→我方）在 02 采购域。访谈 09 L364-394 精确定义：「这是我们跟客户签的订单……每发一次扣减一次……在途数量=未发货数量……是他的在途（客户视角）」。

**典型流程/场景**：
1. SA 看本部门**还有多少货没交**（访谈 09 L376「不能只看签了多少单，还要看还有多少货没交」）。
2. **自动算**：销售在途 = SO 订单数量 − 该 SO 已发货数量合计（按订单号关联发货记录，访谈 09 L364「每发一次发货时间跳一次、在途数量扣减一次」）；一个 SO 分多次发（访谈 09 L364「这一张订单已发了 5 次货，可能还没发完」）。
3. 全部发完（在途=0）→ SO closed（页面 2）。
4. 在途 → 下推应收/催收（访谈 09 L394「从发货再下推到应收、催收」）。

**字段表**（视图列，非独立录入）

| 字段 | 来源 | 类型 | 说明 / 引擎映射 |
|---|---|---|---|
| 订单号（客户订单号） | SO | str | 关联键 |
| 内部订单号 | SO | str | — |
| 客户 / 型号 / 供应商 | SO | str | 带出 |
| 订单数量 | SO 明细 | num | 在途基数 |
| 已发货数量合计 | 发货申请聚合 | num | 按订单号 sum（访谈 09 L364） |
| **在途数量** | 派生 | num | 订单数量 − 已发货数量（**核心派生列**） |
| 在途金额 | 派生 | num | 在途数量 × 单价 |
| 最近发货时间 | 发货申请 | date | — |
| 预计回款日 | 收款 | date | 账期推算（访谈 09 L424「月结 60 天：6/9 发货→6/30 月结→8/30 收款」） |

**引擎映射 & 兼容性**
- ✅ 销售在途=**派生视图**（不单建单据）：SO `doc_type` + 发货聚合，DataExplorer 只读 + `aggregate`（引擎 08 §8.3、引擎 01 §1.4.2）。
- ➕ **在途数量自动派生（SO 数量 − 已发货 sum）** = 跨单据聚合：发货 shipped 时 AUTO effect 扣 SO 在途（页面 4），或报表路由实时算（引擎报表非元数据化，引擎 00 D7）。本 PRD 建议**effect 维护 SO 上"已发货数量/在途数量"派生字段**（发货 shipped 时更新），台账实时读，避免每次报表重算。**engineFlags**。
- ✅ 与采购在途区分（决策②）：两个不同视图/不同 doc_type 来源（销售在途源 SO+发货，采购在途源 PO+入库，在 02 采购域），不混。
- ❌ **无**。

**验收标准**
1. 在途数量 = 订单数量 − 已发货合计，分批发货后实时减。
2. 一个 SO 多次发货，每次扣减；全发完在途=0、SO 可 closed。
3. 与采购在途分两视图、口径不混（客户视角 vs 原厂视角）。

**待甲方确认 gap**
- 在途派生用 effect 维护字段 vs 实时报表算（默认 effect 维护字段，量大更稳）。
- 预计回款日是否按账期自动推算（默认按付款方式 + 月结日推算，访谈 09 L424）。

### 页面 7 · 客户 Forecast 接单对接（占位，决策⑫）

**定位与使用者**：SA 在**客户供应商系统**上的接单/回货期/查 Forecast 工作（访谈 09 L25「进入客户供应商管理系统：接单、货期回复、看 forecast、交付申请」、L64-70「客户在上面发预测，我们去回，内部货期 vs 客户需求做交叉应答」）。**决策⑫范围待定 → 本系统轻量占位**：客户 Forecast 在**外部客户系统**（如烽火多系统、其他客户一套系统），本系统不直连客户系统，仅**占位登记接单/Forecast 应答 + 提醒**，主骨架先行。

**典型流程/场景**（占位，引用访谈 09 L64-70）：
1. SA 在客户供应商系统接单（外部系统）；本系统**占位登记**「已在客户系统接单」。
2. 客户系统发 Forecast → SA 取**内部货期（PA 回的）vs 客户需求**做交叉应答，看 gap（访谈 09 L66-70）→ 该提拉提拉、该推迟推迟，减少库存存量。
3. 本系统占位：登记 Forecast 应答（型号/客户需求量/我方应答货期/gap/动作=提拉/推迟），作为备货/采购的输入提醒（与备货申请 02 采购域联动占位）。

**字段表**（占位，最小骨架）

| 字段 | 类型 | 必填 | 选择器来源 | 校验 | 引擎映射 |
|---|---|---|---|---|---|
| 客户 | fk | 是 | 客户主数据 | 本公司 | fk |
| 客户系统名 | enum | 否 | 烽火/通用一套系统/… | — | 占位（访谈 09 L25「烽火多系统」） |
| 型号 / 产品代码 | fk | 是 | 产品 | — | fk |
| 客户需求量 / 需求货期 | num/date | 否 | 手录（来自客户系统） | — | 外部系统抄录 |
| 我方应答货期 | date | 否 | PA 内部货期 | — | 来自采购（访谈 09 L66） |
| gap（天） | num | 自动 | — | 应答−需求 | 计算值 |
| 动作 | enum | 否 | 提拉/推迟/正常 | — | 减库存存量（访谈 09 L70） |

**引擎映射 & 兼容性**
- ✅ 占位单 = 轻量 `doc_type` 或 `__queryable__` 登记模型（引擎 01 §1.3）。
- ➕ **直连客户供应商系统（接单/拉 Forecast）** = 外部系统集成，各客户系统不同（烽火多系统 vs 通用一套），**引擎无、范围待定**。占位不直连，仅手录抄写。**engineFlags / 占位**。
- ❌ **无**（占位不硬实现）。

**验收标准（占位）**：① 能登记 Forecast 应答（需求 vs 应答 vs gap vs 动作）；② 作为备货/采购输入提醒（联动占位）。
**待甲方确认 gap**：
- Forecast 接单是否纳入系统范围 / 颗粒度（决策⑫范围待定，默认占位手录）。
- 是否直连客户供应商系统（各客户系统异构，烽火多系统，默认不直连）。
- Forecast → 备货申请（02 采购域）的联动是否需自动（默认提醒，不自动建单）。
