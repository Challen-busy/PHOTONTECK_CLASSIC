## 模块定位（业务价值一句话）

把现在散在飞书多维表 / Excel（备货申请表 + 巨大消单总表、SDN Record 表、RMA record 表）+ 微信邮件里的三条"非主线但高风险"流程收口为有状态机、有关卡、有留痕的单据：
- **备货**：让"谁提议谁担风险、谁买谁跟到底"（访谈 07:138/2369）落到一张备货申请单上——金额自动判断走 PM/PD 单批还是 **★≥20万会审多签**，下单后**原始备货数量 vs 最新跟踪消单**持续提醒，避免"一直挂在备货没人卖"（访谈 07:2369）。
- **样品 SDN**：把"申请→原厂寄→入样品仓→寄客户→跟回签→超期催→测试通过转正"全程记录，**超期天数=计算字段自动算 + 提醒**（替代手工算 SDN Record 的"超期天数"列），转正后**可下正式单**。
- **RMA**：一张**统一单共用 RMA 号**，SA 看客户侧（报修/退客户）、PA 看货物侧（核 SN/LOT+PO+过保、报原厂决策），同一单两视图（决策⑨）；RMA 好货**回库带来源标记→可混回可售**（对接库存状态模型 §5.4 / 03 库存）。

## 04b-1 备货申请（StockUpRequest）⭐

### 定位与使用者
销售或 PM 主动提议为某（意向）客户囤货时填的申请单。核心是**风险管控前置**：申请时就要把"给谁备、备多少、当下库存/在途多少、客户欠款情况、风险点、金额"讲清楚（访谈 07:2196）；金额小由 PM/PD（直属领导）单批即可下单，**金额 ≥20万走备货评审会多部门会签**（访谈 07:2196、蓝图备货分支）。批准后由 PA 下 PO，并进入**持续消单跟踪**（见 04b-2）。

### 典型流程 / 场景（端到端，引用蓝图「备货（P02 分支）」）

1. **发起**（访谈 07:1593/2202）：SALES 或 PM 发起——"业务的两块都有权利发起备货流程"。填型号、备货数量、意向客户、签单公司、原因、风险点、金额；系统**自动带出当时库存 + 在途**（从库存台账 + 采购在途投影，访谈 07:2196"搞清楚给谁备/风险点/客户情况"）。
2. **填客户欠款情况**：从应收视图（§00-5 财务视图）带出该意向客户当前欠款/超额，作为风险判断输入（访谈 07:2196"他前面一些订单的情况"）。
3. **金额阈值分支**（关键关卡，决策⑤吸收进审批中心）：
   - **金额 < 20万**：到 **PM/PD（直属领导）单批**节点；批了 → "可以下了"（访谈 07:2196/2276）。
   - **★金额 ≥ 20万**：进 **备货会审多签节点**——多部门会签控风险（访谈 07:2196"金额比较大有一个备货的评审会来管控风险"）。会审需同时通过 **PM/PD + FINANCE 财务**（财务关注欠款/风险/资金占用）等会签角色；全部通过才"可以下"。
4. **PA 先做订单给会审看**（访谈 07:2288/2345）："他评审就是看我的申请单和我的订单，我会先做一个订单给出来给他们看"——会审时关联预览即将下的 PO（草稿）。
5. **批准 → 下单**：进 `APPROVED`，PA 据此下正式 PO（PO 主流程在 04a；备货 PO 的 `关联销售订单` 为空、`是否备货=是`，蓝图 §3.3）。**未通过 → 不下单**（访谈 07:2196"没有通过就不下"）。
6. **消单跟踪**：下单后单据进 `TRACKING`，**原始备货数量保持不变，新增"最新跟踪/已消数量"字段持续更新**——客户真有订单过来就消掉对应数量（访谈 07:2019/2028/2369）；详见 04b-2。
7. **关闭**：备货全部被客户订单消化（已消=备货数量）→ `CLOSED`。

> ★ 财务硬关卡：**仅 ≥20万 会审节点含财务会签**（FINANCE 在会审多签里）；<20万 无财务关卡（PM/PD 自批）。这是本模块唯一的财务关卡，对齐蓝图 §5.1"备货会审(≥20万)"。

### 状态机 `STOCK_UP_REQUEST`

| 状态 | 可操作角色 | 可编辑字段 | 硬规则 / 关卡 | 下一状态 |
|---|---|---|---|---|
| `DRAFT`（初始）| SALES/PM | 全部业务字段 | 风险点/金额/意向客户必填才能提交 | →`PENDING_PM`（<20万）/ →`PENDING_REVIEW`（★≥20万）|
| `PENDING_PM`（<20万单批）| PM(PD) | 批注 | — | →`APPROVED`（批）/ →`REJECTED`（驳）|
| `PENDING_REVIEW`（★≥20万会审）| PM(PD)+FINANCE（多签）| 会签意见、draft_po 关联 | **多签会审：PM/PD 与 FINANCE 都通过才放行**（财务关注欠款/资金占用）| →`APPROVED`（全签）/ →`REJECTED`（任一驳）|
| `APPROVED` | PA | draft_po→正式 PO | 下单后自动进跟踪 | →`TRACKING` |
| `TRACKING`（消单中）| PA | consumed_quantity（由消单 effect 写）| 已消<备货数量则持续提醒 | →`CLOSED`（已消=备货）/ 可 →`CANCELLED` |
| `CLOSED` / `REJECTED` / `CANCELLED`（终态）| — | — | — | — |

> ★关卡说明：`PENDING_REVIEW` 是**多签会审节点**——引擎单节点 `allowed_roles` 是"满足其一即可操作"，**真正的"多人都要签"需拆为串行会签子状态或用计数 hard_rule**（见兼容性 ➕）。金额阈值（<20万 / ≥20万）走**两条出边 + 边级 hard_rules**（`doc.amount < 200000` / `>=`）自动分流。

### 引擎映射 & 兼容性（逐条）

- ➕ **新增 doc_type `STOCK_UP_REQUEST`**：一个 `Base` 子类 `__doc_types__=("STOCK_UP_REQUEST",)` + 一次 alembic 迁移 + 一行 `WorkflowDefinition`（引擎 09 §9.1）。引擎无现成模型 → 全新增，但是标准加单据动作，不破坏底座。
- ➕ **金额阈值分流**：`DRAFT` 的两条出边各带边级 `hard_rules:["doc.amount < 200000"]` / `["doc.amount >= 200000"]`（rules DSL 只读断言，引擎 04 §4.B），预览/枚举时按金额给出可走的边。✅ DSL 支持算术/比较。
- ➕➕ **≥20万多签会审**：引擎节点 `allowed_roles` 语义是"角色命中即可推进"，**不原生支持"N 个角色都必须签"**。落点二选一（待甲方定）：① 拆**串行会签子状态** `REVIEW_PM`→`REVIEW_FINANCE`→`APPROVED`（每态一个 allowed_roles，最稳妥、可兼容）；② 单节点 + **会签计数 hard_rule**（子表记每人签字、hard_rule 断言"PM 与 FINANCE 都已签"）。**推荐方案①**（落在底座现有能力内，纯 ✅）。**必入 engineFlags**。
- ➕ **当时库存/在途快照**：申请创建时由 effect 拍下库存台账 + 采购在途投影写入只读快照列（引擎 04 effect 可读跨表 `lookup/query`）。
- ➕ **消单 effect（跨单据派生）**：销售订单成交并匹配到该备货型号+意向客户时，`@register_transition_effect`（EXPLICIT 或 AUTO 匹配 SO 推进态）累加 `consumed_quantity`（访谈 07:2028/2369）。**幂等/单事务**继承命令四件套（引擎 03）。**这是跨单据派生，必入 engineFlags**。
- ⚠️ **边级角色推进未校验（D-02e）**：会审若用边级 `roles`，引擎 `execute_transition` 推进路径当前只校验节点级 → 须标 FR-2.7 补齐；用方案①串行子状态（节点级 `allowed_roles`）则**规避此坑**，更稳。
- ✅ **审批中心收件箱**：备货 `PENDING_PM`/`PENDING_REVIEW` 节点单据由 `list_user_todos` 推到对应角色"我的审批"（§00-4.7），PM/财务在审批中心推进（预览→ChangeCard→commit）。
- ✅ **行级公司隔离**：`company_id`（签单公司）使备货绝不跨公司（§00-9）；read-privileged（BOSS/财务总监）可跨公司只读看占用。

### 业务单据推送（金蝶）
备货申请本身**不直接推金蝶**（它是内部决策单）；批准后**下的 PO 推金蝶**（PO 审批通过触发，§00-6.2），备货 PO 的 `是否备货=是`、`关联销售订单` 空。后续客户下单消化时，正式 SO/出库各自按主流程推送。

### 验收标准（可测） + 待甲方确认 gap
- ✅ 金额填 19.9万 → 走 `PENDING_PM`；填 20万 → 走 `PENDING_REVIEW`（边级 hard_rule 分流）。
- ✅ `PENDING_REVIEW` 出现在 PM 与财务的审批中心待办；任一驳回则整单 `REJECTED`。
- ✅ 原始 `stockup_quantity` 在消单过程中**不变**，`consumed_quantity` 随匹配 SO 累加。
- ✅ SALES 发起人查看该单看不到成本/买价/利润点列（字段防火墙 query+schema 两路遮蔽）。
- gap-1：**阈值 20万 的币种口径**（RMB/USD/HKD 是否各自 20万、还是统一折算）——蓝图只说"≥20万"，需甲方确认；默认按签单公司本币 20万，跨币种折算规则待定。

## 04b-2 备货消单跟踪台账（原始 vs 最新）➕

### 定位与使用者
PA / PM 用来回答"我备的货卖掉没、还剩多少、备了多久还没消"。本质是 `STOCK_UP_REQUEST` 处于 `TRACKING` 态的单据的**台账视图 + 提醒**，不是另一张表（决策⑦：明细为真相、汇总=同库视图）。替代现状"一个巨大的表去登/维护订单和跟进备货"（访谈 07:2028"所有备货我们要很清楚，他有没有及时备、卖掉没"）。

## 04b-3 样品 SDN（申请 / 回签 / 超期 / 转正）⭐

### 定位与使用者
PA 用来登记和跟进每一批样品。来源：销售向客户要样品 → 找产品部 → PM 决定是否申请、用哪个客户身份申请 → PA 向原厂申请（通常免费、数量很少，不符合 MOQ/MPQ，访谈 07:1278/2603）。样品**走"其他入库"进样品仓**（无 PO 号，仓库靠"无 PO"判定为样品，访谈 07:1278/2576），寄给客户后**跟回签、算超期、跟测试结果**；**测试通过 + code 建好 → 可下正式单**（访谈 07:2567）。**权威字段 = `SDN Record.xlsx` 31 列表头**。

### 典型流程 / 场景（端到端，引用蓝图「样品（SDN）」）

1. **申请**（访谈 07:2564/2603）：销售发起申请（和询价一样，客户找他要样品）→ 产品部确定要不要申请、用哪个客户身份申请 → PA 向原厂申请（如"某型号申请 20 片"）。生成 **SDN 号 `SDN-{C/L}-{YYMM}-{NNN}`**（中间字母=供应商线，§00-7；如 `SDN-C-2605-004` C 线、`SDN-L-2605-009` L 线，见 SDN Record 实样）。
2. **原厂发货 → 其他入库（样品仓）**：原厂把样品随其他货发来、无 PO 号；物流按"无 PO + 数量很少"判为样品，走**其他入库**进**样品仓**（库存状态=`SAMPLE`，§5.4 / 03）。香港"会调拨到样品库再发"（访谈 07:1287/1320）。
3. **寄客户**：PA 跑物流把样品发客户，记 Tracking。
4. **跟回签**：跟进客户**签收 / 回签**（SDN signed return：Y/N，访谈 07:2564"记下来给客户、跟进有没有签收"）；记客户侧 SN/LOT（"也会记 SN note，原厂也要管控"，访谈 07:2564）。
5. **超期计算**：对"需归还 / 未回签"的样品，**超期天数 = 今天 - 应回签/应归还基准日，自动计算 + 提醒催归还**（替代手算 SDN Record"超期天数"列，蓝图 §3.3"超期天数(计算值)"）。
6. **样品性质处理**（SDN Record"样品性质"+"非免费样品是否归还或转销售"列）：
   - **免费（Free）**：客户不付款；多为 PCN 样品 / Dummy 样品 / 机械样品。
   - **非免费（Not Free）**：是否**归还**（如 EVK 评估板 → 已归还）或**转销售**——"需归还可催，可转他客户不可卖；测试通过转正可下正式单"（蓝图样品流）。
7. **测试结果跟进**：发客户后由 PM + 销售沟通测试情况（项目状态：在测/通过/失败，访谈 07:2564"产品经理跟销售沟通样品测试情况"）。
8. **转正**：样品测试通过 + 内部 code 建好 → **样品转正、客户可下正式单**（访谈 07:2567）；库存上"样品→可售"（§5.4 样品转销售）。

## 04b-4 样品超期 / 回签提醒（看板 / 待办）➕

### 定位与使用者
PA 用来看"哪些样品该催回签 / 该催归还 / 测试拖太久"。是 04b-3 的提醒面，不是独立单据。

### 引擎映射 & 兼容性
- ✅ **待办来源**：`list_user_todos` 把 `SENT_TO_CUSTOMER`（待回签）、`RETURNED` 前需归还的、`TESTING`（待跟进）单据按角色推给 PA/PM（引擎 02 §2.5）。

## 04b-5 RMA 退货统一单（SA/PA 双视图，共用 RMA 号）⭐

### 定位与使用者
**一张统一单 = 一个共用 RMA 号**，SA 与 PA 看**同一单的不同视图**（决策⑨）：
- **SA（销售助理）视图**：客户侧——客户报来的失效描述、退回数量、最终退/换给客户、关闭。**对客户、不对原厂**（访谈 07:2429）。
- **PA（采购助理）视图**：货物侧——核 SN/LOT + PO + 入库判"是否我方卖、是否过保"、建 RMA 号、上报 PM、对接原厂换/退/修、跟货回入库。**对原厂、不对客户**（访谈 07:2432）。

替代现状 RMA record Excel（24 列）。**权威字段 = `RMA record.xlsx` 24 列表头**。流程：客户报 CC/销售 → PA 核料 → 建 RMA → 上报 PM → 报原厂(换/退/修) or 内部消化 → 货回入库带来源标记 → SA 退客户 → 关闭（蓝图 P05、访谈 07:2405~2456）。

### 典型流程 / 场景（端到端，引用蓝图 P05）

1. **客户报修**（访谈 07:2426）：客户先联系 CC / 销售，报某批货有问题（失效描述、失效位置）；SA 接收客户信息（销售对客户）。
2. **PA 核料判定**（★是否成立，访谈 07:2405/2435/2450）：PA 凭 **SN/LOT + PO + 入库记录**核——"这只东西是不是我们公司卖出去的？哪个供应商？是否过保？"——靠**SN/LOT 管理 + PO + 发货时间**全链路倒查（访谈 07:2435/2450"通过内部数据直接导出发货时间、是不是我们发的、当时哪个客户"）。
   - **不是我方卖的**（客户多渠道拿货）→ **不收、不成立**（访谈 07:2438/2444）。
   - **过保了**（核发货时间 vs 质保期）→ 内部不算 RMA（访谈 07:2435"过了保在内部就不算了"）。
   - 成立 → **建 RMA 号 `RMA-{YYMM}-{NNN}`**（内部档案号，访谈 07:2411）。
3. **登记到位 + 上报 PM**（访谈 07:2417）：把失效描述、SN/LOT、PO#、发货时间、是否过保等登记后，**上报 PM；PM 决定是否报原厂**。
4. **PM 决策分叉**（决策⑨核心，访谈 07:2417/2423）：
   - **报原厂**：原厂发商务邮件确认是否接受这支 RMA；接受 → 后续**换 / 退 / 修**（RMA record"RMA# Supplier"列记原厂 RMA 号）。
   - **内部消化**：PM 觉得算了不报原厂 → 公司内部跟客户做退换，商务流程**停留在富泰**。
5. **跨境退运（物流/报关侧）**（访谈 07:2456）：货是报关进来的，退不能直接退、要走**退关**——海关审批后 **180 天内必须回来**（提醒）；**香港卖的退回香港公司**（绝不内地退香港，对应一笔生意不跨公司）。退关流程在 04 报关域，本单只记退运状态 + 180 天预警关联。
6. **货回入库**（访谈 07:1120/2453）：原厂换回 / 客户退回的货回到仓库，走入库**带来源标记**（RMA来源：客退/原厂换/原厂修/返工 + 品质好/坏 + 哪个原厂，§5.4）；好货混回可售、坏货报废/返原厂。详见 04b-6。
7. **SA 退客户 + 关闭**（访谈 07:2423）：货回来后**告诉对应 SA 还给客户**；SA 退/换给客户后，PA **把 RMA 关闭**（status=Closed）。

> 角色防火墙铁律（访谈 07:2429/2432）：**采购(PA)这里不对客户、销售(SA)那里不对原厂**——统一单靠字段级视图隔离实现这条铁律。

- ✅ **SA/PA 双视图 = 字段防火墙**：(RMA 表 × SA 角色) 遮蔽采购侧列、(RMA 表 × PA 角色) 给全列——引擎序列化按 (表,角色) 删列，query+schema 两路一致（引擎 01 §1.5、05 §5.3）。**这是引擎原生能力，✅compatible**——决策⑨无需新机制。配置落 `_can_view_*` / 业务侧列可见集（§00-8）。
- ✅ **核料倒查（是否我方卖/过保）**：靠 SN/LOT+PO+入库全链——可用 hard_rules 的跨表 helper `lookup/query`（引擎 04 §4.B）断言"该 SN 存在我方出库记录"，或 effect 自动判定写 `sold_by_us`。**过保 = `ship_date + 质保期 < today`** 计算。➕判定逻辑（effect/rule），✅DSL 支持。

- ✅ **行级隔离**：RMA 带 `company_id`——**香港卖的退回香港公司**（访谈 07:2456）天然由公司隔离保证（一笔生意不跨公司，§00-1.2）。

### 业务单据推送（金蝶）
- RMA 货回入库（其他/退货入库单）按 §00-6.2"其他入库单（样品/退货入库）"在**入库审核通过**推金蝶（入账）。
- RMA 退客户若涉及退款/换货开票（Credit Note）走财务侧（§00-5），本单触发但做账在金蝶；RMA 申请单本身不单独推。

### 验收标准 + gap
- ✅ SA 登录看 RMA 单**看不到** `unit_price`/采购侧列；PA 登录看全列（字段防火墙）。
- ✅ 同一 RMA 号在 SA 视图与 PA 视图状态一致、共享同一条流转日志。
- ✅ PA 核料判"非我方卖"→ 单据 `REJECTED`、不进 PM。
- ✅ PM 决策"报原厂"→ 进 `VENDOR_RMA`、可填原厂 RMA 号；"内部消化"→ 进 `INTERNAL`。
- ✅ 货回入库批次带 RMA 来源标记，好货以 AVAILABLE 出现在库存（§5.4）。
- gap-7：核料**自动判定 vs 人工判定**（是否我方卖/过保自动算还是 PA 手工勾）——默认 effect 自动倒查给建议、PA 确认，待甲方。
- gap-8：RMA 是否需开 **Credit Note / 退款**的财务联动颗粒度（蓝图 §3.6 Credit Note 在财务域）——本模块只触发、做账在金蝶，具体联动待 05 财务模块 + 甲方确认。
- gap-9：内部消化时客户退换的**库存损失记账**口径（公司损失，访谈 07:1051"买保险/公司损失"）——本系统只标记，损益在金蝶，待甲方。

## 04b-6 RMA 货回入库 + 来源标记（对接库存）➕

### 定位与使用者
RMA 流程里"货回来了"这一步的落地：货物物理回到仓库 → 走入库 → 在库存批次上**打来源标记** → 由人决策好货是否混回可售。是 04b-5 `GOODS_RETURNED` 推进的派生动作 + 03 库存的衔接点（对接库存状态模型 §5.4）。

### 典型流程
1. RMA 进 `GOODS_RETURNED`（原厂换回 / 客户退回的货到仓）。
2. 物流收货走**其他入库 / 退货入库**（`inbound_type=退货入库`，03 入库 03a-1 已有该类型 ✅）；**不再挂原订单**（访谈 07:1120"它就不再挂在我的订单下面，描述为 MA 换"）。
3. 入库批次写**来源标记 `source_marker`**：RMA来源(客退/原厂换/原厂修/返工) + 品质(好/坏) + 哪个原厂（§5.4 来源/品质标记，访谈 07:1120/1126）。
4. **品质判定**（FAE/PM 配合，访谈 07:1075"判定标准找产品经理"）：好货 → status=AVAILABLE **混回正常库但保留来源标记，由人决策是否使用**（§5.4 甲方确认）；坏货 → 报废或返原厂。
5. 好货后续可正常出库（"只要可以用的都可以用"，访谈 07:1132）——出库时来源标记可筛选/人工决策。

### 引擎映射 & 兼容性
- ✅ **退货入库类型**：`inbound_type=退货入库` 已在 03a-1 入库枚举（访谈 07:1248 入库表"性质"列含 RMA 类别）。
- ➕ **来源标记 `source_marker` + 原厂报备客户**：03 库存 03a-3 已规划 ➕ `inventory.source_marker`（驱动"RMA好货保留标记/筛选"）——本模块的 `GOODS_RETURNED` effect 是**写入方**，对接同一列。**与 03 共用扩展点**，必入 engineFlags（在 03 已登记，此处引用）。
- ➕ **RMA→入库派生 effect**：`GOODS_RETURNED` 推进 `@register_transition_effect` 生成退货入库单 + 库存批次带标记。AUTO/EXPLICIT，挂同一命令审计链（引擎 04 §4.A）。
- ✅ **好货混回可售**：库存状态置 AVAILABLE + 保留 source_marker（不自动拦截，人工决策，§5.4），符合甲方确认。
- ❌→不适用 **不强做 FIFO/不强制隔离 RMA 库**：出库不区分 RMA 库与成品库（访谈 07:1129"只要可以用都可以用"）——业务取向，不在出库写死 RMA 隔离；列 gap 提醒。

### 验收标准 + gap
- ✅ RMA 货回生成退货入库单、批次带 source_marker（含原厂 + 好/坏）。
- ✅ 好货 status=AVAILABLE 可出库；坏货不可售。
- gap-10：品质好坏的**判定人/动作**（FAE/PM 在哪个界面打好坏标记）——默认入库时由 PA 据 PM/FAE 邮件结论录，待甲方确认是否要独立质检节点。
