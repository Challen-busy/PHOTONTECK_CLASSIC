## 模块定位

「配置 / 模板」是**系统管理员（ADMIN）**与少量业务主管（物流主任维护标签/单据模板、财务参与编号规则）使用的**后台配置中心**，把现在散落在 Excel 里**靠人脑记忆**的客户规格收口成**可配置、可审计、不改代码**的元数据。它解决四类反复出现的痛点：

1. **标签千客千面**：~21 家客户、每家 2~5 种标签（包装1/2、外箱、公司外箱），字段集、字段顺序、二维码拼接分隔符（`&` `;` `+` `,` `_`）、纸张尺寸各不相同，且**随时新增客户/规格**（蓝图 §6、访谈 05 说话人2「它每一个客户都有，他们都要求不一样的资料」）。现状是仓库同事**打开上一单的模板手改数据**（访谈 05「用上一个那模板，然后把它变更成我自己的资料」）——极易错且不可审计。
2. **单据千客千面**：PL / INV / 送货单按客户 + 按公司不同，含盖章版、回签流转（访谈 05「需要盖章版的 word……扫描」）。
3. **编号规则**：公司 × 单据类型 × 前缀 × **月度重置 × 连号**（用于按号计量月度业务量），现状靠人在 Excel「每个月我们自己做出来的表格里挑选一个号码」（访谈 01 说话人4），引擎仅给 `*_number` 唯一前缀，不支持月度重置连号。
4. **审批流配置**：审批节点 / 会签人 / 阈值（如备货 ≥20 万走会审）需可配——引擎天然以 `WorkflowDefinition.states` 承载，是本模块**唯一 ✅ 原生兼容**项。
5. **提成规则配置**：公式/比例甲方未提供，**占位待甲方**（决策⑫、00 总览 gap 3）。

> 设计基调（贯穿本模块）：**模板/编号/审批流都是"元数据 + 数据"，不是代码**。新增一个客户标签、改一个分隔符、加一条编号前缀、调一个审批阈值，都在本配置中心点选完成，**不需要工程改代码、不需要发版**。这正是引擎"元数据优先于代码"哲学（引擎 00 §0.2 INV-3）在配置域的落地——但标签/单据/编号三块引擎**无原生子系统**，是本 PRD 的核心 ➕扩展点（见各页引擎映射 + 文末 engineFlags）。

### 边界（本模块不做）

- **不做财务做账配置**：税率/抬头作为发票模板的字段在本模块配，但凭证科目/账套规则在金蝶云星空（00 总览 §1.2）。
- **不做主数据 CRUD**：客户/供应商/产品/产线/库位在「7 主数据」域（`02-主数据.md`）。本模块的标签/单据模板**引用**主数据（如客户、公司），不重复维护。
- **不实际打印**：本模块产出"渲染好的标签/单据 HTML/数据"，物理打印走仓库本地打印机（BarTender 类，访谈材料 图片5.png 即 BarTender 模板截图）。系统侧负责**把数据按客户规则拼装正确**，打印驱动是占位（待甲方确认打印机集成方式，gap）。
- **不做扫码枪固件配置**：扫码枪每供应商 iCode 配置码（CINO FuzzyScan A680，访谈 01「每个供应商都有我们自己做的 QR 码……先确认扫描了之后才能识别」）是**枪端硬件配置**，不在系统内；系统只需在「供应商主数据」挂一个"扫码枪 iCode 附件"字段供仓库下载/打印（轻量，归 7 主数据，本模块仅备注）。

## 9.1 标签模板

### 定位与使用者
**定位**：把"~21 客户 × 多标签类型 × 各异字段/顺序/分隔符/尺寸"的标签规格做成**可配模板**，新增客户/规格**不改代码**（蓝图 §6）。**使用者**：ADMIN / 物流主任（配置模板本体）、物流专员（出库/入库时选模板→灌当前单数据→渲染→打印）。

**为什么必须可配（一手源证据）**：
- 主清单 xlsx 列出客户与标签类型矩阵（节选，共 ~21 客户 / ~40 个客户×标签类型组合）：Accelink 光迅（包装/外箱/公司外箱）、EOPTOLINK 新易盛（包装1/包装2/外箱/公司外箱1/公司外箱2）、HGO、Linktel、ZTE、SG TFC（含外箱2）、Taclink（包装1/包装2/外箱）、HISENSE 海信、FIBERHOME 烽火、INNOLIGHT（包装1/包装2/外箱1/外箱2）、XUNTE、ACCELIGHT（标签1/标签2）、POTRON、成都储翰 Tsuhan、NB GLOBAL（内标/外标）、SOURCE（×3）、Tri-light（包装/外箱1/外箱2）。
- 字段集每客户不同：EOPTOLINK 包装标签1 = 型号 / LOT NO / 数量 / 数量单位 / DATECODE / COO；INNOLIGHT 外箱标签 = 供貨單位 / 合同號 / 物料號 / 產品型號 / 數量(PCS) / 品牌 / 出廠日期 / 箱號；SOURCE 含**派生字段** 供應商代碼=「CONTRACT NO. 前5位」、流水號=「包數」（资料表表头明示）——证明模板需支持**派生/计算字段**，不只是原样映射。
- 二维码拼接分隔符各客户不同（`&` `;` `+` `,` `_`，蓝图 §6）；字段在二维码里的**顺序**也各客户不同。
- 旭创/智禾要求"数量列加条形码"、外箱"发货日期加条形码"（蓝图 §6）——证明**任意字段可渲染为条码/二维码**，不限固定字段。
- 入仓编号标签（图片5.png）：内容 `PR2604-048-01`、尺寸 **62mm × 29mm**、含 Code128 条码 → 这是系统**自产**的标签（非客户要求），入仓时每包一张。

### 典型流程

**A. 配置一个客户标签模板（ADMIN / 物流主任，离线/初始化）** — 引用蓝图 §6
1. 选 **公司**（6 租户之一，含区域 HK/内地）+ **客户**（FK 主数据）+ **标签类型**（包装1/包装2/外箱/公司外箱）。
2. 录**字段映射**：每个标签字段 →（来源类型：入库批次字段 / 出库登记字段 / 客户邮件附件手填 / 客户系统导出 / 派生公式 / 常量）+ 来源字段名 + 显示标题。来源枚举对齐主清单 xlsx「标签数据来源判断」列（客户系统直接打印 / Photonteck 自制表格）。
3. 配**二维码拼接规则**：勾选进二维码的字段 + 拖拽**字段顺序** + 选**分隔符**（`&`/`;`/`+`/`,`/`_`/无/自定义）。
4. 配**尺寸**（如 62×29mm）+ **任意字段渲条码**开关（指定哪些字段额外渲 Code128/QR，如数量列、发货日期）。
5. 预览（用样例数据渲染一张）→ ChangeCard → 保存。

**B. 出库使用标签模板（物流专员）** — 引用蓝图 P03 ★非财务关卡
1. 出库时选**客户 + 标签类型** → 系统按模板**自动拉当前出库单/批次数据灌入**（替代现状"打开上一单模板手改数据"，访谈 05）。
2. 系统渲染：内箱标签（按 seal number/SN 对应每包）+ 外箱标签（按分箱）。空白分隔标签照旧打印（访谈 05「中间加一个空白格用来辨别哪一家」）。
3. 物流专员核对原厂标签 ↔ 出库标签（型号/SN/数量/日期），边贴边检查（访谈 05），拍快照传共享（占位，归 3 WMS 出库流程）。
4. 第二位出库同事**互检快照**（访谈 05「给另外一个同事先检查，没问题才会出」）——此互检在 3 WMS 出库状态机，本模块只提供模板渲染。

### 引擎映射 & 兼容性 ➕extension

- **模板本体 = doc_type + 子表**：`label_template`（`__doc_types__`）+ `label_template_field_line`（`__queryable__`，名含 `_line`→SubTableEditor 自动多行网格录入，对齐 UX 公约 §1）。字段映射、二维码字段序、条码开关都是**数据行**，新增客户=新增模板记录，**零代码**（满足蓝图 §6"新增客户/规格无需改代码"）。
- **渲染落点 = custom_html 逃生舱**：引擎前端 `NodeView` 对有 `custom_html` 的状态走 `innerHTML` 注入（引擎 08 §8.3、FR-8.10）。标签渲染产物（含条码/二维码图、布局）作为模板的 `render_html` 由后端按字段映射 + 拼接规则**服务端生成**后注入预览。**条码/二维码图**用服务端库（如 python-barcode/qrcode）生成为 `<img>` data-uri 嵌入，避免前端拼。
- **二维码拼接 = 后端命令**：`build_label_payload(template_id, doc_id)` 按 `qr_field_order` + `qr_separator` 拼字符串，按 `barcode_fields` 渲条码——建议落为 `@register_command`（幂等/留痕，引擎 03），供出库渲染调用。
- ⚠️ **XSS 面（必管控，引擎 08 D-08d）**：`custom_html` 经 `innerHTML` 注入有 XSS 风险。治理约束：**render_html 只由后端按白名单字段拼装生成，不接受用户直接写 HTML**；客户字段值做转义；模板结构（布局骨架）只由 ADMIN/物流主任配置。**必入 engineFlags**。
- ➕ **派生字段/计算 DSL**：source_type=派生公式（如"合同号前5位"=供应商代码、"包数"=流水号）引擎无原生计算层；建议复用规则 DSL（rules.py）或在 `build_label_payload` 命令内实现有限函数集。**派生表达式语法待甲方/工程定义**（gap）。
- ➕ **打印驱动集成占位**：渲染产物→物理打印机（BarTender 类）集成方式未定（现状 BarTender + Excel 数据源，图片5.png）。**占位待甲方**。

### 验收 + 待甲方 gap
- 验收：新增 1 个客户标签模板（如 SOURCE 包装标签）全程无需改代码；二维码按配置分隔符+字段序拼装正确；指定字段渲出 Code128；预览与样例一致；停用不影响历史。
- gap：① 各客户标签字段"待确认"来源需逐一签字（主清单 xlsx 多处"需确认数据来源"，00 gap 2）；② 派生字段表达式语法；③ 打印驱动集成；④ 二维码内容是否需符合客户系统校验规则（如某些客户扫描入库校验）——待甲方。

## 9.3 编号规则

### 定位与使用者
**定位**：维护**公司 × 单据类型 × 前缀 × 月度重置 × 连号**的编号生成规则（蓝图 §5.5 全表）。**使用者**：ADMIN（配）、财务（参与口径评审，因编号=金蝶单号映射键）。

### 典型流程

## 9.4 审批流配置

### 定位与使用者
**定位**：可配审批**节点 / 会签人 / 阈值**（如备货 ≥20 万走会审；采购审批线；出库财务放行）。**使用者**：ADMIN（配流程定义）、财务（参与阈值评审）。**这是本模块唯一 ✅ 引擎原生兼容项**——审批流即 `WorkflowDefinition.states`，引擎已提供完整管理态编辑器。

**一手源**：蓝图 §3.7「审批流 ApprovalFlow + 实例」、§5.1「审批节点：采购审批、入库审核、发票审核、备货会审(≥20万)、出库财务审核」、§备货「金额<阈值 PM/PD 批 / ≥20 万会审」；00 决策⑤（审批中心统一收件箱吸收全部 ★关卡）。

### 典型流程（ADMIN 配置审批流）— 引用引擎 02 §2.4/§8.6
1. 在 **FlowEditor 管理态**（引擎 08 §8.3、FR-8.6）打开/新建某 doc_type 的 `WorkflowDefinition`。
2. 增删**节点**（状态）：设 `code/name/allowed_roles/editable_fields/description/hard_rules/effects`；增删**边**（动作）：设 `to/label/roles/editable_fields/hard_rules`。
3. **会签人**：节点 `allowed_roles` 多角色 = 并签可推进（任一推进）；**真会签（多人都须签）**需拆为串联节点（节点1=会签人A、节点2=会签人B），引擎不原生支持"同节点多人全签"（见兼容性）。
4. **阈值关卡**（备货 ≥20 万会审）：用节点/边 `hard_rules`（rules.py DSL 声明式断言，如 `amount >= 200000` 时强制走会审分支）或 `@register_transition_validator`（引擎 04 §4.A.1）。
5. 拖拽布局 → publish（发布即锁内容，danger 覆盖需审计，引擎 05 §5.4）→ 写 `WorkflowDefAuditLog` 前后快照。

| 关卡 | doc_type | 节点 allowed_roles | 阈值/硬规则 | 落点 |
|---|---|---|---|---|
| 采购审批 ★ | PO | FINANCE | 金额线（待甲方） | hard_rules / validator |
| 入库审核 ★ | InboundReceipt | PA | — | 节点角色闸 |
| 进项发票审核 ★ | 进项发票 | PA + FINANCE | — | 串联会签节点 |
| 出库财务放行 ★ | OutboundOrder | FINANCE（继仓库互检后） | 货不能出仓 | 两节点（互检+放行，决策⑥） |

| SO 审核 ★ | SALES_ORDER | SA leader | 电子章 | 节点角色闸 |

### 字段/状态机
本页**不新增 doc_type**——配置对象就是引擎 `WorkflowDefinition`（管理态，引擎 02 §2.2 数据契约）。配置动作的状态机 = FlowEditor 生命周期：草稿（draft）→ 发布（published，内容锁）→ 停用（disabled）/ fork 改版（引擎 05 §5.6）。

### 引擎映射 & 兼容性 ✅compatible
- **✅ 完全原生**：审批节点/会签/分支 = `WorkflowDefinition.states` JSONB；管理用 FlowEditor（引擎 08 FR-8.6 全 CRUD states + publish 锁 + danger 审计）；阈值 = hard_rules DSL / validator（引擎 04）；统一收件箱 = `list_user_todos`/`list_user_actions`（引擎 02 §2.5）。
- **➕ 真会签（多人全签）不原生**：节点 `allowed_roles` 是"任一角色可推进"，非"多人都须签"。≥20 万备货"会审"若指**多人各自签字才放行**，需拆串联节点或新增"签字计数 validator"——**待甲方确认会审语义**（是"多人都签"还是"升级到更高角色单签"）。落地若需多人全签 → engineFlags ➕（新增会签计数 validator）。
- ⚠️ **边级角色推进路径缺陷（引擎 02 D-02e）**：`next[].roles` 在 preview/枚举强制、但 `execute_transition` 推进路径只校验节点级 `allowed_roles`——**凡依赖边级角色的审批关卡，须标"需引擎补齐 FR-2.7"**（00 §4.2）。本模块配置审批流时**优先用节点级 `allowed_roles` 承载关卡角色**，规避该缺陷；如必须用边级角色 → 必入 engineFlags。
- ⚠️ 已发布流程内容锁定，改须 danger 覆盖 + 审计（引擎 05 §5.4）——这是**特性非缺陷**（防止在途单据被改流程坑），PRD 须告知 ADMIN。
