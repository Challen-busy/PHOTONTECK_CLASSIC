# 03 · 命令引擎

## 3.1 模块定位

一个**业务无关的统一写入口**：任何主体（人/Agent/系统）发起的跨模块写，都路由到一个具名、已注册的命令处理器。引擎给每次调用裹上四件套——幂等去重、只增命令日志、单一全有或全无的事务边界、统一成功/错误信封——并向处理器递交共享执行上下文。领域逻辑只活在注册的插件处理器里；引擎本身无任何业务语义，只负责**发现、派发、记录、事务保护**，并在执行历史上提供只读可观测 API。

> 关键文件：`services/commands.py`、`services/command_registry.py`、`services/command_context.py`、`routers/command_center.py`、`models.py:1171 CommandLog`

## 3.2 不变量

- **INV-3.1 单一写入口**：跨模块写只经 `execute_command`；未注册命令名返回错误信封且**不落任何状态**。
- **INV-3.2 单事务**：处理器的写 + 命令日志状态更新**一起提交**，任一失败**一起回滚**；无部分提交。
- **INV-3.3 处理器不管事务**：处理器只用 `ctx.db` 写，**不得自行 commit**——commit/rollback 由引擎掌管。
- **INV-3.4 幂等由 DB 兜底**：`(command_name, idempotency_key)` 的部分唯一索引保证"每键至多一个活动(运行中/已成功)命令"。
- **INV-3.5 留痕**：每次调用都写 `command_log`（RUNNING→SUCCESS/FAILED）。

## 3.3 注册（唯一扩展点）

```python
@register_command(
    "<name>", *, module="GENERAL", title="", description="",
    affected_tables=(), supports_retry=False, supports_rollback=False, supports_preview=False,
)
async def handler(ctx: CommandContext, payload: dict) -> dict:
    # 所有写经 ctx.db；返回 dict（引擎补 success/command/events/logs/command_log_id）
    ...
```

- 装饰器把 `func` 存入 `_COMMANDS[name]`，把冻结 `CommandMetadata` 存入 `_METADATA[name]`，原样返回 `func`。
- **发现**：`load_commands()` 惰性幂等——首次 lookup 时 import 插件模块（其 `@register_command` 在 import 时执行完成注册），`_loaded=True`。**无启动钩子、无文件扫描**；加插件模块要改这段硬编 import 列表（见差距 D-03a）。
- 处理器签名固定：`async (ctx, payload) -> dict`。

## 3.4 执行器机制（`execute_command`，`commands.py:97`）

签名 `execute_command(db, user, command_name, payload=None, *, idempotency_key=None, log_payload=None)`：

1. 取 `actor_id=user.id`、`company_id=user.company_id`。
2. `get_command_handler(name)`；缺失→`{success:False, error:'未注册命令', status_code:400}`，不写日志。
3. **幂等前置**（给了 key）：查最近 10 条同 `(name,key)` 且状态∈(SUCCESS,RUNNING)；已 SUCCESS→回放其 `result_payload` + `{idempotent:True}`；并发 RUNNING→`409 '相同幂等键的命令正在执行'`。
4. **写 RUNNING 日志** + `flush`；`IntegrityError`（撞部分唯一索引）→回滚→重查幂等→否则 `409 '命令日志创建失败'`。
5. **派发**：`ctx=CommandContext(db,user,command_log=log)`，`await handler(ctx,payload)`；结果 `setdefault` 补 `success/command/events/logs`，强置 `command_log_id`。
6. **提交**：log.status=SUCCESS、result_payload、completed_at，`db.commit()`——处理器写与日志一起原子提交。
7. **错误信封**：`CommandError`（带 message/status_code/details）→回滚 + `_failed_result`；其他异常→回滚 + 500。`_failed_result` 另写一条 FAILED 日志（RUNNING 行已回滚），返回 `{success:False, command, error, status_code, [details]}`。
8. `_json_safe` 递归把 `Decimal→str`、`date/datetime→isoformat`，保证 JSONB 可存。
9. `log_payload` 让调用方持久化"脱敏/摘要版"请求，区别于实际执行的 payload。

## 3.5 执行上下文（`command_context.py`）

`CommandContext(db, user, command_log, events:list, logs:list)`；`add_event(type,payload)` / `add_log(msg)` 是结构化事件/轨迹旁路——经 execute_command 的 setdefault 回灌进结果信封并存进 `result_payload`，让处理器不碰事务也能发事件/留可读轨迹。

## 3.6 命令中心 API（`routers/command_center.py`，前缀 `/api/commands`，全程鉴权 + 租户作用域）

| 端点 | 作用 |
|---|---|
| `GET /catalog` | 列全部注册命令元数据（name/module/title/affected_tables/supports_*） |
| `GET /logs` | 过滤 + 租户作用域执行日志（按 name/module/status/actor/company/key/时间范围，限条数） |
| `GET /logs/{id}` | 单条执行详情，请求/结果 payload 脱敏（`_redact_payload` 按敏感子串遮蔽 + 截断 + 限深） |
| `POST /logs/{id}/retry` | 仅 `status=='FAILED'` 且 `metadata.supports_retry` 才允许；复用原 payload + 原幂等键重跑（已成功的重试自动去重），打 `retried_from_id` |
| `GET /failures/summary` | 按命令聚合失败数 + 最后错误/时间 |

租户：`_company_ids(user)` 包 `_company_filter`，查询按 `company_id IN (...)`，`_assert_command_access` 越租户 403。

## 3.7 数据契约（引擎级）

```
CommandLog (models.py:1171):
  id, command_name(idx), idempotency_key, actor_id(idx), company_id(idx),
  status∈{RUNNING,SUCCESS,FAILED}(idx), request_payload JSONB, result_payload JSONB,
  error_message, created_at(idx), completed_at
  ↳ 部分唯一索引 ux_command_log_name_key_active
      ON (command_name, idempotency_key) WHERE key IS NOT NULL AND status IN (RUNNING,SUCCESS)
      —— DB 级"每键至多一个活动命令"，是 IntegrityError 去重的兜底
  ↳ ix_command_log_name_created (command_name, created_at) —— 供目录/失败汇总

CommandMetadata (内存 dataclass，不持久化):
  name, module(默认 GENERAL), title, description, affected_tables, supports_retry/rollback/preview
```

## 3.8 功能需求

- **FR-3.1** 提供单一执行器入口，把每次跨模块写派发到具名预注册处理器；未知命令名返回结构化错误信封且不持久化任何变更。
- **FR-3.2** 命令经装饰器注册：处理器 + 声明式元数据（name/module/title/description/affected_tables/能力标志）。
- **FR-3.3** 处理器须收到执行上下文（事务句柄、鉴权主体、命令日志行、事件/轨迹累加器）与自由 payload，返回 dict，**不得自管事务**。
- **FR-3.4** 执行器须把每次调用裹进恰好一个事务：成功时处理器写 + 日志状态更新一起提交，失败时一起回滚。
- **FR-3.5** 每次调用须写只增命令日志：命令名、可选幂等键、主体、租户、状态、（可脱敏的）请求 payload、结果 payload、错误、时间戳。
- **FR-3.6** 给幂等键时须去重：同 (命令,键) 的先前成功须回放并标 idempotent，并发在途须返回 409；DB 级部分唯一索引须强制"每键单活动"。
- **FR-3.7** 成功与失败须返回统一信封（success/command/status_code/error/details/command_log_id），受控领域错误（带状态码/details）与意外异常（500）一致处理。
- **FR-3.8** 入库 payload 须强制序列化安全（小数→串、日期→ISO，递归）。
- **FR-3.9** 须提供只读 API：列命令目录、按 命令/模块/状态/主体/租户/键/时间 浏览日志（租户隔离 + 限条数）、取单条（敏感字段脱敏）、按命令聚合失败。
- **FR-3.10** 仅当元数据允许重试时才许重跑失败命令，复用原 payload + 幂等键（已成功的重试去重），并记来源日志 id。
- **FR-3.11** 命令发现须经惰性幂等加载插件模块，使装饰器注册在首次 lookup 前完成，且业务模块不硬接进引擎核心。
- **FR-3.12** 所有命令中心读端点须强制租户作用域。

## 3.9 边界：本模块明确排除的业务

- 注册插件**模块名**（WMS/FINANCE/WORKFLOW 是业务命名空间，不是引擎模块）。
- `services/wms_commands.py`、`services/finance_commands.py` 内全部命令体；`services/workflow_commands.py` 的 `workflow_transition/workflow_commit_card`（属 Workflow 引擎包装）。
- 业务调用路由（`routers/wms.py::_run_command`、`routers/workflow.py /api/transition`…）。
- `@register_transition_validator/effect` 注册器（属 **Workflow** 引擎，见 04，**不在** `command_registry.py`）。
- `affected_tables` 引用的业务表与 `/logs/{id}/inventory-movements`、库存成本脱敏等领域逻辑。

## 3.10 现状与差距

- **D-03a** 命令发现是 `load_commands()` 里**硬编 import 列表**的惰性加载，无文件扫描/启动钩子；加插件模块须改这段 import（FR-3.11 的"自动"是有限的）。
- **D-03b** `supports_preview/supports_rollback` 仅作元数据广而告之，**引擎无 enforcement**（无 preview/rollback 端点或分支）；只有 `supports_retry` 真被门控。属能力高估，需补实现或如实降调。
- **D-03c** `ARCHITECTURE.md` 把 Validator/Effect 注册写得像挨着命令注册器；实际它们在 Workflow 侧，`command_registry.py` 只含命令处理器与元数据两张注册表。
- **D-03d** 引擎对 actor 无 user/agent/system 分支；主体身份完全来自传入的 `user` 对象。
