# 05 · 权限 · 会话 · 多租户引擎

## 5.1 模块定位

多租户 FastAPI 后端的认证与授权底座。它对哈希口令库认证用户，持久化一份服务端签名、只引用 user id 的会话，每次请求解析活体用户，并强制三个正交访问维度：(1) 超管标志门控所有元数据管理端点；(2) 角色驱动的**表级**可见性 + **字段级**遮蔽；(3) 租户（公司）**行级**隔离。它还为引擎元数据（流程定义）的改动提供只增的前后快照审计。这一切都是业务无关的管道；具体角色/表/价格字段是被宿主业务注入的配置。

> 关键文件：`core/auth.py`、`main.py`、`routers/auth.py`、`routers/admin.py`、`services/tools.py`（三层闸）、`models.py:67 UserAccount / :1375 WorkflowDefAuditLog`

## 5.2 会话与认证

- **会话**：`main.py:11` 装 Starlette `SessionMiddleware(secret_key=...)`，状态存在**签名客户端 Cookie**（无服务端会话库）。登录（`auth.py:34`）`authenticate()` 后 `request.session["user_id"]=user.id`——载荷只是个整数 id。登出（`:43`）`session.clear()` 只清本端 Cookie。
- **每请求解析**：`get_current_user`（`core/auth.py:35`）从会话取 `user_id`，无则 401「未登录」；再按 id **且 `is_active==True`** 查 `UserAccount`，禁用/不存在即 401。
- **口令**：`hash_password`（`auth.py:14`）= 全局静态盐 `"photonteck_salt"` + SHA-256（非 bcrypt/argon2、无每用户随机盐，代码注释自陈非生产级）。`authenticate` 按 `username` 且 `is_active` 查后比对哈希。

## 5.3 三个授权维度

### 维度 1 · 超管（`is_admin`）
`require_admin`（`admin.py:18`）依赖 `get_current_user`，非 `is_admin` 即 403「需要管理员权限」。所有元数据管理路由都依赖它。

### 维度 2 · 角色 → 表级权限（"allowed_tables"）
- `UserAccount.role`（自由文本）键入 `services/tools.py` 的硬编白名单：`FULL_ACCESS_ROLES` 绕过表限；`ROLE_ALLOWED_TABLES` 角色→允许表集，与 `_COMMON_TABLES` 并；`_user_allowed_tables(user)` 全权角色返回 `None`(不过滤)，否则角色集（未知角色落 `_DEFAULT_ALLOWED_TABLES` 最小集）。
- `_user_can_access_table` 是闸：`None`→放行，否则成员判定。`_visible_tables` 与活体 `table_map()` 取交集防笔误。闸在通用查询工具内强制（`query_data/aggregate`），拒绝时回带 `available_tables`。
- **注意**：`/api/me`、`/api/login` 载荷里的 `allowed_tables`（`None`=全、排序列表=受限）是**角色的计算投影，不是 UserAccount 存储列**。

### 维度 2b · 字段级防火墙
`_serialize_row`（`tools.py:244`）序列化时按 (表集 × 角色谓词) 遮蔽：买价/卖价列对无相应可见权角色删；`user_account.password_hash` 永删，`is_admin` 对非 ADMIN/BOSS 删。

### 维度 3 · 多租户行隔离
`_company_filter(user)`（`tools.py:23`）privileged 角色返回 `None`(看全部)，否则 `[user.company_id]`(单公司)；调用方仅当 `company_ids` 非空**且**模型有 `company_id` 列时加 `model.company_id.in_(...)`。`_assert_company_access`（`data.py:56`）对越权访问外部公司 403。隔离是行级、按查询 opt-in、对无 `company_id` 列的表自动跳过。

## 5.4 元数据管理 + 配置审计（`routers/admin.py`）
- CRUD：用户（建/列）、流程**定义**（建/列/fork/publish/disable/enable/delete/regroup/edit-states/save-positions）、知识条目。
- 引擎不变量：已发布定义内容锁定（除非 force/danger 标志）；publish 要求非空 states；非终态单据存在时禁 disable；已发布禁 delete，delete 在应用层级联审计行（FK 非 DB 级联）。
- `_audit`（`admin.py:120`）写 `WorkflowDefAuditLog`：`change_type`、自动 `summary`（`_diff_states_summary` 按节点码 diff states JSONB→新增/删除/改键）、before/after JSONB 快照、danger_mode、changed_by_id、请求 IP。读端点 join 操作者姓名、按时间倒序回全量前后快照。只增（除父删级联外无改/删路径）。

## 5.6 接口

| 端点 | 说明 |
|---|---|
| `POST /api/login` | `{username,password}`；成功置签名会话 Cookie，回当前用户（含计算 allowed_tables） |
| `POST /api/logout` | 清本端会话（无服务端失效） |
| `GET /api/me` | 身份 + 角色 + 租户 + is_admin + 计算 allowed_tables |
| `GET/POST /api/admin/users` | 列/建用户（超管） |
| `GET/POST /api/admin/{defs}` | 列/建流程定义（超管） |
| `PATCH /api/admin/{defs}/{id}/states` | 改定义体；已发布须 force；写前后快照审计（超管） |
| `POST /api/admin/{defs}/{id}/fork\|publish\|disable\|enable` | 生命周期 + 不变量 + 审计（超管） |
| `DELETE /api/admin/{defs}/{id}` | 删未发布定义，应用层级联审计（超管） |
| `GET /api/admin/{defs}/{id}/audit` | 前后快照变更史（含操作者 + IP，超管） |

## 5.7 功能需求

- **FR-5.1** 须以用户名/口令认证，对单向哈希校验，认证期即拒绝非活跃账户。
- **FR-5.2** 须经签名 Cookie 建会话，只承载身份的不透明引用，Cookie 内无业务数据。
- **FR-5.3** 每个鉴权请求须从会话引用解析当前用户并**重校 is_active**，否则 401（惰性吊销）。
- **FR-5.4** 登出须清本端会话；PRD 须明示会话是否服务端可吊销——**本实现不可**（无服务端会话库），唯一运维 kill-switch 是停用账户，于下次请求生效。
- **FR-5.5** 须支持超管标志，用单一可复用授权依赖门控所有元数据管理端点，缺失返 403。
- **FR-5.6** 须支持角色→可见表集的可配置注册，全权角色绕过、未知角色落最小默认集，允许集须与活体表注册取交集。
- **FR-5.7** 须在身份端点把用户有效表权限作为角色的计算投影暴露（null=不限、列表=受限）。
- **FR-5.8** 须支持角色驱动的字段级遮蔽（凭证哈希、特权标志、可配敏感字段）。
- **FR-5.9** 须把数据读作用域限定到调用者租户：每用户租户过滤返回"全部"或"本租户 id"，任何声明租户列的实体查询都须套用；无租户列的实体不受影响。
- **FR-5.10** 须对 privileged 但被作用域限制的越租户访问返 403。
- **FR-5.11** 须允许管理员 CRUD 用户（哈希口令/角色/租户/admin 标志）与流程定义（建/改/fork/publish/enable/disable/delete），受不变量约束（已发布内容锁定除非 danger 覆盖、publish 要求非空、在途单据时禁 disable、已发布禁 delete）。
- **FR-5.12** 须对每次元数据改动写只增审计：变更类型、自动可读摘要（diff 定义体得出）、全量前后快照、danger 覆盖标志、操作管理员、请求 IP、时间戳，并只读暴露（解析操作者姓名）。

## 5.8 边界：本模块明确排除的业务

- 具体角色名（BOSS/FINANCE/OPERATIONS/SALES_*/PRODUCT_*/LOGISTICS）与 `FULL_ACCESS_ROLES` 具体集。
- `ROLE_ALLOWED_TABLES` 里全部业务表名、`BUY_TABLES/SELL_TABLES/_can_view_*` 价格防火墙配置。
- `WorkflowDefinition` 的业务语义（doc_type/states/具体生命周期）、`KnowledgeEntry` 业务内容。
- Admin Agent / LLM 聊天端点（属 Agent 子系统，见 06）。
- 具体公司集与币种/编码。

## 5.9 现状与差距

- **D-05a** auth/RBAC 引擎在 `ARCHITECTURE.md` 里几乎无文档（只在 §四高层提 `_company_filter`）；会话模型、口令哈希、`is_admin` 闸、`ROLE_ALLOWED_TABLES`、字段防火墙都靠代码为准。
- **D-05b** `allowed_tables` 是角色的计算投影，**非** UserAccount 列（models.py 无此列）。
- **D-05c** `UserCompanyAccess`（多公司授权表）已建并 seed，但 `_company_filter` 从不读它——多公司访问靠角色判定（None=全部），该表对隔离逻辑是**死代码**。
- **D-05d** `WorkflowDefinition.is_frozen` 为遗留未用列。
- **D-05e** 生产化缺口（代码注释自陈）：会话用硬编 dev `secret_key`；CORS `allow_origins=['*'] + allow_credentials=True`；口令单一全局盐。**上生产前必整改**（详见 00 §0.7 D10）。
- **D-05f** 会话**不可服务端吊销**（无会话/令牌库），只能靠 `is_active=False` 于下次请求 401——非实时踢人。
- **D-05g** "privileged" 在代码里其实是**两个不同集合**：行级隔离绕过集（`_company_filter`，`tools.py:25`）与表级访问全权集（`FULL_ACCESS_ROLES`，`tools.py:69`）成员不同。本 PRD 统称"privileged"是为剥离业务角色名；落地时勿假设两者一致（具体成员属业务配置）。
