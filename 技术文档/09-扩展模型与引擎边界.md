# 09 · 扩展模型与引擎边界

> 这是把"引擎/业务边界"落成**可操作契约**的一篇：接一个新业务到底要做哪些动作、从哪些口子接、付出多大代价；以及哪些东西**永远不许动**。

## 9.1 接入一个新业务的全部动作

业务模块通过四个注册点（+ 元数据）把自己绑上引擎，引擎自身永不动：

| 扩展点 | 怎么接 | 代价 | 详见 |
|---|---|---|---|
| **新单据/实体** | `models.py` 加 `Base` 子类并写 `__doc_types__=("KEY",)`（或子表/日志写 `__queryable__=True`）；alembic 迁移；插一行 `WorkflowDefinition`（states JSONB） | 一个模型类 + 一次迁移 + 一行 JSONB | [01](01-元数据与注册引擎.md) [02](02-流程与状态机引擎.md) |
| **新跨模块命令** | 一个 `@register_command(name, module, affected_tables=...)` 异步函数，并把模块加进 `load_commands()` 的 import | 一个函数（+ 一行 import） | [03](03-命令引擎.md) |
| **新流程校验** | 一个 `@register_transition_validator(name, doc_type=, to_state=)` 函数（返回失败串列表即阻断） | 一个函数 | [04](04-校验副作用与规则钩子引擎.md) |
| **新流程副作用** | 一个 `@register_transition_effect(name, doc_type=, to_state=)` 函数；`auto=False` 时还须在 state JSONB 的 `effects` 数组列出 | 一个函数（+ 可能一行 JSONB） | [04](04-校验副作用与规则钩子引擎.md) |

**纯配置类改动**（连函数都不用写）：
- 改流程（加节点/改边/调字段白名单/加 hard_rules）= 改 `WorkflowDefinition.states` JSONB（admin 端 PATCH，发布前可改、发布后 danger 覆盖 + 审计）。
- 加可查实体到 Agent/数据浏览器 = 给模型加标记（`__queryable__`/`__doc_types__`）。
- 加可读标签 = 在 `labels.py` 字典加条目（机制保证未映射也不报错）。
- 加角色可见表 = 编辑 `ROLE_ALLOWED_TABLES`（**当前硬编**，待迁 DB）。

> 全链路打通（多单据派生）只在 Effect 这一层完成——业务把 `询价→报价→订单→采购→入库→出库→发票→应收/应付` 串成一组 effect，引擎只负责按 `(doc_type,to_state)` 调度。

## 9.2 接入时必须遵守的契约（否则破坏不变量）

- **C-1 只走唯一写入路径**：任何跨模块写经 `Command → Workflow → Domain`；处理器/effect 只用 `ctx.db`/传入 `db` 写，**不得自管事务**（提交/回滚归引擎）。
- **C-2 副作用归 Effect，不写内联钩子**：新业务流用注册 effects/commands；`hooks` 写能力默认禁用，只为迁移期老定义保留。
- **C-3 字段变更走白名单**：能改哪些字段由 state/edge 的 `editable_fields` 声明，不在代码里直接 set 业务字段绕过白名单。
- **C-4 数学不变量硬编**（INV-6）：复式平衡、借贷之类**不要**元数据化。
- **C-5 留痕不可省**（INV-5）：不得新增任何绕过四本日志的"静默写"。
- **C-6 引擎不认业务术语**（INV-2）：业务名词只能出现在业务插件文件里，不得渗进引擎层。

## 9.3 不可触线（动了就说明通用化方向走错了）

**永远不应该改动**：
- 引擎本身：`core/registry.py`、`services/workflow.py`、`services/commands.py`、`services/command_registry.py`、`services/rules.py`、`services/hooks.py`、`services/tools.py`（机制部分）、`core/auth.py`、`core/database.py`。
- Command 框架（执行器/上下文/幂等/事务/信封）。
- Registry（注册约定与两张映射）。
- 审计四本的表结构与"只增"语义。
- 唯一写入路径的三层契约。

> **通用化硬标准**：要接一个新行业，如果你发现非改上述引擎件不可——那不是引擎缺功能，是**抽象漏了**。正确做法是把变化点挪到四个扩展点之一，而不是给引擎打业务补丁。

## 9.4 自检清单（判断"是否走对通用化方向"）

- [ ] 删光所有业务插件后，引擎能否独立启动？
- [ ] admin 能否纯靠 JSONB 拖出一条全新流程、并跑起一份空单据走完创建→推进→终止？
- [ ] 新业务的写是否 100% 经 Command/Workflow？有没有任何 effect/路由直接 commit？
- [ ] 引擎层文件里能否 grep 到任何业务名词（客户/库存/凭证/订单…）？应为 0。
- [ ] 新单据是否零硬编进入数据浏览器/Agent/流程图/待办（仅靠 `__doc_types__`/`__queryable__`）？
- [ ] 每个新写动作是否都在四本日志里留痕？

任一项为否，回到 9.2/9.3 找漏掉的抽象。

## 9.5 迁移/特性开关
- `ALLOW_LEGACY_WORKFLOW_WRITE_HOOKS=1`：仅为跑迁移期残留的老流程写钩子；新流程不许依赖（见 [04](04-校验副作用与规则钩子引擎.md)）。
- `OPENROUTER_API_KEY` / `LLM_MODEL` / `LLM_BASE_URL`：Agent 引擎配置；未设时引擎照常运行，仅 Agent 端点失效（见 [06](06-Agent引擎.md)）。
- `DATABASE_URL`：唯一必需运行配置（async asyncpg；alembic 自动剥 `+asyncpg` 走同步 psycopg2）。

## 9.6 接入代价总览（一句话）

> 接一个全新行业的**典型代价** = 若干模型类 + 一次迁移 + 每条流程一行 JSONB + 每个跨单据派生一个 effect 函数 + 每条领域硬约束一个 validator 函数。**引擎件改动应为 0。**

## 9.7 当前需要补齐的引擎级缺口（不接业务，属引擎自身完善）

> 这些是引擎侧的真实欠账（汇总自各分册「现状与差距」），优先级高于任何业务接入：

1. **边级角色不对称**（[02](02-流程与状态机引擎.md) D-02e）：`execute_transition` 推进路径补上 `next[].roles` 校验，与 preview/枚举一致。
2. **命令能力名实不符**（[03](03-命令引擎.md) D-03b）：`supports_preview/supports_rollback` 要么落地、要么从元数据移除。
3. **表权限 DB 化**（[01](01-元数据与注册引擎.md)/[05](05-权限会话与多租户引擎.md)）：`ROLE_ALLOWED_TABLES` 硬编 → `role_definition` 表，使 RBAC 可配置不发版。
4. **会话安全**（[05](05-权限会话与多租户引擎.md) D-05e/f）：换掉硬编 dev `secret_key`、收紧 CORS、口令换 bcrypt/argon2 + 每用户盐；若需"实时踢人"，引入服务端会话/令牌库或短期吊销名单。
5. **AgentLog 可观测**（[07](07-审计与可追溯.md) D-07c）：补一个租户作用域只读端点。
6. **孤儿清理**（[02](02-流程与状态机引擎.md) D-02c、[05](05-权限会话与多租户引擎.md) D-05c）：`edge_layout` 列、`UserCompanyAccess` 表二选一——接线或删除，别留歧义。
7. **文档纠偏**（多处）：`models.py` 的过期 docstring（hard_rules/节点 schema/AgentLog 注释）与 `ARCHITECTURE.md` 的"三本/注册器归属"措辞，统一以代码为准。

> 这些都属"引擎自身完善"，不违反 9.3 不可触线（是补齐机制，不是打业务补丁）。
