# 07 · 审计与可追溯

## 7.1 模块定位

引擎的追溯脊柱：**四本只增、永不更新的日志**，覆盖运行面（命令/状态变更/Agent）与配置面（流程定义变更）。它们让"系统内每一次写都能被归因、能被回放"成为硬约束。对应 INV-5：**"静默写"是 bug，不是优化**。

> 关键文件：`models.py:1171 CommandLog / :1307 WorkflowLog / :1361 AgentLog / :1375 WorkflowDefAuditLog`；写者 `services/commands.py`、`services/workflow.py`、`agents/agent.py`、`agents/admin_agent.py`、`routers/admin.py`

## 7.2 四本日志

| 日志 | 写者 | 捕获 | 用途 | Registry 可见 |
|---|---|---|---|---|
| **CommandLog** | `execute_command`（先 RUNNING 预 flush，再 SUCCESS/FAILED 带 result/completed_at；失败路径另写） | command_name、idempotency_key、actor/company、request_payload、result_payload、error、时间戳 | 幂等、原子边界、整命令回放 | 否（内部） |
| **WorkflowLog** | `execute_transition`（+ 创建路径 + effect 内 status log） | doc_type/doc_id、workflow_version、transition_name、from/to、changed_fields（含 status 与子表 diff）、data_snapshot、hooks_executed（hook/effect op_log）、comment、ip | 单据状态变更 + 字段 diff + 副作用轨迹 | **是**（`__queryable__`） |
| **AgentLog** | `agent.py:220/345`、`admin_agent.py:358` | agent_type、user/company、user_query、tools_called(JSONB)、response、tokens_used、duration_ms | LLM 调用/工具调用/提案归因 | 否（内部） |
| **WorkflowDefAuditLog** | `admin.py:120 _audit` | workflow_id、change_type、auto summary、before/after 快照、danger_mode、changed_by、ip、comment | 配置面变更史（前后快照） | 否（内部） |

**两面合一**：三本运行日志（Command/Workflow/Agent）+ 一本配置日志，给出运行线与配置线的完整回放 + 归因。

## 7.3 关键不变量与机制

- **INV-7.1 只增**：四表均无 update/delete 写者（唯一例外是流程定义被删时在应用层级联其审计行）。
- **INV-7.2 幂等兜底在日志层**：`CommandLog` 的部分唯一索引 `ux_command_log_name_key_active`（`(command_name, idempotency_key) WHERE key NOT NULL AND status IN (RUNNING,SUCCESS)`）保证每键至多一个活动命令——审计表本身就是幂等的强制点。
- **INV-7.3 写必留痕**：任何跨模块写都至少落一条对应日志；预 flush 的 RUNNING 行使"在途"也可见、可诊断。
- **INV-7.4 可见性分级**：`WorkflowLog` 标 `__queryable__`（业务可经通用 API/Agent 查自己的流转史）；`CommandLog/AgentLog/WorkflowDefAuditLog` 不注册（内部，仅经专用端点访问）。

## 7.4 数据契约（引擎级）

见各表定义（`models.py`）：
- `command_log`：见 [03 命令引擎](03-命令引擎.md) §3.7（含两索引）。
- `workflow_log`：`changed_fields JSONB{field:{old,new}}(+sub_changes)`、`data_snapshot`、`hooks_executed`，索引 `ix_wflog_doc(doc_type,doc_id,timestamp)`。
- `agent_log`、`workflow_def_audit`：见 [06](06-Agent引擎.md) §6.9 / [05](05-权限会话与多租户引擎.md) §5.5。

## 7.5 接口（读侧，分散在各模块）

- `GET /api/history/{doc_type}/{doc_id}` —— 单据流转史（WorkflowLog，附操作者）。
- `GET /api/commands/logs` `/logs/{id}` `/failures/summary` —— 命令执行史（脱敏、租户作用域、失败汇总）。
- `GET /api/admin/{defs}/{id}/audit` —— 流程定义变更史（前后快照、操作者、IP）。
- AgentLog 当前无专用读端点（仅写入 + 可经内部查询），是可观测性的待补点。

## 7.6 功能需求

- **FR-7.1** 须维护四本只增、永不更新的审计日志构成统一追溯脊柱：命令日志、状态变更日志、Agent 交互日志、配置变更日志。
- **FR-7.2** 命令日志须含幂等键、原子边界状态、请求/结果 payload，并由 (命令+键) 对活动行的部分唯一索引强制单活动。
- **FR-7.3** 状态变更日志须含 from/to、逐字段 old/new diff、已执行 hook/effect 轨迹、操作者、IP。
- **FR-7.4** Agent 日志须含查询、调用工具、token、时长。
- **FR-7.5** 配置变更日志须含定义变更的前后快照。
- **FR-7.6** 系统内每次写都须产生归因且可从这些日志回放；**禁止静默写**。
- **FR-7.7** 须按可见性分级：业务自助可查的流转日志暴露给通用查询面，敏感运行/配置日志仅经专用端点。

## 7.7 边界：本模块明确排除的业务

- 各业务命令/流程/Agent 产生的**日志内容**（具体 payload、具体单据流转、具体提案）——是数据，不是引擎机制。
- 业务侧对日志的二次投影（如命令日志的 `inventory-movements` 子路由）。
