# 02 · 流程与状态机引擎

## 2.1 模块定位

元数据驱动的状态机引擎：**每个单据类型的整套生命周期就是一个 JSONB 字段**（`WorkflowDefinition.states`）。引擎对外只暴露**一个写入口** `execute_transition`，它用 `to_state` 区分创建/编辑/推进三种模式，强制节点级与边级角色校验、按出边的 `editable_fields` 做字段白名单，跑声明式 `hard_rules` 与 `hooks`，并按 `(doc_type, to_state)` 调度领域 Validator/Effect。它还提供确定性的 **预览/提交（ChangeCard）** 两步协议、按角色/租户的动作与待办枚举、只增的流转日志、以及节点坐标持久化。引擎不认业务术语：业务靠模型约定 + JSONB 节点/边配置 + 注册的 Validator/Effect 接入。

> 关键文件：`services/workflow.py`、`services/workflow_extensions.py`、`services/workflow_commands.py`、`routers/workflow.py`、`routers/chain.py`、`models.py:1274 WorkflowDefinition / :1307 WorkflowLog`

## 2.2 数据契约：states JSONB（唯一事实源）

```jsonc
WorkflowDefinition {
  id, doc_type(idx), name, version,            // UNIQUE(doc_type, version)
  description,
  states: [ <StateNode> ],                     // ★ 整套流程就这一个字段
  node_positions: { "<code>": {x, y} },        // 管理员拖拽保存
  group_name(idx), is_published, is_active, is_frozen(legacy), created_by_id, created_at
}

StateNode {
  code, name, is_initial?, is_terminal?,
  allowed_roles: [role],          // 节点级角色闸
  editable_fields: [field],       // 该步可填/可改字段
  description,                    // 给 Agent 看的中文规则主战场
  agent_tools?: [tool],           // 仅供 Agent/UI；状态机本身不消费
  custom_html?,                   // 节点自定义 HTML（前端 innerHTML 注入）
  hard_rules?: [expr],            // 声明式硬规则（rules.py DSL）
  hooks?: [script],               // 钩子脚本（hooks.py DSL，默认只读）
  effects?: [name],               // 命中即按名派发的已注册副作用
  next: [ <Edge> ]
}

Edge {
  to,                            // 目标状态码
  label,                         // 动作名
  editable_fields?, hard_rules?, hooks?, effects?,
  roles?                         // 边级角色闸（见差距 D-02e）
}
```

> 节点 schema **以本节与 `ARCHITECTURE.md §二` 为准**；`models.py:1287` 的 docstring 是过期简化版（缺 hooks/effects 与边级字段）。

## 2.3 唯一写入口：execute_transition（`services/workflow.py:210`）

解析模型 `doc_model_map().get(doc_type)`、激活流程 `get_active_workflow`（`is_active==True`），再按输入选模式：

| 模式 | 触发 | 行为 |
|---|---|---|
| **创建** | `doc_id` 为空 | 进入唯一 `is_initial` 态（零/多个初始态都报错）；按目标态 `allowed_roles` 校验（ADMIN/BOSS 绕过）；自动置系统字段 `status/company_id/created_by_id`；`_auto_fill_required_fields` 为 `*_number` 列生成前缀唯一标识、为无默认的 NOT NULL 列兜底；写 `WorkflowLog(transition_name='创建', from='', to=初始码)` |
| **编辑** | `to_state` 空或等于当前态 | 只改字段不切态；`editable` = **所有出边 editable_fields 的并集** |
| **推进** | `to_state` 在当前态 `next` 中 | 改字段 + 切态；`editable` = 命中边的 `editable_fields`；非法目标 → '非法跳转' |

### 2.3.1 字段级白名单（核心不变量）
有 `field_updates` 但 `editable` 为空 → '当前动作不允许录入字段'；任一字段不在 `editable` → 指名拒绝。**只有当前边（编辑模式为全边并集）声明可改的字段才能写。**

### 2.3.2 调度顺序（推进路径的引擎契约，全程单事务、任何一步失败整体回滚）
1. **领域 Validator**（仅推进）：`run_transition_validators` 跑所有 `auto` 且匹配 `(doc_type,to_state)` 的校验器，返回失败串列表→`{error:'领域校验未通过', rule_failures}`。
2. **hard_rules**：`current_state.hard_rules + edge.hard_rules` 交 `rules.evaluate_rules`，失败→`{error:'自动校验未通过'}`。
3. **状态提交**：切态写在 hooks 之前，使 hooks 看到新态。
4. **hooks**：`current + edge + 进入态` 的 hooks 经 `execute_hooks_sync` 跑（写能力由 `ALLOW_LEGACY_WORKFLOW_WRITE_HOOKS=1` 门控），异常→'钩子执行失败'。
5. **领域 Effect**（仅推进）：`_collect_effect_names` 取节点/边显式 `effects` 有序去重；`run_transition_effects` 先按名跑显式 effect（未注册名→`ValueError` 中止），再跑所有匹配 `(doc_type,to_state)` 的 `auto` effect。
6. **日志**：写 `WorkflowLog`（version/transition_name/from/to/changed_fields[含子表变更]/data_snapshot/hooks_executed）。

> 字段值入库前一律 `_coerce_value` 按列类型强转（容忍来自 JSON/UI/Agent 的字符串）。子表 create/update/delete 经 `_apply_sub_updates` 在同事务内完成，新行自动挂父 FK，并在规则前 `flush` 以便校验看到全貌。

## 2.4 预览/提交（ChangeCard，确定性、非 LLM）

- `preview_transition`（:574）**不写库**地重跑同一批检查（角色、边合法性、边级 `roles`、字段白名单、hard_rules），产出卡片 `_make_card`：
  `{card_id, doc_type, doc_id, from_state, to_state, action_label, transition_name, changes:[{field,from,to}], checks:[...], recommendation:'proceed'|'reject', reason}`。
- `commit_card`（:693）从卡片重建 `field_updates` 调 `execute_transition` 真正落库。
- 这正是 Agent `request_action` 复用的同一原语（见 06）。

## 2.5 枚举：动作与待办

- `list_user_actions`（:723）遍历 `doc_model_map × 每状态 × 每出边`，按节点 `allowed_roles` 再按边 `roles` 过滤，产出 `{doc_type,from_state,to_state,action_label,editable_fields}`。
- `list_user_todos`（:776）= 待办生成器：遍历每个 doc_type 的激活流程，跳过 `is_terminal` 态，按角色过滤（admin/BOSS/is_admin 绕过），查该态下本租户单据（limit 200），产出 `{doc_type,doc_id,workflow_id/name,state_code/name,is_initial,summary,actions,updated_at}`；`_doc_summary` 通用地取首个非空标识 + 金额 + 币种。

## 2.6 命令层包装与路由

- `workflow_transition` / `workflow_commit_card`（`workflow_commands.py`）是注册命令，以 `manage_transaction=False` 调 `execute_transition`，把 `command_log_id` 串进 effect，失败抛 `CommandError(400, details=result)`。
- 路由：`POST /api/transition`、`/api/transition/preview`、`/api/transition/commit`、`GET /api/transitions`、`/api/my-todos`、`/api/history/{doc_type}/{doc_id}`、`/api/workflows`、`/api/state-detail/{doc_type}/{state_code}`、`POST /api/admin/save-positions`。

## 2.7 链路监控（通用、只读）

`routers/chain.py` 沿声明的 FK 把相关单据串成链：`_status_labels_by_doc_type` 从所有激活流程的 states 构 `{doc_type:{code→name}}`，`_doc_payload` 序列化并把 status 映射成中文，`_select_visible` 套租户作用域 + 表权限闸 + 排序。`GET /api/order-chains`。**具体表/FK/分组/终态集是业务**。

## 2.8 功能需求

- **FR-2.1** 每个单据类型的整套生命周期存为单个 JSONB 定义：有序状态节点 + 版本 + 激活/发布标志 + 可选分组，按 (类型, 版本) 唯一。
- **FR-2.2** 节点须支持：码、名、初始/终止标记、允许角色、可编辑字段、描述、声明式 rules/hooks、命名 effects、出边列表。
- **FR-2.3** 边须支持：目标态码、动作名、自有可编辑字段白名单、自有 rules/hooks/effects、可选边级允许角色。
- **FR-2.4** 单一写入口按输入选三模式：创建（进唯一初始态）、编辑（仅改字段）、推进（改字段+切态）。
- **FR-2.5** 创建须要求恰好一个初始态（零/多个清晰报错），自动置系统字段并为标识列生成唯一前缀标识。
- **FR-2.6** 字段级白名单：推进只许写命中边的可编辑字段；编辑用全出边并集；越界按字段名拒绝。
- **FR-2.7** 强制节点级角色校验（privileged 角色集绕过），且边级角色校验须在预览/执行/枚举三处一致。
- **FR-2.8** 入参须按目标列类型强转（日期/时间/整数/小数/布尔），容忍字符串输入。
- **FR-2.9** 须支持同事务子行 create/update/delete，新子行按声明 FK 自动挂父，并在规则前 flush。
- **FR-2.10** 推进须按序原子执行：匹配 Validator → 节点+边 hard_rules → 切态 → hooks → 匹配/命名 Effect，任一失败整体回滚并返回结构化失败。
- **FR-2.11** Effect 须支持 AUTO（匹配 doc_type+to_state）与 EXPLICIT（节点/边 effects 数组列名）两种派发；命名未注册 effect 须中止事务。
- **FR-2.12** 业务须能用装饰器按 (doc_type, to_state) + auto 标志注册 Validator/Effect，引擎惰性幂等自动发现扩展模块。
- **FR-2.13** 须提供确定性预览/提交：预览返回自描述变更卡（拟变更字段/状态、所做检查、建议、理由）且不写库；提交按卡重放经唯一写入口。
- **FR-2.14** 每次成功推进须写只增日志：版本、动作、from/to、逐字段 old/new diff、变更前快照、hook/effect 轨迹、操作者、IP、时间；并按单据暴露历史。
- **FR-2.15** 须按用户枚举可触发动作与"角色+租户"作用域的待办（排除终止态，附简短摘要与可选动作）。
- **FR-2.16** 单据类型与表发现须由模型注解约定得出（无手维护映射）；加新类型只需声明模型 + 迁移 + 插一行流程定义。
- **FR-2.17** 推进须可作为命令被调用，由外层命令层掌管事务/幂等/命令日志，并把命令 id 串进 effect。
- **FR-2.18** 须按流程持久化管理员定义的节点坐标，并暴露定义（含布局）供可视化；所有定义变更须带前后快照审计。
- **FR-2.19** 须提供通用链路监控：沿声明 FK 遍历单据，按激活流程把状态码映射为可读名，套表级访问与租户作用域并分组——不硬编任何具体单据图。

## 2.9 边界：本模块明确排除的业务

- 具体单据类型及其状态机（询价/报价/销售订单/采购通知/采购单/入库/出库/退货/预收/预付/采购发票/销售发票/应收/应付…）。
- 注册的领域扩展模块（`services/phase1_effects.py`、`services/wms_workflow_extensions.py`）与其中的整条业务链。
- 具体 Validator/Effect（`wms.validate_shipment`、`wms.apply_shipment_costs`、成本结转、凭证生成…）。
- 链路监控业务配置（`routers/chain.py` 的 `GROUPS/STAGE_LABELS/TERMINAL_OK/TERMINAL_BAD`、具体 FK 遍历、阶段分类）。
- 创建兜底里的业务细节（`period_id→AccountingPeriod`、硬编数值字段名集、`_doc_summary` 的标识/金额字段列表）。
- `allowed_roles` 里的具体职能角色（ADMIN/BOSS 绕过是引擎，具体角色名是业务）。

- **D-02a** `models.py:1287` 称 `hard_rules` "未实现执行器"——**假**：执行器已实现并接线（`workflow.py:333-347` + `preview:677-688`）。

- **D-02c** `edge_layout` 列（迁移 `f3d4e5f6`）未在模型声明、无代码读写——孤儿；布局只用 `node_positions`。
- **D-02d** `agent_tools / custom_html` 是给 Agent/UI 的元数据，状态机本身不消费。
- **D-02e** **边级角色不对称**：`next[].roles` 在 `preview_transition`/`list_user_actions`/`list_user_todos` 强制，但 `execute_transition` 的推进路径只校验节点 `allowed_roles`——直接调 `execute_transition` 可绕过被预览拒绝的边级角色。**需补齐**（见 FR-2.7）。
- **D-02f** `is_frozen` 为遗留未用列。
