# PHOTONTECK 平台引擎 PRD

> 本目录只描述 **底座 / 引擎**——一套元数据驱动、注册器接入、命令化写入、全程留痕的通用业务流程平台。
> **不含任何业务**（CRM / WMS / ERP / 财务、具体单据、具体流程、具体公司都被抽走）。业务只是注册到引擎上的插件，引擎本身搜不到一个业务名词。

## 这套 PRD 回答什么

- 引擎到底"是什么、能做什么、有什么不变量"（产品需求，不是代码讲解）。
- 一个业务模块要接入，**只能**从哪些扩展点接、付出多大代价。
- 哪些东西**永远不许动**（动了就说明通用化方向走错了）。
- 与当前代码的**真实差距**（哪些是已实现，哪些是声明了没落地）——每篇都有「现状与差距」节，不粉饰。

## 阅读约定

- **引擎 / 业务**：本 PRD 里"实体 / 单据类型 / 角色 / 租户 / 字段"都是占位词。凡出现具体业务名词（客户、库存、凭证、销售订单…）一律视为**业务实例**，不属于引擎，只在「边界」节以"排除项"列出。
- **FR-xx** = 功能需求（可测）；**INV-xx** = 不变量（违反即系统崩坏）；**现状与差距** = 截至当前代码（commit `7a06196`）的真实落地度。
- 行号引用形如 `services/workflow.py:210` 指向后端源码，可点击跳转。

## 一句话定位

> **业务 = 状态机 + 字段变更 + 副作用。** 引擎把这三件事拆开、各自有家、互不耦合，再用"命令"这一条路串起来，让幂等、事务、审计三件套天然成立。把这套底座抽出来，它不认识任何行业；接哪套行业，就注册哪套 Validator / Effect / Command。
