# 01 · 元数据与注册引擎

## 1.1 模块定位

把普通 ORM 模型类变成"自描述、可查询"的一等公民——**无需任何手写映射表**。模型用类属性约定声明自己，引擎启动时扫描一次 ORM mapper 注册表，构建两张缓存映射（逻辑单据类型→类、表名→类），并在其上提供一套完全通用的数据 API（行查询、schema 自省、外键关联遍历、聚合）。没声明的模型一律隐身。同一份注册结果喂给所有元数据消费者（通用 API、Agent 工具、流程引擎、前端流程图/数据浏览器/待办），所以**加一个新实体不需要改引擎一行**。

> 关键文件：`core/registry.py`、`routers/data.py`、`services/tools.py`、`services/labels.py`、`core/database.py:15 Base`

## 1.2 不变量

- **INV-1.1 零硬编映射**：单据类型↔模型、表名↔模型只能由扫描约定得出，不得手维护 `DOC_MODEL_MAP/TABLE_MAP`。
- **INV-1.2 默认隐身**：未声明任何标记的模型，对通用 API 与 Agent **不可寻址**（如 `command_log/agent_log/workflow_def_audit/user_company_access` 等内部表）。
- **INV-1.3 单据类型键全局唯一**：两个不同类声明同一 key 必须在启动时 `RuntimeError`（fail loud）。
- **INV-1.4 进程内不可变**：两张映射用 `@lru_cache(maxsize=1)` 缓存，进程生命周期内不失效（改模型需重启）。

## 1.3 注册约定（唯一扩展点）

模型继承 `Base`（`core/database.py:15`），通过两个类属性接入：

| 标记 | 含义 | 进哪张图 |
|---|---|---|
| `__doc_types__ = ("KEY", ...)` | 主单据类，元组内是逻辑类型键（一个类可背多个键，如同一实体走多套状态机） | `doc_model_map()` + `table_map()`，永远可查 |
| `__queryable__ = True` | 纯子表 / 主数据 / 日志等辅助实体 | 仅 `table_map()`（通用 API + 流程子表操作可见） |
| （都不加） | 内部表（权限映射、配置审计…） | 都不进，默认隐身 |

```python
# 主单据：
class <Entity>(Base):
    __doc_types__ = ("<TYPE_KEY>",)
# 辅助/子表/日志：
class <Aux>(Base):
    __queryable__ = True
```

## 1.4 机制

### 1.4.1 扫描与缓存（`core/registry.py`）
- `_ensure_models_loaded()`（:20）`import models` 强制所有类注册到 `Base.registry`。
- `doc_model_map()`（:25-38）遍历 `Base.registry.mappers`，对每个 `__doc_types__` 键建 `key→class`；键冲突→`RuntimeError`。
- `table_map()`（:41-51）：`exposed = bool(__doc_types__) or __queryable__`，建 `__tablename__→class`。这是"哪些表对通用 API/Agent 可达"的唯一事实源。
- `doc_types()`（:54）排序返回全部键，便于稳定输出。

### 1.4.2 通用数据 API（`routers/data.py` + `services/tools.py`）

| 能力 | 入口 | 机制要点 |
|---|---|---|
| **行查询** | `POST /api/query` → `tools.query_data`(:164) | `table_map()` 解析模型；租户过滤（模型有 `company_id` 才加）；精确 `filters`（`hasattr` 守卫）；模糊 `search`（固定标识列 + 任意 `*_number` 列 ILIKE）；`order_by`（`-` 前缀降序）；`limit≤100`；返回序列化行 + 同条件 `total` |
| **schema 自省** | `GET /api/schema/{table}` (:48) | 遍历 `__table__.columns` 产出 `{name,label,type,nullable,primary_key,fk,has_default}`；类型由 SQL 类型名归一为 integer/number/datetime/date/boolean/json/text/string；扫描所有表中指向本表且名含 `_line/_entry` 的 FK 得出**子表**；被字段防火墙隐藏的列从 schema 一并删，保证与 query 一致 |
| **关联遍历** | `GET /api/related/{table}/{id}` (:115) | 正向：遍历本行 FK（跳过 `created_by_id/updated_by_id/company_id`）取目标行；反向：扫所有表中指向本表的 FK，纯子表(`_line/_entry`)跳过，其余采样≤5 行 + 总数（带租户过滤），并把嵌套 FK id 用显示列优先级（`name/full_name/short_name/code/sku/*_number…`）解析成可读标签，兜底 `#id` |
| **聚合** | `POST /api/aggregate` → `tools.aggregate`(:324) | SUM/COUNT/AVG/MAX/MIN，`hasattr` 校验字段，可选 `group_by`，同样租户过滤 + 精确 filters |

### 1.4.3 序列化与标签
- `_serialize_row`（`tools.py:244`）：`Decimal→float`、`.isoformat()→ISO 串`，并应用字段防火墙（见 §1.5）。
- 标签三级回退（`labels.py`）：`get_label(table,field)` = 表级覆盖 `TABLE_LABELS[table][field]` → 全局 `GLOBAL_LABELS[field]` → 原字段名；`get_table_label` 同理；`get_value_label` 做枚举/布尔值翻译。**机制是三级回退链（永不因未映射而报错）；字典内容是业务**。

## 1.5 字段防火墙（在 query 与 schema 两处一致执行，杜绝任一路径泄露）

- **敏感字段遮蔽**：身份表上"永远隐藏集"（如口令哈希）对所有人删；"仅管理员集"对非提升角色删（`tools.py:262-266`）。
- **价格类防火墙**：买价表集隐藏买价列、卖价表集隐藏卖价列，除非角色可见该价类。
- **租户隔离**：`_company_filter(user)`（:23）privileged 角色返回 None（不过滤），否则限本租户；模型有 `company_id` 才加。
- **表级访问**：`_user_can_access_table/_visible_tables`（:137-157）限制角色可命名哪些表；`_visible_tables` 与 `table_map()` 取交集，使白名单笔误也无法暴露未注册表。

> 防火墙的**机制**是引擎；具体的"哪些列/哪些表/哪些角色"是业务配置（排除）。

## 1.6 数据契约（引擎级）

```
注册标记（类属性，非列）：
  __doc_types__ : tuple[str]   # 可多键；存在=主单据，恒可查
  __queryable__ : bool         # 辅助/子表/日志可查
  （皆无）        → 内部隐身

doc_model_map() : { 逻辑类型键 → 模型类 }   # 缓存一次，键全局唯一
table_map()     : { 表名 → 模型类 }          # 上述 + __queryable__

/schema 字段记录：{ name, label, type∈{integer|number|datetime|date|boolean|json|text|string},
                  nullable, primary_key, fk:{table,column}|null, has_default }
```

引擎依赖的**约定列名**（模型提供即自动获益）：`id`（主键/关联）、`company_id`（租户过滤，可选）、FK 约束（参与 schema/related）、显示名优先列（标签解析）；子表识别靠表名含 `_line/_entry` + 指向父表的 FK。

## 1.7 功能需求

- **FR-1.1** 模型仅凭类属性约定接入：元组标记声明主单据（支持一类多键），布尔标记声明辅助可查实体；皆无即内部不可见。
- **FR-1.2** 启动时扫描 ORM 注册表恰好一次，构建并缓存两张映射；两个类争用同一单据类型键须启动即失败。
- **FR-1.3** 提供通用查询端点：精确过滤 + 跨标识列模糊搜索（含 `*_number` 约定）+ 可排序 + 硬性条数上限，返回行与同条件总数。
- **FR-1.4** 提供 schema 自省端点：每列归一类型/可空/主键/外键目标/默认存在/可读标签 + 表标签 + 自动探测的子表。
- **FR-1.5** 提供关联端点：单行的正向外键目标（剔除审计/租户 FK）与反向引用（剔除纯子表），带租户过滤，并把外键 id 解析为可读显示值。
- **FR-1.6** 提供聚合端点（sum/count/avg/max/min + 可选分组 + 精确过滤）。
- **FR-1.7** 序列化统一把小数转浮点、时间转 ISO-8601。
- **FR-1.8** 字段防火墙必须在 query 与 schema 两路一致地剔除敏感/受限列，配置以 (表, 列, 角色) 集表达。
- **FR-1.9** 租户隔离过滤对任何声明租户列的实体透明生效，privileged 角色可绕过。
- **FR-1.10** 表级访问闸限制角色可查哪些表，并与真实注册表取交集；拒绝或未知表时回传调用者可见表清单。
- **FR-1.11** 标签解析走确定回退链（表级覆盖→全局→原名），对未映射永不报错。

## 1.8 边界：本模块明确排除的业务

- 一切具体单据类型键与实体（CUSTOMER/SUPPLIER/SALES_ORDER/INVENTORY/VOUCHER/…）及其表。
- `services/labels.py` 的 `GLOBAL_LABELS/TABLE_LABELS/VALUE_LABELS` 字典内容（中文标签/枚举翻译）。
- `services/tools.py` 的 `BUY_TABLES/SELL_TABLES/BUY_PRICE_FIELDS/SELL_PRICE_FIELDS` 与 `ROLE_ALLOWED_TABLES/FULL_ACCESS_ROLES/_COMMON_TABLES` 具体内容（业务 RBAC/价格策略实例）。
- 其余 Agent 工具（calculate/compare/request_action）与 Command/Workflow/Effect 业务层（各见对应分册）。

## 1.9 现状与差距

- 与 `ARCHITECTURE.md §三.1` 完全一致：`__doc_types__/__queryable__/无标记` 三态如实现。多处消费者（`data.py/tools.py/workflow.py/hooks.py/rules.py/agent.py/admin.py/chain.py`）都 import `table_map/doc_model_map`，无漂移。
- `lru_cache` 不可失效是隐含属性，文档未明说；改模型需重启进程（开发期 `--reload` 会自动重启）。
- `ROLE_ALLOWED_TABLES` 旁注称将迁到 `role_definition` 表，**当前仍是硬编字典**——表级权限尚无 DB 驱动模型（详见 05）。
