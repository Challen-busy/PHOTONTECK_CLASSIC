# 00 · 引擎总览与边界

## 0.1 产品定位

PHOTONTECK 底座是一套**元数据驱动的通用业务流程平台引擎**。它把"任意业务对象的全生命周期"抽象成可配置的状态机 + 字段白名单 + 注册式副作用，提供一条命令化的唯一写入通道，并对每一次写动作强制留痕。

引擎本身**不认识任何业务术语**——代码里搜不到"客户/库存/凭证"。CRM / WMS / ERP / 财务在引擎眼里都是注册进来的插件：想接哪套行业，就注册哪套 Validator / Effect / Command，引擎自身永不改动。

目标用户视角下，它能做三件事：
1. 让"加一个新单据类型 / 改一条流程"≈ 改一行 JSONB、不发版。
2. 让任何人/Agent/系统的跨模块写，都自带幂等、事务、审计。
3. 让每一次状态变更、字段变更、副作用、配置变更都能被回放、被归因。

## 0.2 六条核心哲学（不变量级）

> 来源 `backend/ARCHITECTURE.md §0`，已逐条对代码核验。

- **INV-1 业务 = 状态机 + 字段变更 + 副作用。** 三件事各有归属、互不耦合：流程引擎只管状态机，出边的 `editable_fields` 只管字段变更，Effect 注册器只管副作用。禁止三者混在同一段代码里。
- **INV-2 引擎不认业务术语。** 引擎层（`core/`、`services/workflow.py`、`services/commands.py`、`services/command_registry.py`、`services/rules.py`、`services/hooks.py`、`services/tools.py`、`core/registry.py`）不得出现任何具体业务实体名。
- **INV-3 元数据是一等公民，代码是最后手段。** 能用 `WorkflowDefinition.states` JSONB 表达的不写代码；能用 `__doc_types__` 声明的不维护映射表；能被注册器自动发现的不写硬编 if-else。
- **INV-4 唯一写入路径。** 所有跨模块写都走 `Command → Workflow → Domain` 这一条路，没有捷径、没有后门。这是幂等/事务/审计三件套成立的前提。
- **INV-5 追溯优先于灵活。** `CommandLog / WorkflowLog / AgentLog / WorkflowDefAuditLog` 强制留痕，任何写动作都必须能被回放、能被归因。**"静默写"在这套架构里是 bug，不是优化。**
- **INV-6 该硬编就硬编。** 数学性的业务不变量（如复式记账借贷平衡）**故意不做元数据化**——它们不是配置，是数学。元数据化只用在真正会因客户/行业不同而变的地方。

## 0.4 唯一写入路径（最重要的契约）

跨模块业务写**只允许**从三层下去，严禁绕过：

| 层 | 入口 | 职责 |
|---|---|---|
| **Command** | `services/commands.py::execute_command` | 用户/Agent/系统写的统一入口：命令日志、幂等校验、事务边界、统一错误结构 |
| **Workflow** | `services/workflow.py::execute_transition` | 状态机推进、角色校验、`editable_fields` 白名单、`hard_rules`、待办、流程日志 |
| **Domain** | 注册的 Validator / Effect / 业务命令 | 行业自有的不变式与数据变更；由 Command 直接调用或被 Workflow 的 effect 触发 |

注意：流程推进本身也是一条注册命令（`workflow_transition` / `workflow_commit_card`），以 `manage_transaction=False` 跑在 Command 的事务里——所以"状态机推进"天然继承幂等/事务/审计。

**允许的例外**（仅此三类）：
1. Admin/元数据写：用户、流程定义、知识条目，及 admin 写的配置审计日志。
2. 仅追加的运行时日志：`CommandLog / WorkflowLog / AgentLog`。
3. Seed / migration 脚本。

## 0.5 引擎 / 业务边界

### 属于引擎（本 PRD 覆盖）

**引擎表**（与具体行业无关、承载机制本身）：

| 表 | 角色 | Registry 可见性 |
|---|---|---|
| `user_account` | 身份/角色/租户锚点 | 内部（字段防火墙后可查） |
| `workflow_definition` | 流程定义（states JSONB 即整套状态机） | 内部（经 /workflows、/admin 暴露） |
| `command_log` | 命令执行日志（幂等/事务/回放） | 内部不可见 |
| `workflow_log` | 状态变更日志（字段 diff + 副作用轨迹） | `__queryable__` 可见 |
| `agent_log` | LLM 调用/工具调用/提案日志 | 内部不可见 |
| `workflow_def_audit` | 配置变更前后快照 | 内部不可见 |
| `knowledge_entry` | 通用知识/提示/告警条目（容器是引擎，内容是业务） | 内部 |

**引擎模块**：Registry、通用数据 API、Workflow 状态机、Command 执行器、Validator/Effect 注册器、Rules/Hooks DSL、权限/会话/多租户、Agent 运行时、审计四本、前端通用外壳与设计 token。

### 属于业务（本 PRD 一律排除）

- 一切具体 `__doc_types__` 与实体表（客户/供应商/物料/库存/仓库/凭证/会计科目/各类订单…）。
- 一切具体业务命令（`reserve_inventory`、`create_accounts_receivable` …）、Validator/Effect（`wms.validate_shipment`、`apps_shipment_costs` …）、业务流程定义（询价→报价→订单→采购→入库→出库→发票→应收/应付 整条链）。
- 一切业务字典与策略：`services/labels.py` 的标签内容、`services/tools.py` 的 `ROLE_ALLOWED_TABLES / BUY_TABLES / SELL_TABLES` 集合、具体角色名、具体报表体、具体告警检查器。
- 前端业务页面与字典：`BusinessPortal/OrderChainMonitor/WmsWorkbench/Dashboard`、`TABLE_META/VALUE_LABELS/stateColors` 等。

> 判别法：**把所有业务插件删光后引擎还能启动、还能让 admin 拖出一条全新流程并跑起一份空单据**——能，就说明它在边界的引擎侧。

## 0.6 全局非目标（Non-Goals）

- ❌ 不内置任何行业模型、不预设任何单据/科目/库存语义。
- ❌ 不做"配置化的数学不变量"（复式平衡之类按 INV-6 硬编）。
- ❌ 不提供分布式/多副本会话（当前为单进程签名 Cookie 会话，见 §0.7 差距）。
- ❌ 不在引擎层做业务报表/业务告警的具体计算（只提供路由+租户作用域+可插拔位）。
- ❌ 不替代财务记账系统（财务是被推送的下游，不在本引擎职责内）。

## 0.7 现状与差距（截至 commit `7a06196`，诚实披露）

> 本 PRD 描述"项目**当前**的底座核心"，因此必须区分**已落地** vs **声明未落地/弱实现**。各模块细节见对应分册，此处汇总全局项。

| # | 事项 | 真相 | 影响 |
|---|---|---|---|
| D1 | `WorkflowDefinition.states` 模型 docstring 称 `hard_rules` 为"暂留字段，未实现执行器" | **过期**：执行器已实现并接线（`rules.py` + `workflow.py:333-347`），hard_rules 真在跑 | 文档误导，机制可用 |
| D2 | 模型 docstring 的 state 节点结构 | 缺列 `hooks/effects` 与边级 `editable_fields/hard_rules/effects/roles`；**以 `ARCHITECTURE.md §二` 与代码为准** | 取节点 schema 时别信 docstring |
| D3 | `CommandMetadata.supports_preview / supports_rollback` | 仅作为元数据在目录/日志里"广而告之"，**引擎无任何 enforcement**；只有 `supports_retry` 真被门控 | 能力被高估，需补实现或降调 |
| D4 | 会话可吊销性 | **不可服务端吊销**：Starlette 签名 Cookie，无服务端会话表/令牌库/吊销名单。唯一 kill-switch 是 `user_account.is_active=False`，在下次请求时 401 生效 | 安全模型须明示；非"实时踢人" |
| D5 | `UserCompanyAccess`（多公司授权表） | 已建表、seed 有数据，但 `_company_filter` **从不读它**；privileged 角色靠"角色=看全部"实现 → 该表对隔离逻辑是死代码 | 多公司授权按角色，不按授权表 |
| D6 | Validator/Effect 注册器位置 | `ARCHITECTURE.md` 把它写得像挨着 Command 注册器；实际在 **Workflow** 侧（`services/workflow_extensions.py`），`command_registry.py` 只管命令处理器 | 归属：属 Workflow 引擎 → 04 |
| D7 | 报表 / 告警 | **未元数据化**：`routers/reports.py` 报表体是手写 SQL；`alerts.py` 靠标题子串硬路由到手写检查器。引擎只提供"路由 + 租户作用域 + 可插拔位" | 报表/告警的"通用"仅限脚手架 |
| D8 | `edge_layout` JSONB 列 | 迁移 `f3d4e5f6` 建了列，但模型类未声明、无代码读写 → 孤儿列。布局只用 `node_positions` | 忽略 edge_layout |

| D12 | Agent 消息格式 | 内部是 Anthropic 风格 block，但实际 loop 把工具结果摊平成 `[Tool Result]: ...` 纯文本轮次；`llm.py` 的 block 转换分支对当前调用方是死路径 | 实现细节，无功能影响 |
